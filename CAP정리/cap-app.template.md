# cap-app/template 분석

## 1. 개요

`cap-app/template` 폴더는 이름에서 알 수 있듯이, 새로운 UI5 애플리케이션을 만들 때 기반으로 사용할 수 있는 템플릿 프로젝트입니다. 이 폴더는 실행 가능한 완성된 애플리케이션이 아니라, UI5 애플리케이션에 필요한 최소한의 기본 구조와 설정 파일만을 포함하고 있는 뼈대(Scaffolding)입니다.

이 문서는 `template` 프로젝트의 구조와 그 용도를 설명합니다.

## 2. 프로젝트 구조 및 내용

`template` 폴더의 구조는 매우 간단합니다.

```
cap-app/template/
├── webapp/
│   └── manifest.json  <- 가장 기본적인 앱 설명자
├── mta.yaml
├── package.json
└── ui5.yaml
```

*   **최소한의 파일**: 프로젝트에는 `mta.yaml`, `package.json` 등 배포와 빌드에 필요한 기본 파일들과, 앱의 정체성을 정의하는 `webapp/manifest.json` 파일만이 존재합니다.
*   **컨텐츠 부재**: 앱의 실제 화면을 구성하는 뷰(View), 동작을 정의하는 컨트롤러(Controller), 시작점인 `index.html` 파일 등이 전혀 존재하지 않습니다.

### `webapp/manifest.json` 분석

템플릿의 목적을 가장 잘 보여주는 것이 `manifest.json` 파일의 내용입니다.

```json
{
	"sap.app": {
		"id": "template",
		"type": "application",
		// ...
	},
	"sap.ui5": {
        "dependencies": {
            "libs": {
				"sap.ui.core": {},
				"sap.m": {},
				// ...
            }
        }
    }
}
```

*   **`sap.app.id`**: "template"으로, 이 프로젝트가 템플릿임을 명시합니다.
*   **핵심 설정 부재**: 앱의 시작 뷰를 지정하는 `rootView`, 백엔드 연결을 정의하는 `dataSources`, 화면 전환 로직을 담는 `routing` 등 애플리케이션의 구체적인 동작을 정의하는 핵심적인 설정이 모두 빠져있습니다. 오직 기본적인 라이브러리 의존성만 선언되어 있습니다.

## 3. 용도 및 활용 방법

`cap-app/template`의 주된 용도는 프로젝트 내에서 새로운 UI5 애플리케이션을 개발할 때 일관성을 유지하고 초기 설정에 드는 시간을 줄이는 것입니다.

### 활용 시나리오

개발자가 '구매 오더 관리'라는 새로운 UI 애플리케이션을 만들어야 하는 상황을 가정해 보겠습니다.

1.  **복사**: 개발자는 `cap-app` 폴더 아래에 `cap-app/template` 폴더를 통째로 복사하여 `cap-app/purchaseorder` 라는 새로운 폴더를 만듭니다.
2.  **ID 변경**: 복사된 파일들(`mta.yaml`, `package.json`, `manifest.json` 등)을 열어 ID를 'template'에서 'purchaseorder' 등으로 변경합니다.
3.  **내용 추가**:
    *   `webapp` 폴더 안에 `view`, `controller` 폴더를 생성합니다.
    *   `Main.view.xml`, `Main.controller.js` 등 실제 화면과 로직을 구현하는 파일들을 추가합니다.
    *   `manifest.json` 파일에 `rootView`, `routing`, `dataSources` 등의 설정을 추가하여 애플리케이션의 동작을 구체적으로 정의합니다.
4.  **개발 진행**: 이렇게 준비된 기본 구조 위에서 실제 비즈니스 로직 개발을 시작합니다.

### 기대 효과

*   **일관성**: 모든 개발자가 동일한 템플릿에서 시작하므로, 프로젝트의 기본 구조, 빌드 및 배포 스크립트, 기본 라이브러리 의존성 등이 통일됩니다.
*   **생산성**: 새로운 앱을 만들 때마다 `ui5 init` 명령을 실행하거나 기존 프로젝트의 파일을 수동으로 정리할 필요 없이, 검증된 템플릿을 복사해서 바로 개발을 시작할 수 있어 생산성이 향상됩니다.
*   **오류 감소**: 초기 설정 과정에서 발생할 수 있는 잠재적인 오류를 줄일 수 있습니다.

## 4. 결론

`cap-app/template`은 비록 실행되는 코드는 한 줄도 없지만, 프로젝트의 개발 표준과 일관성을 유지하는 중요한 역할을 하는 '씨앗'과 같은 프로젝트입니다. 이는 잘 설계된 대규모 프로젝트에서 공통된 개발 프랙티스를 강제하고 개발자들의 생산성을 높이기 위해 사용하는 효과적인 방법론 중 하나입니다.
