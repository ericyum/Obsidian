# SAP CAP 프로젝트 구조 상세 분석

## 1. 프로젝트 개요

이 문서는 `C:\Users\염정운\project\CAP_Template`에 위치한 SAP CAP(Cloud Application Programming Model) 프로젝트의 전체 구조를 상세하게 분석하고 설명합니다. 이 프로젝트는 백엔드 서비스와 프론트엔드 UI5 애플리케이션으로 구성된 모노레포(Monorepo) 구조를 가지고 있습니다.

### 1.1. 최상위 디렉토리 구조

- **`/` (Root)**: 프로젝트 전체의 의존성 관리 및 실행 스크립트를 포함하는 메인 제어 센터.
  - `package.json`: 전체 프로젝트를 관리하기 위한 스크립트(실행, 빌드, 배포)를 정의.
- **`/cap-node`**: Node.js 기반의 CAP 백엔드 서비스. 두 개의 마이크로서비스로 분리되어 있음.
  - `core`: 사용자, 권한, 메뉴 등 시스템의 핵심 공통 기능을 담당.
  - `mdm`: 자재 마스터 데이터(Master Data Management)를 전문적으로 관리.
- **`/cap-app`**: SAP UI5 기반의 프론트엔드 애플리케이션.
  - `approuter`: 모든 요청의 단일 진입점(Single Entry Point) 역할을 하는 라우터.
  - `portal`: 사용자가 로그인 후 처음 마주하는 메인 포털/런치패드 애플리케이션.
  - `sysmgt`: 시스템 관리(사용자, 권한 등) UI 애플리케이션 모음.
  - `mdm`: 마스터 데이터 관리 UI 애플리케이션 모음.
  - `library`: 여러 UI 애플리케이션에서 공통으로 사용하는 UI 컨트롤, 로직 등을 모아둔 라이브러리.

## 2. 백엔드 분석: `/cap-node`

백엔드는 두 개의 마이크로서비스(`core`, `mdm`)로 구성되어 있으며, 각각 독립적으로 실행되고 배포될 수 있습니다. 두 서비스 모두 TypeScript로 작성되어 코드의 안정성과 유지보수성을 높였습니다.

### 2.1. Core 서비스 (`/cap-node/core`)

시스템의 기반이 되는 공통 기능과 데이터를 관리하는 서비스입니다.

#### 2.1.1. 데이터 모델 (`/db`)

`com.cap.core` 네임스페이스를 사용하며, 애플리케이션의 핵심 데이터 구조를 정의합니다.

- **사용자 및 권한 관리**:
  - `User`: 사용자 정보 (ID, 이름, 이메일, 개인 설정 등).
  - `Role`: 역할 (예: "관리자", "조회 전용").
  - `RoleGroup`: 여러 역할을 묶는 그룹 (예: "재무팀 역할 그룹").
  - `User_RoleGroup`, `Role_RoleGroup`: 사용자-역할그룹, 역할-역할그룹 간의 다대다 관계를 정의하는 매핑 테이블.
  - **상호작용**: `User`는 `RoleGroup`에 속하고, `RoleGroup`은 여러 `Role`을 가집니다. 이를 통해 유연한 권한 관리가 가능합니다.

- **메뉴 및 화면 제어**:
  - `Menu`: 애플리케이션의 네비게이션 메뉴 구조 (이름, 경로, 아이콘 등)를 정의. 부모-자식 관계를 통해 계층 구조를 가집니다.
  - `MenuFunction`: 메뉴(화면) 내의 특정 기능(예: "생성", "삭제" 버튼)을 정의하여 세밀한 제어가 가능.
  - `Role_Menu`, `Role_MenuFunction`: 역할별로 접근 가능한 메뉴와 사용 가능한 기능을 매핑.
  - `Favorite`: 사용자가 즐겨찾는 메뉴를 관리.
  - **상호작용**: 사용자의 `Role`에 따라 접근할 수 있는 `Menu`가 결정되고, 각 메뉴 안에서 사용할 수 있는 `MenuFunction`이 결정됩니다. 이는 프론트엔드 `portal` 앱에서 동적으로 메뉴를 구성하는 데 사용됩니다.

- **공통 유틸리티**:
  - `CodeHeader`, `CodeItem`: 공통 코드(드롭다운, 목록 값)를 관리. (예: `LANGUAGE_CODE` 그룹에 'en', 'ko' 등의 아이템).
  - `Message`: 다국어 지원을 위한 시스템 메시지를 관리.
  - `Attachment`: 파일 업로드를 위한 엔티티. 파일 메타데이터만 DB에 저장하고, 실제 파일은 AWS S3와 같은 외부 저장소에 저장 (`@cds.persistence.skip`).

#### 2.1.2. 서비스 계층 (`/srv`)

데이터 모델을 OData API로 외부에 노출합니다. 모든 서비스는 `/core` 경로 아래에 위치하며 인증된 사용자만 접근할 수 있습니다.

- **주요 서비스**:
  - `UserManagementService`: 사용자 정보 제공. 특히, `UserSessionInfo` 뷰는 현재 로그인한 사용자의 세션 정보(언어, 날짜 형식 등)를 제공하여 프론트엔드 초기 설정에 사용됩니다.
  - `MenuManagementService`: 사용자의 권한을 기반으로 접근 가능한 메뉴 목록(`MenusRoleAppliedList`)과 기능 목록(`MenuFunctionAppliedList`)을 동적으로 계산하여 제공. `portal` 앱이 이 서비스를 호출하여 개인화된 UI를 구성합니다.
  - `CodeManagementService`, `MessageManagementService`: 공통 코드와 메시지를 제공.
  - `AttachmentService`: 파일 업로드를 처리하는 로직(`Attachment.handler.ts`)을 포함.
  - `MailService`, `FormService`: 이메일 발송, PDF 생성과 같은 커스텀 `action`을 정의.

- **커스텀 로직 (`/srv/src`)**:
  - `*.handler.ts`: 각 서비스의 실제 비즈니스 로직을 TypeScript로 구현. 예를 들어, 파일 업로드 시 S3에 저장하는 로직, 이메일 발송 로직 등이 여기에 위치합니다.
  - `AWSS3StorageService.ts`: AWS S3 연동 로직을 캡슐화한 유틸리티.

### 2.2. MDM 서비스 (`/cap-node/mdm`)

자재 마스터 데이터를 생성하고 관리하는 전문 서비스입니다. `core` 서비스에 대한 의존성을 가지고 있어, `core`의 데이터 모델(예: `User`)을 재사용합니다.

#### 2.2.1. 데이터 모델 (`/db`)

`com.cap.mdm` 네임스페이스를 사용하며, 자재 마스터 생성 요청 프로세스를 중심으로 설계되었습니다. SAP ERP의 데이터 구조와 유사한 필드명(예: `ZMTART`)을 많이 사용합니다.

- **자재 마스터 요청**:
  - `MaterialMaster` / `MaterialMasterBasicInfo`: 자재 생성 요청의 헤더 역할. 자재의 기본 정보를 담습니다.
  - `...PlantInfo`, `...SalesOrganizationInfo`, `...AccountingInfo`, `...SpecInfo`: 각각 플랜트, 영업, 회계, 상세 사양 등 SAP의 Material Master Views와 유사한 자식 엔티티들로 구성됩니다. `REQ_NUM`(요청 번호)으로 부모와 연결됩니다.
  - `...InterfaceInfo`: 생성된 데이터가 다른 시스템(ERP, PDM, SAP)과 동기화되었는지 상태를 추적.

- **자재 분류 및 규칙**:
  - `MaterialClassificationCode`: 자재를 분류하기 위한 계층 구조(Level 1~4)를 정의.
  - `SpecDetail`: 분류에 따른 상세 사양 항목을 정의.
  - `MaterialAttributes`: 자재 유형별로 UI에 표시될 필드의 속성(필수, 선택 등)을 정의하여 동적 UI를 가능하게 합니다.
  - `VerificationRulesHeader`/`Item`: 데이터 입력 시 유효성을 검사하는 규칙 엔진을 정의하여 데이터 정합성을 보장.

- **제조사(Maker) 정보**:
  - `MakerInformation`, `MakerSpecMapping`: 제조사 정보와 내부 사양과의 매핑을 관리.

#### 2.2.2. 서비스 계층 (`/srv`)

MDM 데이터 모델을 OData API로 노출하며, 복잡한 로직을 백엔드에서 처리하여 프론트엔드의 부담을 줄여줍니다.

- **주요 서비스**:
  - `MaterialMasterRequestService`: 자재 생성 요청의 전체 라이프사이클을 관리. 데이터 생성, 상태 변경, 그리고 최종적으로 S/4HANA 시스템으로의 전송을 담당하는 커스텀 로직(`MaterialMasterRequest.handler.ts`)을 포함합니다.
  - `MaterialClassificationCodeService`: `MaterialClassificationHierarchyAllView` 뷰를 통해 복잡한 계층 구조의 분류 데이터를 프론트엔드에서 사용하기 쉬운 평탄화된 목록으로 제공합니다.
  - `MaterialMasterService`: 다양한 `...View` 엔티티를 통해 자재 마스터 데이터를 제공. 이 뷰들은 공통 코드 테이블과 조인하여 코드 값(예: `PLANT: '1000'`)뿐만 아니라 텍스트(예: `PLANTNA: '평택 공장'`)를 함께 제공하여 프론트엔드의 효율성을 높입니다.

- **외부 시스템 연동**:
  - `package.json`의 `cds.requires` 섹션에 `zapi_mm_create_o4` 라는 외부 OData 서비스가 정의되어 있습니다. 이는 SAP Cloud Connector를 통해 온프레미스 S/4HANA 시스템과 연결됩니다.
  - `ExternalS4hana-service.cds`: 이 외부 서비스를 CAP 내부에서 쉽게 호출할 수 있도록 프록시(Proxy) 서비스를 정의합니다.
  - **전체 흐름**: MDM UI에서 자재 생성 요청 데이터가 모두 입력되고 승인되면, `MaterialMasterRequest.handler.ts`가 `ExternalS4hana-service`를 호출하고, 이 서비스는 S/4HANA 시스템에 OData 요청을 보내 실제 자재 마스터를 생성합니다.

## 3. 프론트엔드 분석: `/cap-app`

SAP UI5로 개발된 웹 애플리케이션들로, 백엔드 서비스가 제공하는 OData API를 소비하여 사용자 인터페이스를 제공합니다.

### 3.1. 애플리케이션 라우터 (`/approuter`)

- **역할**: 전체 애플리케이션의 유일한 진입점. 인증을 처리하고, URL 패턴에 따라 요청을 적절한 백엔드 서비스나 프론트엔드 애플리케이션으로 전달합니다.
- **`xs-app.json` (라우팅 설정)**:
  - **백엔드 라우팅**:
    - `/srv-api/*` 요청은 `core` 서비스(`onCapService` 목적지)로 전달.
    - `/mdm-api/*` 요청은 `mdm` 서비스(`onMDMService` 목적지)로 전달.
  - **프론트엔드 라우팅**:
    - `/portal/*`, `/sysmgt/*`, `/mdm/*` 등의 요청은 SAP BTP의 HTML5 Application Repository에서 해당 UI5 애플리케이션을 찾아 정적 파일(html, js, css 등)을 사용자에게 제공.

### 3.2. 포털 애플리케이션 (`/portal`)

- **역할**: 사용자가 로그인 후 가장 먼저 보게 되는 메인 셸(Shell) 애플리케이션.
- **`manifest.json` 분석**:
  - `rootView`는 `Layout.view.xml`로, 전체 화면의 레이아웃(헤더, 네비게이션, 콘텐츠 영역)을 정의.
  - `UserManagementService`를 호출하여 로그인한 사용자의 이름과 같은 정보를 가져와 헤더에 표시.
- **동작 방식**:
  1. `Layout.controller.js`가 초기화될 때, `core` 서비스의 `MenuManagementService`를 호출하여 현재 사용자가 접근할 수 있는 메뉴 목록을 받아옵니다.
  2. 받아온 메뉴 목록을 기반으로 동적으로 상단 네비게이션 메뉴나 홈페이지의 타일(Tile)을 생성합니다.
  3. 사용자가 메뉴나 타일을 클릭하면, 해당 UI5 애플리케이션의 URL로 이동시켜주는 역할을 합니다.

### 3.3. 시스템 관리 앱 (`/sysmgt/userManagement`)

`sysmgt`와 `mdm` 폴더에 있는 다른 애플리케이션들의 대표적인 패턴을 보여주는 예시.

- **`manifest.json` 분석**:
  - `UserManagement`와 `RoleGroupManagement` 서비스를 `dataSources`로 사용.
  - 라우팅(Routing) 설정에서 `sap.f.FlexibleColumnLayout`을 사용하여 Master-Detail 형태의 화면을 구현.
- **동작 방식**:
  1. 초기 화면(`MainList.view.xml`)에 사용자 목록을 표시 (Master View).
  2. 사용자가 목록에서 특정 사용자를 클릭하면, 해당 사용자의 `user_id`를 파라미터로 하여 `DetailList` 라우트로 이동합니다.
  3. `DetailList.view.xml` (Detail View)이 화면 중앙(Mid Column)에 나타나며, 선택된 사용자의 상세 정보를 표시하고 편집할 수 있게 합니다.

## 4. 전체 아키텍처 및 상호작용

1. **사용자 접속**: 사용자가 애플리케이션 URL에 접속하면 **Approuter**가 요청을 받아 **Portal** 앱을 로드합니다.
2. **포털 초기화**: **Portal** 앱은 **Core 서비스**(`MenuManagement`)를 호출하여 사용자의 권한에 맞는 메뉴를 동적으로 생성하여 보여줍니다.
3. **앱 이동**: 사용자가 포털에서 '사용자 관리' 메뉴를 클릭하면, **Approuter**는 URL을 해석하여 **userManagement** UI5 앱을 로드합니다.
4. **데이터 조회/수정**: **userManagement** 앱은 필요한 사용자 데이터를 **Core 서비스**(`UserManagement`)에 OData로 요청하고, 받은 데이터를 화면에 표시합니다. 데이터 수정 시에도 동일한 서비스를 통해 백엔드에 변경사항을 전송합니다.
5. **자재 관리**: 사용자가 '자재 마스터 생성' 메뉴를 클릭하면, **mdm** 관련 UI5 앱이 로드되고, 이 앱은 **MDM 서비스**와 통신하여 자재 데이터 생성/조회/수정 작업을 수행합니다.
6. **ERP 연동**: **MDM 서비스**는 자재 생성 요청이 최종 승인되면, **S/4HANA OData 서비스**를 호출하여 실제 ERP 시스템에 자재 마스터를 생성하는 것으로 전체 프로세스를 완료합니다.

이 프로젝트는 관심사의 분리(Separation of Concerns) 원칙에 따라 공통 기능(`core`), 특정 도메인 기능(`mdm`), 그리고 각 기능에 맞는 UI 애플리케이션을 명확하게 분리한 매우 잘 설계된 마이크로서비스 아키텍처(MSA)의 모범 사례입니다.
