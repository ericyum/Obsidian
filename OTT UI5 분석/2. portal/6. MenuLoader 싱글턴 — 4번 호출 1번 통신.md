# MenuLoader 싱글턴 패턴 — 4번 호출, 1번 통신

> **작성일**: 2026-07-13
> **목적**: `new MenuLoader()`가 Component·BaseController·Layout 곳곳에서 호출되는데, 왜 HTTP 요청은 1번만 발생하는지 그 내부 메커니즘을 완전히 이해한다.
> **핵심**: 모듈 레벨 변수 `oStaticModel` + 발행-구독 패턴

---

## 1. 문제 인식

portal_ott 실행 시 `new MenuLoader()`가 **최소 4곳**에서 호출된다:

| # | 위치 | 파일 |
|---|------|------|
| ① | `Component.init()` | `portal_ott/webapp/Component.js` |
| ② | `BaseController.onInit()` | `library_ott/src/controller/BaseController.js` |
| ③ | `Layout.setHeaderNavigation()` | `portal_ott/webapp/controller/Layout.controller.js` |
| ④ | `Layout.setSideNavigation()` | `portal_ott/webapp/controller/Layout.controller.js` |

얼핏 보면 같은 OData 요청이 4번 발생할 것 같다. 하지만 **실제 HTTP 요청은 딱 1번**이다. 어떻게 가능한가?

## 2. 정답: `let oStaticModel` — 모듈 레벨 싱글턴

```javascript
// library_ott/src/util/MenuLoader.js

sap.ui.define([...], (EventProvider, JSONWithReadModel, ODataV4Model) => {
    "use strict";

    let oStaticModel;  // ← ★ 모듈이 처음 로드될 때 한 번만 생성됨.
                        //    이후 모든 MenuLoader 인스턴스가 이 변수를 공유한다.

    return EventProvider.extend("common_ott.lib.util.MenuLoader", {
        // ...
    });
});
```

**핵심**: `oStaticModel`은 클래스의 인스턴스 변수(`this.oStaticModel`)가 아니라, **모듈 클로저에 갇힌 변수**다. 즉 MenuLoader가 몇 번 `new`되든, `oStaticModel`은 단 하나만 존재한다.

JavaScript 모듈 시스템에서:
```javascript
// 첫 번째 sap.ui.require("common_ott/lib/util/MenuLoader")
// → 팩토리 함수 실행 → let oStaticModel; 생성
// → MenuLoader 클래스 반환

// 이후 sap.ui.require("...MenuLoader")
// → 캐시된 클래스 반환 (팩토리 함수 재실행 안 함)
// → oStaticModel은 그대로 유지됨
```

## 3. 생성자(`constructor`) — 분기 로직

```javascript
constructor: function() {
    EventProvider.prototype.constructor.apply(this, arguments);

    if (!oStaticModel) {                          // ← 첫 번째 호출만 true!
        // [A 경로] — 최초 실행
        oStaticModel = new JSONWithReadModel();    // 모델 생성
        oStaticModel.attachEvent("ready", this._onModelReady, this);
        this._initializeModel();                   // ← 실제 OData 호출 시작!
    } else {
        // [B 경로] — 두 번째 이후
        this._fireWhenReady();                     // ← 그냥 ready 이벤트 구독
    }
}
```

### 3.1 [A 경로] — 첫 번째 `new MenuLoader()`

```
if (!oStaticModel) → true (아직 null)
  → oStaticModel = new JSONWithReadModel()   // 빈 모델 생성
  → _initializeModel()
      → _fetchList()                          // ★ HTTP 요청 발사!
          → GET /srv-api/odata/v4/ott-core/MenuManagement/MenusRoleAppliedList
  → (HTTP 응답이 오면 oStaticModel.setReady(data) → 'ready' 이벤트 발사)
```

### 3.2 [B 경로] — 두 번째 이후 `new MenuLoader()`

```
if (!oStaticModel) → false (이미 존재)
  → _fireWhenReady()
      → oStaticModel.isReady() 확인
          → 아직 안 됐으면: setTimeout(tryFire, 300) → 300ms 후 재확인
          → 이미 됐으면:   setTimeout(fireEvent, 10) → 즉시 'ready' 이벤트 발사
```

---

## 4. `_fireWhenReady()` — 폴링 구독자

```javascript
_fireWhenReady: function() {
    const tryFire = () => {
        if (oStaticModel.isReady()) {
            // 이미 데이터 있음 → 10ms 후 바로 ready 이벤트
            setTimeout(() => {
                if (this.mEventRegistry.ready) {
                    this.fireEvent("ready", { model: oStaticModel });
                }
            }, 10);
        } else {
            // 아직 데이터 없음 → 300ms 후 재시도
            setTimeout(tryFire, 300);
        }
    };
    tryFire();
}
```

이 메서드는 두 가지 상황을 모두 처리한다:

| 상황 | oStaticModel.isReady() | 동작 |
|------|:---:|------|
| HTTP 응답이 **이미 도착**한 후 호출 | `true` | 10ms 후 즉시 ready 이벤트 |
| HTTP 응답이 **아직 안 온** 상태에서 호출 | `false` | 300ms 간격으로 폴링 |

> 💡 **이 폴링 로직이 "의도된 설계"의 증거다.** 만약 MenuLoader가 한 번만 호출될 거라고 가정했다면, 이런 "아직 준비 안 됐을 때의 처리"는 필요 없었을 것이다. **여러 곳에서 독립적으로, 순서를 예측할 수 없이 호출될 것을 염두에 둔 코드**다.

---

## 5. `_onModelReady()` — 내부 ready → 외부 ready 전파

```javascript
_onModelReady: function(oEvent) {
    if (this.mEventRegistry.ready) {
        this.fireEvent("ready", { model: oEvent.getParameter("model") });
    }
}
```

`oStaticModel`이 `setReady(data)`를 호출하면 `oStaticModel` 자체의 `"ready"` 이벤트가 발생한다. `_onModelReady`는 그걸 받아서 **이 MenuLoader 인스턴스**의 `"ready"` 이벤트로 다시 발사한다.

> 주의: `attachEvent("ready", ...)`로 oStaticModel에 등록된 이 핸들러는 **최초 MenuLoader 인스턴스의 것만** 동작한다. 하지만 `fireEvent("ready")`는 각 MenuLoader 인스턴스가 각자의 `mEventRegistry`를 확인하기 때문에, **모든 attachEventOnce 구독자**에게 전달된다.

## 6. 네 곳의 호출부

### 6.1 ① Component.init()

```javascript
// portal_ott/webapp/Component.js (line 56)
init: function() {
    UIComponent.prototype.init.apply(this, arguments);
    // ...
    new MenuLoader().attachEventOnce('ready', (oEvent) => {
        const oModel = oEvent.getParameter("model");
        if (!sap.ui.getCore().getModel("Menu")) {
            sap.ui.getCore().setModel(oModel, "Menu");
        }
        this._setRouteAndTarget(oModel.getData());  // routes, targets, componentUsages 채움
    });
}
```

**하는 일**: 받은 메뉴 데이터로 라우트·타겟·componentUsages를 동적 생성 → `Router.initialize()`

### 6.2 ② BaseController.onInit()

```javascript
// library_ott/src/controller/BaseController.js (line 46)
onInit: function() {
    // ...
    new MenuLoader().attachEventOnce('ready', (oEvent) => {
        const oModel = oEvent.getParameter("model");
        this.getView().setModel(oModel, "Menu");       // 현재 View의 Menu 모델
        if (!sap.ui.getCore().getModel("Menu")) {
            sap.ui.getCore().setModel(oModel, "Menu");  // Core의 전역 Menu 모델
        }
    });
}
```

**하는 일**: UI5 데이터 모델로 등록 → `{Menu>menu_name}` 같은 XML 바인딩을 위해

> 이 코드는 portal의 Layout 뿐만 아니라, **모든 앱(detailPage, myPage 등)의 모든 View에서 상속되어 실행**된다. 그래서 어느 View에서든 `{Menu>...}` 바인딩이 가능하다.

### 6.3 ③ Layout.setHeaderNavigation()

```javascript
// portal_ott/webapp/controller/Layout.controller.js (line 67)
setHeaderNavigation: async function() {
    new MenuLoader().attachEventOnce('ready', async function (oEvent) {
        const oModel = oEvent.getParameter("model");
        const aMenuList = oModel.getData();
        const aFavoriteMenuList = await this.loadFavoriteMenu();
        const oMenuTree = this.buildTree(aMenuList, aFavoriteMenuList);
        this.createHeaderNavigation(oMenuTree);  // 상단 메뉴바 버튼 생성
    }.bind(this));
}
```

**하는 일**: `parent_menu_code`로 부모-자식 트리를 만들고, 상단 `ToolHeader`에 `MenuButton`들을 동적 생성

### 6.4 ④ Layout.setSideNavigation()

```javascript
// portal_ott/webapp/controller/Layout.controller.js (line 111)
setSideNavigation: async function() {
    new MenuLoader().attachEventOnce('ready', async function (oEvent) {
        const oModel = oEvent.getParameter("model");
        const aMenuList = oModel.getData();
        const aFavoriteMenuList = await this.loadFavoriteMenu();
        const oMenuTree = this.buildTree(aMenuList, aFavoriteMenuList);
        this.createNavigationList(oMenuTree);  // 사이드바 메뉴 생성
    }.bind(this));
}
```

**하는 일**: ③과 유사하게 사이드 내비게이션 메뉴 생성

## 7. 전체 실행 타임라인

```
시간 →

① Component.init()
   new MenuLoader()           ← [A 경로] 최초 생성!
     oStaticModel = new JSONWithReadModel()
     _fetchList() 시작 ────────────────────▶ HTTP 요청 발사!
     attachEventOnce('ready', 콜백A)        (콜백A 등록)
     
② BaseController.onInit() (Layout의 부모)
   new MenuLoader()           ← [B 경로] oStaticModel 이미 있음
     _fireWhenReady()
       isReady()? false → 300ms 후 재시도 등록
     attachEventOnce('ready', 콜백B)        (콜백B 등록)
     
③ Layout.setHeaderNavigation()
   new MenuLoader()           ← [B 경로]
     _fireWhenReady()
       isReady()? false → 300ms 후 재시도 등록
     attachEventOnce('ready', 콜백C)        (콜백C 등록)

④ Layout.setSideNavigation()
   new MenuLoader()           ← [B 경로]
     _fireWhenReady()
       isReady()? false → 300ms 후 재시도 등록
     attachEventOnce('ready', 콜백D)        (콜백D 등록)

...시간 경과...

⑤ HTTP 응답 도착!
   _fetchList().then(aContexts => { ... })
     oStaticModel.setReady(sessionData)     // ★ 데이터 저장!
       → oStaticModel의 'ready' 이벤트 발생
       → _onModelReady() → fireEvent("ready", { model: oStaticModel })
       
⑥ 네 개의 콜백 모두 실행 (순서는 등록 순서와 무관)
   콜백A: _setRouteAndTarget(data)          → routes 채움 + Router.initialize()
   콜백B: setModel(oModel, "Menu")          → UI5 Menu 모델 등록
   콜백C: buildTree() + createHeaderNav()   → 상단 메뉴바 생성
   콜백D: buildTree() + createNavList()     → 사이드바 생성
```

> **HTTP 요청은 ①에서 딱 1회.** ②③④는 모두 폴링하다가 ⑤ 이후 같은 데이터를 받는다.

## 8. 왜 이렇게 설계했는가? — 발행-구독 패턴

### 8.1 관심사 분리

만약 한 곳에서 모든 걸 처리하면:

```javascript
// ❌ 결합도 높은 방식
Component.init() {
    new MenuLoader().ready(data => {
        this._setRouteAndTarget(data);       // Component의 일
        this._setMenuModel(data);            // BaseController의 일
        this._buildHeaderMenu(data);         // Layout의 일
        this._buildSideMenu(data);           // Layout의 일
    });
}
```

이러면 Component가 BaseController와 Layout의 내부 구현까지 전부 알아야 한다. 메뉴바 방식을 바꾸려면 Component.js를 수정해야 한다.

### 8.2 발행-구독 패턴 적용

```
                    ┌─────────────────┐
                    │   MenuLoader    │  ← 발행자 (Publisher)
                    │  (oStaticModel) │     HTTP 1회, 데이터 1벌
                    └───────┬─────────┘
                            │  'ready' 이벤트 + 동일한 데이터
            ┌───────────────┼───────────────┬───────────────┐
            ▼               ▼               ▼               ▼
    ┌───────────┐   ┌───────────┐   ┌───────────┐   ┌───────────┐
    │ Component │   │BaseControl.│   │  Layout   │   │  Layout   │
    │   .init() │   │  .onInit() │   │ headerNav │   │  sideNav  │
    └───────────┘   └───────────┘   └───────────┘   └───────────┘
    
    routes 채움      Menu 모델 등록   상단 메뉴바 UI    사이드바 UI
    Router 가동      (View + Core)   동적 생성        동적 생성
```

각 소비자는 **서로의 존재를 전혀 모른다.** 그저 "나는 메뉴 데이터가 필요하니까 MenuLoader한테 준비되면 알려달라고 할게" 하고 끝이다.

### 8.3 장점

| 장점 | 설명 |
|------|------|
| **낮은 결합도** | Component, BaseController, Layout은 서로를 참조하지 않는다 |
| **변경 용이성** | 메뉴바 UI 변경 시 Layout만 수정. Component는 무관 |
| **재사용성** | BaseController의 MenuLoader 구독은 모든 앱(detailPage, myPage 등)에서 자동 상속 |
| **효율성** | HTTP 요청 1회. 각 소비자가 각자 호출해도 중복 통신 없음 |
| **순서 독립성** | 누가 먼저 `new`하든 `_fireWhenReady()` 폴링이 알아서 처리 |

## 9. 같은 데이터, 완전히 다른 네 가지 용도

```
oStaticModel.getData()
  = [ {menu_code:"OTT_MAIN", menu_app_id:"mainPage", ...},
      {menu_code:"OTT_DETAIL", menu_app_id:"detailPage", ...},
      {menu_code:"OTT_MYPAGE", menu_app_id:"myPage", ...},
      ... ]
          │
          │  "피자 한 판을 네 사람이 각자 다르게 먹는다"
          │
    ┌─────┼─────────┬──────────┬──────────┐
    ▼     ▼         ▼          ▼          ▼
   ①     ②         ③          ④

   ① Component: menu_app_id, menu_route_path만 뽑아서 라우트 지도 그림
      → addRoute({pattern: "detailPage/Detail/:content_id:", ...})
      → Router.initialize()

   ② BaseController: 배열 전체를 UI5 Menu 모델로 등록
      → setModel(oModel, "Menu")
      → XML에서 {Menu>menu_name} 바인딩 가능

   ③ setHeaderNavigation: parent_menu_code로 계층 분석 → 상단 메뉴바
      → buildTree() → OTT 그룹 아래 MAIN, DETAIL, MYPAGE...
      → createHeaderNavigation() → <MenuButton>들 생성

   ④ setSideNavigation: 같은 데이터로 사이드바 생성
      → buildTree() → createNavigationList()
```

## 10. 핵심 요약

| # | 내용 |
|---|------|
| 1 | `let oStaticModel`은 모듈 레벨 변수 — 모든 `new MenuLoader()`가 단 하나의 모델을 공유 |
| 2 | 첫 번째 `new MenuLoader()`만 HTTP 요청. 나머지는 `_fireWhenReady()`로 폴링 |
| 3 | `_fireWhenReady()`의 폴링 로직은 "여러 곳에서 독립적 호출"을 **의도한 설계**의 증거 |
| 4 | HTTP 요청은 **1회**, 4곳 모두 같은 `oStaticModel.getData()`를 받는다 |
| 5 | 각 소비자는 같은 데이터를 **완전히 다른 용도**로 사용 (라우트, 모델 등록, 메뉴바 UI, 사이드바 UI) |
| 6 | 발행-구독 패턴으로 관심사 분리 — 서로의 존재를 모르고 독립적으로 개발·수정 가능 |
| 7 | `attachEventOnce` — 각 소비자는 ready 이벤트를 **딱 한 번만** 받는다 |
| 8 | BaseController의 구독은 **모든 하위 앱의 모든 View**에서 자동 상속됨 |
