### 4.7 심층 분석: `_setRouteAndTarget()`의 네 가지 작업

`_setRouteAndTarget()` 안의 네 가지 작업은 **각자 완전히 다른 질문에 답한다.** 하나씩 분해해보자.

#### 배경: Router가 Component 타겟을 만나면 무슨 일이 일어나는가?

Router의 target에는 두 가지 `type`이 있다:

| type | 동작 | 예 |
|------|------|-----|
| `"View"` | XML 파일을 로드해서 App에 꽂음 | Home, Search |
| `"Component"` | **완전히 독립된 앱**을 통째로 생성해서 App에 꽂음 | detailPage, myPage |

`type: "Component"`일 때 Router는 다음 질문들에 답해야 한다:

> "detailPage라는 앱을 로드해야 하는데... **파일이 어디 있지?** **무슨 Component 클래스지?**"

이 질문들에 대한 답이 바로 네 가지 작업이다.

---

#### 작업 ① `sap.ui.loader.config` — "파일 어디 있어?"

```javascript
sap.ui.loader.config({ paths: { "detailPage": "/sysmgt_ott/detailPage/webapp" } });
```

**문제**: Router가 `detailPage.Component`를 로드하려면, UI5 모듈 로더는 파일 경로를 알아야 한다. 기본적으로 portal_ott 자신의 위치(`/portal_ott/webapp/`)를 기준으로 찾으려고 한다:

```
GET /portal_ott/webapp/detailPage/Component.js  → ❌ 404! (없음)
```

실제 파일은 여기 있다:
```
/sysmgt_ott/detailPage/webapp/Component.js
```

**해결**: `sap.ui.loader.config`가 네임스페이스→URL 매핑을 등록한다.

```javascript
sap.ui.loader.config({
    paths: {
        "detailPage": "/sysmgt_ott/detailPage/webapp"
    }
});
```

**의미**: "`detailPage`로 시작하는 모든 모듈 요청은 `/sysmgt_ott/detailPage/webapp`에서 찾아라."

```
변환 전:  detailPage/Component.js
                 ↓ sap.ui.loader.config 변환
변환 후:  /sysmgt_ott/detailPage/webapp/Component.js  → ✅ 찾았다!
```

각 메뉴마다 등록:

```javascript
// Menu.csv에서 가져온 값들로 매핑
// OTT_MAIN    → "mainPage"    → "/sysmgt_ott/mainPage/webapp"
// OTT_DETAIL  → "detailPage"  → "/sysmgt_ott/detailPage/webapp"
// OTT_MYPAGE  → "myPage"      → "/sysmgt_ott/myPage/webapp"
// OTT_BOARD   → "board"       → "/board"
```

---

#### 작업 ② `oRouter.addRoute()` — "URL이 어떻게 생겼어?"

```javascript
oRouter.addRoute({
    pattern: "detailPage/Detail/:content_id:",
    name: "OTT_DETAIL",
    target: { name: "OTT_DETAIL", prefix: "detailPage" }
});
```

**의미**: "URL 해시가 `#detailPage/Detail/C001` 이면 OTT_DETAIL 타겟으로 가라."

- `pattern`: URL 패턴. `:content_id:`는 파라미터 (crossroads 라이브러리 문법)
- `name`: 이 라우트의 이름 (디버깅/로깅용)
- `target.name`: 어떤 타겟을 활성화할지
- `target.prefix`: 타겟의 Component prefix (하위 앱의 namespace)

---

#### 작업 ③ `oRouter.getTargets().addTarget()` — "도착하면 뭘 보여줘?"

```javascript
oRouter.getTargets().addTarget("OTT_DETAIL", {
    type: "Component",     // ← View가 아니라 Component 타입!
    usage: "OTT_DETAIL",   // ← componentUsages에서 찾을 키
    viewId: "detailPage",
    rootView: this.getAggregation("rootControl").getId()
});
```

**의미**: "OTT_DETAIL 타겟이 활성화되면, View가 아니라 **Component**를 생성해서 보여줘. 어떤 Component인지는 `usage: "OTT_DETAIL"`을 키로 componentUsages에서 찾아."

- `type: "Component"` → 이게 핵심. View면 XML 파일을 로드하고 끝이지만, Component면 **완전히 독립된 SAPUI5 앱**을 생성한다 (자체 manifest, Router, 모델을 가짐)
- `usage` → 작업 ④에서 설명할 componentUsages의 키와 연결됨

---

#### 작업 ④ `oConfig.componentUsages[...]` — "그게 어떤 앱이야?"

```javascript
oConfig.componentUsages["OTT_DETAIL"] = {
    name: "detailPage",    // ← 로드할 Component 클래스 이름
    lazy: true             // ← 필요할 때만 로드 (성능 최적화)
};
```

이것이 manifest.json의 `sap.ui5.componentUsages`에 **런타임에** 추가하는 코드다. manifest.json 초기 상태는 텅 비어 있다:

```json
// manifest.json 초기 상태
"componentUsages": {}
```

런타임에 MenuLoader 결과를 받아서 채워진다:

```json
// 런타임에 채워진 상태 (U001 로그인 시)
"componentUsages": {
    "OTT_MAIN":    { "name": "mainPage",    "lazy": true },
    "OTT_DETAIL":  { "name": "detailPage",  "lazy": true },
    "OTT_MYPAGE":  { "name": "myPage",      "lazy": true },
    "OTT_BOARD":   { "name": "board",       "lazy": true }
}
```

**`componentUsages`의 역할**: "usage 이름 `OTT_DETAIL`은 `detailPage.Component`를 로드하라."

Router 내부 동작:
```
① target "OTT_DETAIL" 활성화 → usage: "OTT_DETAIL"
② componentUsages에서 "OTT_DETAIL" 찾음 → name: "detailPage"
③ UI5 모듈 로더: "detailPage" + ".Component" → detailPage.Component 로드
   → ① sap.ui.loader.config 매핑으로 파일 찾음
④ new detailPage.Component({...}) 생성
⑤ portal의 App에 자식 Component로 삽입
```

**왜 manifest.json에 미리 안 써놨을까?** 메뉴는 DB에서 동적으로 결정되기 때문이다:
- U001에게는 SETTLEMENT 메뉴가 없음 → `componentUsages.SETTLEMENT`를 만들 필요 없음
- admin1에게만 SETTLEMENT가 있음 → 그때만 `componentUsages.SETTLEMENT` 생성

---

#### 네 가지 작업 요약

| # | 코드 | 해결하는 질문 | 비유 |
|---|------|--------------|------|
| ① | `sap.ui.loader.config` | **"파일 어디 있어?"** | 지도에 주소 찍기 |
| ② | `oRouter.addRoute()` | **"URL이 어떻게 생겼어?"** | "여기로 오세요" 표지판 |
| ③ | `oRouter.getTargets().addTarget()` | **"도착하면 뭘 보여줘?"** | "이 건물은 카페입니다" |
| ④ | `oConfig.componentUsages[...]` | **"그게 어떤 앱이야?"** | "○○ 프랜차이즈 1호점" |

① 없으면 → 파일을 못 찾아서 404  
② 없으면 → URL 매칭 실패  
③ 없으면 → 타겟이 뭘 보여줄지 모름  
④ 없으면 → Router가 무슨 Component를 로드해야 할지 몰라서 에러

### 4.8 전체 흐름: URL 입력부터 화면 렌더링까지

OTT_DETAIL 메뉴를 예시로, `#detailPage/Detail/C001`이 어떻게 화면이 되는지 추적해보자.

```
사용자가 "컨텐츠 상세" 메뉴 클릭
  → window.location.hash = "detailPage/Detail/C001"
        │
   ┌────┴────┐
   │ Router  │  ② addRoute()로 등록된 패턴과 매칭
   │         │  "detailPage/Detail/:content_id:" → OTT_DETAIL
   └────┬────┘
        │
   ┌────┴────┐
   │ Target  │  ③ addTarget() → type: "Component", usage: "OTT_DETAIL"
   │         │  "Component 타입이니까, usage 키로 componentUsages 뒤져야지"
   └────┬────┘
        │
   ┌────┴────────────┐
   │ componentUsages │  ④ "OTT_DETAIL" → name: "detailPage"
   │                 │  "detailPage.Component를 로드해라"
   └────┬────────────┘
        │
   ┌────┴────────────┐
   │ UI5 모듈 로더    │  ① sap.ui.loader.config → "detailPage" → "/sysmgt_ott/detailPage/webapp"
   │                 │  → GET /sysmgt_ott/detailPage/webapp/Component.js ✅
   │                 │  → GET /sysmgt_ott/detailPage/webapp/manifest.json ✅
   └────┬────────────┘
        │
   ┌────┴────────────┐
   │ detailPage      │  new detailPage.Component()
   │ Component       │  → 자체 manifest.json 파싱 (OData 모델, i18n 등)
   │                 │  → 자체 Router 가동
   │                 │  → "Detail/:content_id:" 패턴 매칭 → content_id = "C001"
   │                 │  → _loadDetail("C001") → 화면 렌더링
   └─────────────────┘
```

**핵심**: portal은 단지 공간(App)과 표지판(Router)만 제공한다. 각 메뉴(앱)은 자체 manifest, Router, OData 모델을 가진 **완전히 독립된 세계**다. portal은 건물주, detailPage는 입점 가게.

## 5. 서비스 생명주기: 언제 SQL이 실행되는가?

### ❌ `cds watch` 시점이 아니다

```bash
cd core_ott && npx cds watch --port 8084
```

이 명령을 치면:
1. CDS 파일을 읽고 **서비스 정의(스키마)**를 메모리에 로드
2. `$metadata` 문서 준비 (엔티티 구조 설명서)
3. HTTP 서버 시작 → **요청 대기 상태**

**SQL은 단 한 줄도 실행되지 않는다.**

### ✅ HTTP 요청 시점에 실행된다

```
MenuLoader._fetchList()
  → GET /srv-api/odata/v4/ott-core/MenuManagement/MenusRoleAppliedList
```

이 HTTP 요청이 도착하는 **그 순간**:

```
① CAP: "MenusRoleAppliedList 요청이 왔네?"
② CAP: "@restrict 있네. $user는 누구지?" → req.user.id = "admin1"
③ CAP: "그럼 AND user_id = 'admin1' 추가해서 SQL 만들어야지"
④ CAP: SQLite에 최종 SQL 전송
⑤ SQLite: JOIN 6개 테이블 + WHERE 필터 → 6행 반환
⑥ CAP: 결과를 JSON으로 직렬화 → HTTP 응답
```

### 타임라인

```
시간 ──────────────────────────────────────────────────────→

[터미널] cds watch                          ← 서비스 정의만 로드, SQL 실행 0회
         │
         │  (아무 일도 안 일어남. 조용히 대기)
         │
[브라우저] index.html 로드                   ← portal_ott 시작
         │
[portal]  new MenuLoader()                  ← MenuLoader 생성
         │
[loader]  _fetchList()                      ← ★ 이 순간!
         │  GET /MenusRoleAppliedList ──────→ SQL 실행! (딱 1회)
         │  ← 6개 메뉴 JSON 응답
         │
[portal]  _setRouteAndTarget(6개 메뉴)       ← 동적 라우트 생성
         │  Router.initialize() → 시동!
         │
         │  (이후로는 같은 URL 재요청 없으면 SQL 실행 안 됨)
```

## 8. 정적 target vs 동적 target 비교

| | manifest.json targets | DB 기반 동적 메뉴 |
|---|---|---|
| 정의 시점 | 빌드타임 (소스코드) | **런타임** (DB 조회 결과) |
| 권한 연동 | ❌ 모든 사용자 동일 | ✅ `$user`로 사용자별 필터링 |
| 변경 방법 | 코드 수정 → 재배포 | DB 데이터만 수정 |
| 사용처 | Home, Search 등 공통 화면 | OTT_MAIN, MYPAGE 등 역할 기반 메뉴 |
| manifest.json 초기값 | targets 정의되어 있음 | `routes: []`, `componentUsages: {}` (텅 빔) |

## 10. 핵심 요약

| # | 내용 |
|---|------|
| 1 | `@(restrict: { where: 'user_id = $user' })` 한 줄이 사용자별 메뉴 필터링의 전부 |
| 2 | JavaScript 코드 어디에도 `if (role === 'Admin')` 같은 분기문 없음 |
| 3 | DB의 7개 테이블은 **3개의 다대다 관계**(User↔RoleGroup, RoleGroup↔Role, Role↔Menu)를 풀기 위한 구조 |
| 4 | `view as select ... join`은 여러 테이블을 합칠 수 있는 유일한 방법. `entity as projection on`은 단일 테이블만 가능 |
| 5 | `@restrict`는 **SQL WHERE 절을 주입**하는 방식. 사후 필터링이 아니라 DB 레벨에서 실행 전에 완료 |
| 6 | `core_user.user_id`를 SELECT에 넣는 이유: CDS View 엔티티에 `user_id` 컬럼이 존재해야 `@restrict`가 참조 가능 |
| 7 | SQL은 `cds watch` 시점이 아니라 **HTTP 요청 시점**에만 실행됨 |
| 8 | MenuLoader가 OData 호출 → CAP이 `$user`로 자동 필터 → 결과 반환 |
| 9 | `_setRouteAndTarget()`의 네 가지 작업은 각자 다른 질문에 답한다: 파일 위치(loader.config), URL 패턴(addRoute), 목적지 유형(addTarget), 앱 신원(componentUsages) |
| 10 | `componentUsages`는 "이 usage 이름은 어떤 Component 클래스를 로드할지"를 정의. `type: "Component"` 타겟이 활성화될 때 Router가 여기서 찾음 |
| 11 | `sap.ui.loader.config`는 네임스페이스→URL 매핑. "detailPage" → "/sysmgt_ott/detailPage/webapp" 변환으로 실제 파일을 찾음 |
| 12 | manifest.json의 `routes: []`, `componentUsages: {}`는 초기에 텅 비어 있다가 런타임에 DB 결과로 채워짐 |
| 13 | 모든 동적 라우트가 `addRoute()`로 채워진 **마지막에** `Router.initialize()` 호출 — 빈 routes로 매칭할 위험 없음 |
| 14 | portal은 건물주(공간+표지판), 각 메뉴 앱은 독립 입점 가게(자체 manifest, Router, OData 모델) |
| 15 | 권한 기반 메뉴는 전적으로 DB + CAP `@restrict` + 동적 Router로 구현 |
