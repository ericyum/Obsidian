# 1. core_ott — 공통 백엔드 (인증·메뉴·사용자)

> **작성일**: 2026-07-13
> **역할**: ZEN-OTT 전체 서비스의 기반이 되는 공통 백엔드. 사용자 인증, 메뉴 관리, 권한, 공통 코드를 담당하며 **모든 서비스 중 가장 먼저 배포**되어야 합니다.

---

## 📂 모듈 개요

```
cap-node/core_ott/
├── .cdsrc.json          ← CAP 프로젝트 설정 (DB, 인증)
├── mta.yaml             ← MTA 배포 정의 (가장 중요!)
├── package.json         ← npm 스크립트 + 의존성
├── index.cds            ← CDS 진입점
├── db/data/             ← BTP HANA 배포용 CSV 데이터
├── srv/server.ts        ← CDS 서버 커스텀 로직 (i18n, x-forwarded-for 패치)
└── tsconfig.json        ← TypeScript 설정
```

## 🔑 핵심 배포 파일들

### ① `.cdsrc.json` — 프로덕션 vs 개발 환경 분기

```json
{
  "requires": {
    "db": { "kind": "hana" },              // ← 프로덕션(BTP): HANA DB
    "[development]": {
      "db": { "kind": "sqlite", "credentials": { "url": ":memory:" } },  // ← 로컬: SQLite in-memory
      "auth": {
        "kind": "mocked",
        "users": {
          "admin1": { "roles": ["Admin", "authenticated-user"] },
          "U001": { "roles": ["authenticated-user"] },
          ...
        }
      }
    }
  }
}
```

**핵심**: CAP은 `[development]` 프로파일을 통해 환경별 설정을 분기합니다.
- 로컬 `cds watch` → SQLite 인메모리 DB + Mock 인증
- BTP `cds build --production` → HANA DB + XSUAA JWT 인증

`"kind": "hana"`가 상단에 선언되어 있어, `--production` 플래그 없이도 프로덕션 빌드 시 HANA로 설정됩니다.

### ② `mta.yaml` — MTA 배포 정의

```yaml
_schema-version: 3.3.0
ID: com-cap-ott-core
version: 1.0.0
description: "OTT common backend services (Menu, User, Code, Message)."
parameters:
  enable-parallel-deployments: true

build-parameters:
  before-all:
    - builder: custom
      commands:
        - npm install
        - npx cds build --production     # ← CDS → HANA 변환
        - tsc -p tsconfig.cdsbuild.json  # ← TypeScript 컴파일

modules:
  # 모듈 1: Node.js 서버 (CAP 서비스)
  - name: com-cap-ott-core
    type: nodejs
    path: gen/srv                          # ← CDS 빌드 결과물
    parameters:
      instances: 1
      buildpack: nodejs_buildpack
    build-parameters:
      builder: npm-ci
    provides:
      - name: srv-api
        properties:
          srv-url: ${default-url}          # ← 이 URL이 BTP Destination에 등록됨
    requires:
      - name: com-ott-core-hdi-container   # ← HANA DB 연결
      - name: com-cap-ott-core-destination  # ← Destination 서비스 (approuter와 공유)
      - name: com-cap-ott-core-xsuaa        # ← XSUAA 인증 (approuter와 공유)

  # 모듈 2: HANA DB Deployer
  - name: com-ott-core-db-deployer
    type: hdb
    path: gen/db                           # ← CDS → HANA 테이블 정의
    parameters:
      buildpack: nodejs_buildpack
    requires:
      - name: com-ott-core-hdi-container

resources:
  - name: com-ott-core-hdi-container
    type: com.sap.xs.hdi-container         # ← HANA HDI 컨테이너 (DB)
    parameters:
      service: hana
      service-plan: hdi-shared

  - name: com-cap-ott-core-destination
    type: org.cloudfoundry.existing-service # ← approuter가 먼저 만든 서비스 재사용
    parameters:
      service-name: com-app-destination-${space}

  - name: com-cap-ott-core-xsuaa
    type: org.cloudfoundry.existing-service # ← approuter가 먼저 만든 서비스 재사용
    parameters:
      service-name: com-app-xsuaa-${space}
```

**핵심 포인트**:

| 항목 | 설명 |
|------|------|
| `cds build --production` | CDS 모델을 HANA DB 테이블 + Node.js 서버 코드로 변환 |
| `gen/srv` | CDS 빌드 결과물 폴더. 실제로 배포되는 Node.js 앱 |
| `gen/db` | HANA 테이블 정의. DB Deployer가 이걸로 HDI 컨테이너를 채움 |
| `hdi-shared` | HANA DB를 "컨테이너" 단위로 독립 관리. core_ott 전용 HDI |
| `existing-service` | approuter_ott가 배포할 때 만든 서비스 인스턴스를 **재사용**. Destination과 XSUAA는 앱 전체가 하나만 있음 |
| `srv-url: ${default-url}` | 배포 후 자동으로 할당된 CF URL. BTP Cockpit의 Destination 설정에 이 URL을 넣어야 함 |

### ③ `package.json` — 빌드·배포 스크립트

```json
{
  "name": "@com/cap-ott-core",
  "dependencies": {
    "@cap-js-community/odata-v2-adapter": "^1.15.1",  // V2 OData 지원
    "@cap-js/hana": "^2.2.0",                          // HANA DB 드라이버
    "@sap/cds": "^9.3.0",                              // CAP 런타임
    "@sap/xssec": "^4.8.0"                             // XSUAA JWT 검증
  },
  "scripts": {
    "start": "cds-serve",
    "watch": "cds watch --port 8084",                  // ← 로컬 개발
    "build": "cds build",
    "build:cf": "npm run build:cds && npm run cleanup:ts && npm run build:ts",
    "build:mbt": "mbt build",                          // ← MTA 아카이브 생성
    "deploy:cf": "cf deploy mta_archives/com-cap-ott-core_1.0.0.mtar",
    "deploy:mta": "npm run build:mbt && npm run deploy:cf"  // ← 전체 배포
  }
}
```

**배포 명령어**:
```bash
cd cap-node/core_ott
npm run deploy:mta
# → mbt build (CDS 빌드 + MTA 아카이브 생성)
# → cf deploy (Cloud Foundry에 배포)
```

**스크립트 체인** (mbt build 내부):
```
npm install
npx cds build --production   ← CDS → HANA 테이블 + Node.js 코드
tsc -p tsconfig.cdsbuild.json ← TypeScript 컴파일
→ mta_archives/com-cap-ott-core_1.0.0.mtar 생성
```

### ④ `srv/server.ts` — 서버 커스텀 로직

```typescript
// 1. x-forwarded-for 패치 (중요!)
cds.on('bootstrap', (app) => {
    app.use((req, res, next) => {
        if (req.headers['x-forwarded-for'] === 'undefined') {
            delete req.headers['x-forwarded-for'];
        }
        next();
    });
});

// 2. 동적 i18n 엔드포인트
cds.on('served', () => {
    // /i18n_KO.properties  → DB Message 테이블에서 한글 메시지 조회
    // /i18n_EN.properties  → DB Message 테이블에서 영문 메시지 조회
    // /i18n.properties     → 기본값 KO
});
```

**두 가지 역할**:
1. `x-forwarded-for` 패치 — 로컬 approuter가 `"undefined"` 문자열을 보내면 제거 (없으면 $batch 500 에러)
2. 동적 i18n — DB의 Message 테이블에서 실시간으로 properties 파일 생성

### ⑤ `db/data/` — BTP HANA CSV 데이터

```
db/data/
├── Menu.csv           ← 메뉴 구조 (OTT_DETAIL, OTT_MYPAGE 등)
├── Role.csv           ← 권한 정의
├── RoleGroup.csv      ← 권한 그룹
├── Role_Menu.csv      ← 권한별 접근 가능 메뉴
├── Role_RoleGroup.csv ← 권한-그룹 매핑
└── User_RoleGroup.csv ← 사용자-그룹 매핑
```

**중요**: `db/data/`의 CSV는 **BTP HANA 배포 시에만** 로드됩니다. 로컬 SQLite에서는 `test/data/`를 사용합니다. User.csv는 PII(이메일) 포함 가능성이 있어 `db/data/`에 직접 넣지 않습니다.

## 🔄 core_ott가 제공하는 서비스들

| 서비스 | 용도 | approuter의 라우트 |
|--------|------|-------------------|
| UserManagement | 사용자 조회·인증 | `odata/v4/ott-core/UserManagement` |
| MenuManagement | 메뉴 구조·권한 | `odata/v4/ott-core/MenuManagement` |
| CommonCode | 공통 코드 | `odata/v4/ott-core/CommonCode` |
| Message (i18n) | 다국어 메시지 | `/i18n_KO.properties` |

이 모든 요청은 approuter의 `onCapService` destination을 통해 core_ott로 전달됩니다.

## ⚠️ 배포 시 주의사항

| # | 주의사항 |
|---|---------|
| 1 | `existing-service`는 **approuter_ott가 먼저 배포**되어 있어야 함. 그러므로 approuter_ott를 최소 한 번은 먼저 배포한 후 core_ott 배포 |
| 2 | `db/data/`에 PII(개인정보) 포함 CSV 절대 커밋 금지 |
| 3 | 포트: 로컬은 **8084** (ott의 8083과 겹치지 않도록) |
| 4 | `hdbtable` deploy-format — HANA Cloud 호환 |
