# 4. portal_ott — 포털·셸 (메뉴·레이아웃)

> **작성일**: 2026-07-13
> **역할**: ZEN-OTT의 진입점이자 모든 앱을 담는 셸(Shell). 헤더, 메뉴바, 레이아웃을 제공하고 하위 앱(detailPage, myPage, mainPage, board 등)을 Router로 전환합니다. **approuter_ott의 catch-all 타겟**으로, 모든 트래픽의 최종 목적지입니다.

---

## 📂 모듈 개요

```
cap-app/portal_ott/
├── mta.yaml             ← MTA 배포 정의
├── package.json         ← npm 스크립트
```

## 🔑 핵심 배포 파일들

### ① `mta.yaml` — MTA 배포 정의

```yaml
_schema-version: "3.2"
ID: comportalott
version: 1.0.0
parameters:
  deploy_mode: html5-repo
  enable-parallel-deployments: true

modules:
  - name: com-portalott-content
    type: com.sap.application.content
    path: .
    requires:
      - name: com-portalott-repo-host
        parameters:
          content-target: true
    build-parameters:
      build-result: resources
      requires:
        - name: comportalott
          artifacts:
            - comportalott.zip
          target-path: resources/

  - name: comportalott
    type: html5
    path: .
    build-parameters:
      build-result: dist
      builder: custom
      commands:
        - npm install
        - npm run build:cf

resources:
  - name: com-portalott-repo-host
    type: org.cloudfoundry.managed-service
    parameters:
      service: html5-apps-repo
      service-plan: app-host
      service-name: com-portalott-repo-host-${space}
```

**다른 프론트엔드 모듈과 동일한 패턴**입니다:
1. `npm install` + `npm run build:cf` 로 UI5 번들링
2. `ui5-task-zipper`로 zip 생성
3. MTA 빌드 → html5-apps-repo에 업로드

### ③ `ui5-deploy.yaml` — UI5 배포 빌드 설정

```yaml
specVersion: '3.0'
metadata:
  name: comportalott
type: application
resources:
  configuration:
    propertiesFileSourceEncoding: UTF-8
builder:
  resources:
    excludes:
      - "/test/**"
      - "/localService/**"
  customTasks:
    - name: webide-extension-task-updateManifestJson   # ⚠️⚠️⚠️ 위험!
      afterTask: replaceVersion
      configuration:
        appFolder: webapp
        destDir: dist
    - name: ui5-task-zipper
      afterTask: generateCachebusterInfo
      configuration:
        archiveName: comportalott
        additionalFiles:
          - xs-app.json
```

> ⚠️ **중요 경고**: `webide-extension-task-updateManifestJson`가 있습니다! 이 태스크는 manifest.json의 OData URI 앞 `/`를 제거하여 **절대경로를 상대경로로 바꿔버립니다.** portal_ott 자체는 OData URI를 직접 사용하지 않아 영향이 적을 수 있지만, **안전을 위해 제거하는 것이 좋습니다.**

**이 태스크의 영향**:
```
빌드 전: "uri": "/srv-api/odata/v4/ott-core/MenuManagement"
빌드 후: "uri": "srv-api/odata/v4/ott-core/MenuManagement"  ← 앞에 /가 사라짐!
```

### ④ `ui5.yaml` — 로컬 개발용 설정

```yaml
specVersion: "3.1"
metadata:
  name: comportalott
type: application
server:
  customMiddleware:
    - name: fiori-tools-proxy
      afterMiddleware: compression
      configuration:
        backend:
          - path: /srv-api
            url: http://localhost:8080
    - name: fiori-tools-servestatic
      afterMiddleware: compression
      configuration:
        paths:
          - path: /common_ott.lib
            src: "../library_ott/src"         # ← 공유 라이브러리 경로
          - path: /sysmgt
            src: "../sysmgt/webapp"
```

**`ui5.yaml` vs `ui5-deploy.yaml`**:

| 파일 | 용도 | 사용 시점 |
|------|------|-----------|
| `ui5.yaml` | 로컬 개발 서버 설정 | `ui5 serve`, `fiori run` |
| `ui5-deploy.yaml` | BTP 배포 빌드 설정 | `ui5 build preload --config ui5-deploy.yaml` |

로컬에서는 `fiori-tools-servestatic`으로 `common_ott.lib`와 `sysmgt`를 디스크에서 직접 서빙합니다. BTP에서는 approuter의 `xs-app-dev.json`이 이 역할을 대신합니다.

### ⑤ `xs-app.json` — 정적 파일 라우팅

```json
{
  "welcomeFile": "/index.html",
  "authenticationMethod": "none",
  "routes": [
    {
      "source": "^/resources/(.*)$",
      "target": "/resources/$1",
      "authenticationType": "none",
      "destination": "ui5"
    },
    {
      "source": "^/test-resources/(.*)$",
      "target": "/test-resources/$1",
      "authenticationType": "none",
      "destination": "ui5"
    },
    {
      "source": "^(.*)$",
      "target": "$1",
      "service": "html5-apps-repo-rt",
      "authenticationType": "none"
    }
  ]
}
```

> 이 파일은 각 모듈이 **독립 배포**(자체 mta.yaml로 `deploy:mta`)될 때만 사용됩니다. approuter_ott의 통합 배포 시에는 approuter의 `xs-app-dev.json`이 모든 라우팅을 처리하기 때문에 이 파일은 사용되지 않습니다.

## 🔄 portal_ott의 역할 — 건물주

portal_ott는 "건물주" 역할입니다:

```
portal_ott (건물주)
├── Layout (헤더, 메뉴바) ← portal이 직접 관리
└── <App id="app"> (빈 공간)
     ├── Home.view             ← portal Router가 직접 관리
     ├── detailPage Component  ← 독립 입점 가게
     ├── myPage Component      ← 또 다른 독립 가게
     ├── board Component
     ├── settlement Component
     └── trendAnalysis Component
```

**핵심**: portal은 공간만 제공하고, 각 하위 앱은 자체 Router·manifest·OData 모델로 완전히 독립 작동합니다.

---

## 🔗 approuter와의 연결

```json
// approuter_ott/xs-app-dev.json
{
  "source": "^.*/portal_ott/(.*)$",
  "target": "/comportalott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa"
},
{
  "source": "(.*)",               // ← catch-all: 모든 경로
  "target": "/comportalott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa"
}
```

**catch-all 라우트** 덕분에 어떤 URL로 접속해도 최종적으로 portal_ott이 로드됩니다. portal의 Router가 URL 해시(`#`)를 읽고 적절한 하위 앱을 표시합니다.

## ⚠️ 배포 시 주의사항

| # | 주의사항 |
|---|---------|
| 1 | `ui5-deploy.yaml`에서 **`webide-extension-task-updateManifestJson` 제거** 권장 |
| 2 | portal은 approuter의 **catch-all 타겟** — 항상 마지막 라우트로 등록 |
| 3 | library_ott보다 **나중에** 배포 (의존성) |
| 4 | `service-name`에 `${space}` 포함 — dev/qa/prod 환경 구분 |
| 5 | `menu_display_flag` 필터링이 Layout.controller.js에 적용되어 있어야 메뉴 숨김 정상 작동 |
