# 5. sysmgt_ott — OTT 핵심 기능 (detailPage·myPage·mainPage)

> **작성일**: 2026-07-13
> **역할**: OTT의 핵심 사용자 기능을 담은 모듈. 컨텐츠 상세 페이지(detailPage), 마이페이지(myPage), 메인 목록(mainPage)을 포함합니다. **중첩 앱 구조**가 특징입니다.


## 🔑 핵심 배포 파일들

### ① `mta.yaml` — MTA 배포 정의

```yaml

  deploy_mode: html5-repo
  enable-parallel-deployments: true

    path: .

      - name: com-sysmgtott-repo-host

          content-target: true
    build-parameters:
      build-result: resources
      requires:
        - name: comsysmgtott
          artifacts:
            - comsysmgtott.zip
          target-path: resources/

  - name: comsysmgtott
    type: html5
    path: .

      build-result: dist
      builder: custom

        - npm run build:cf

  - name: com-sysmgtott-repo-host
    type: org.cloudfoundry.managed-service

      service: html5-apps-repo
      service-plan: app-host
```

### ③ `ui5-deploy.yaml` — UI5 배포 빌드 설정

```yaml
specVersion: '3.0'
metadata:
  name: comsysmgtott
type: application
resources:
  configuration:
    propertiesFileSourceEncoding: UTF-8
builder:
  resources:
    excludes:
      - "/test/**"
      - "/localService/**"
      - "/**/node_modules/**"
      - "/*/node_modules/**"
  customTasks:
    - name: ui5-task-zipper
      afterTask: generateCachebusterInfo
      configuration:
        archiveName: comsysmgtott
        additionalFiles:
          - xs-app.json
```

> ✅ **좋은 소식**: sysmgt_ott의 `ui5-deploy.yaml`에는 `webide-extension-task-updateManifestJson`가 **이미 제거**되어 있습니다! OData URI의 `/`가 보존되어 BTP에서도 정상 작동합니다.

**`webide-extension-task-updateManifestJson` 제거 전/후**:

| 상태 | manifest.json URI | 결과 |
|------|-------------------|------|
| 제거 전 | `"srv-api/odata/v2/detail/"` ← `/` 없음 | BTP에서 404 (상대경로로 잘못 해석) |
| 제거 후 | `"/srv-api/odata/v2/detail/"` ← `/` 있음 | BTP에서 정상 작동 (절대경로) |

## 🧩 중첩 앱 구조 — sysmgt_ott만의 특별한 점

sysmgt_ott는 **하나의 MTA 모듈 안에 여러 독립 앱**이 들어 있습니다:

```
sysmgt_ott/webapp/
├── detailPage/webapp/
│   ├── manifest.json     ← sap.app.id: "detailPage"
│   ├── Component.js      ← detailPage.Component
│   └── controller/DetailMain.controller.js
├── myPage/webapp/
│   ├── manifest.json     ← sap.app.id: "myPage"
│   ├── Component.js      ← myPage.Component
│   └── controller/MyMembership.controller.js
└── mainPage/webapp/
    ├── manifest.json
    ├── Component.js
    └── controller/Main.controller.js
```

**각 하위 앱은**:
- 자체 `manifest.json` (독자적인 `sap.app.id`)
- 자체 `Component.js`
- 자체 Router, OData 모델, i18n

**배포 관점의 영향**:
- `ui5 build preload`는 sysmgt_ott **루트**의 `Component-preload.js`만 생성
- 하위 앱(detailPage, myPage, mainPage)의 `Component-preload.js`는 생성되지 않음
- BTP에서 UI5가 하위 앱의 preload를 요청 → 404 → 자동으로 개별 파일로 폴백 → **기능 정상 작동**

## 🔗 approuter와의 연결

```json
// approuter_ott/xs-app-dev.json
{
  "source": "^.*/sysmgt_ott/~[0-9]+~/(.*)$",    // ← 캐시 무효화 토큰 포함
  "target": "/comsysmgtott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa"
},
{
  "source": "^.*/sysmgt_ott/(.*)$",             // ← 일반 요청
  "target": "/comsysmgtott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa"
}
```

**두 가지 source 패턴**:
1. `~[0-9]+~` — 캐시 무효화 토큰 포함 URL (SAP 표준)
2. 일반 경로 — 런타임 파일 요청

## ⚠️ 배포 시 주의사항

| # | 주의사항 |
|---|---------|
| 1 | `webide-extension-task-updateManifestJson`가 `ui5-deploy.yaml`에 **없음**을 확인 (이미 제거된 상태) |
| 2 | 하위 앱의 `Component-preload.js` 404는 **정상** — UI5 자동 폴백 |
| 3 | 각 하위 앱(detailPage, myPage, mainPage)의 manifest.json에 `"async": true` 필수 (portal 연동) |
| 4 | `manifest.json`의 OData URI는 반드시 `/`로 시작하는 절대경로 |
