### 🔴 #1 — Dialog Memory Leak (매번 새 Dialog 생성, destroy 누락)

| 항목           | 내용                                                                |
| ------------ | ----------------------------------------------------------------- |
| **Severity** | **High**                                                          |
| **Location** | `MyMembership.controller.js` / `_showConfirmDialog()` / line ~104 |
| **Category** | Memory Leak & Retained Reference                                  |

**Issue**:
```javascript
var oDialog = new Dialog({ ... });
oDialog.open();
// close()만 호출하고 destroy()는 호출 안 함
```

확인·취소 버튼 press 핸들러에서 `oDialog.close()`만 호출하고 있다. `close()`는 DOM에서 다이얼로그를 숨길 뿐, UI5 컨트롤 인스턴스를 파괴하지 않는다. 사용자가 해지·재개 다이얼로그를 반복해서 열 때마다 Dialog + Button + Text 객체가 메모리에 누적된다.

**Technical Cause**: `sap.m.Dialog.close()` ≠ `destroy()`. UI5 컨트롤은 명시적으로 `destroy()`를 호출해야 GC 대상이 된다.

**Potential Runtime Impact**: 장시간 사용 시 DOM 노드 누적 → 메모리 사용량 증가 → UI 응답성 저하. SPA 특성상 페이지를 떠나지 않고 Dialog를 여러 번 열 수 있어 누적 위험이 크다.

**Estimated Frequency**: Always (사용자가 해지/재개 다이얼로그를 열 때마다)

**Confidence**: High

**Recommended Fix**:
```javascript
// press 핸들러에서
oDialog.close();
oDialog.destroy();  // 추가
```

또는 `afterClose` 이벤트 사용:
```javascript
var oDialog = new Dialog({
    // ...
    afterClose: function () {
        oDialog.destroy();
    }
});
```

**Estimated Improvement**: 장기 사용 시 메모리 누적 방지. 반복 다이얼로그 사용 시 heap memory 사용량 5~10% 감소 예상.

### 🟡 #2 — `console.log` 프로덕션 코드 잔존

| 항목 | 내용 |
|------|------|
| **Severity** | **Medium** |
| **Location** | `BaseController.js` / `onInit()` → MenuFunctionLoader callback / line ~56 |
| **Category** | Resource Lifecycle / Log Accumulation |

**Issue**:
```javascript
new MenuFunctionLoader().attachEventOnce('ready', (oEvent) => {
    // ...
    this._oMenuFunction = oModel.getProperty(`/${sMenuId}`) || {};
    console.log(this._oMenuFunction);  // ← 프로덕션 코드에 남은 로그
});
```

매번 컨트롤러가 초기화될 때마다 `console.log`가 실행된다.

**Technical Cause**: 디버깅용 로그가 제거되지 않음.

**Potential Runtime Impact**: 프로덕션 환경에서 불필요한 콘솔 출력. Browser DevTools가 닫혀 있어도 `console.log`는 실행되며, 대량 호출 시 미세한 성능 영향.

**Estimated Frequency**: Always (컨트롤러 초기화 시마다)

**Confidence**: High

**Recommended Fix**: 삭제하거나 `if (window.DEBUG) console.log(...)` 등으로 조건부 처리.

### 🟡 #3 — Dialog 확인 버튼 연타 시 중복 Action 호출

| 항목 | 내용 |
|------|------|
| **Severity** | **Medium** |
| **Location** | `MyMembership.controller.js` / `_showConfirmDialog()` / line ~112 |
| **Category** | Async Safety & Race Conditions |

**Issue**:
```javascript
press: function () {
    that._callAction(sAction);  // 확인 버튼 누를 때마다 OData 호출
    oDialog.close();
}
```

확인 버튼에 **중복 클릭 방지**가 없다. 사용자가 버튼을 빠르게 2번 누르면 `_callAction`이 2번 호출되어 OData action이 중복 실행될 수 있다.

**Technical Cause**: 버튼 press 핸들러에 guard condition 없음. Dialog가 close되기 전에 재클릭 가능.

**Potential Runtime Impact**: 구독 해지/재개가 중복 호출 → 백엔드 race condition 가능성. UI5의 `Dialog.close()`가 즉시 동작하지 않을 수 있어 연타 window가 존재함.

**Estimated Frequency**: Rare (사용자 연타 시)

**Confidence**: Medium

**Recommended Fix**:
```javascript
var bActionCalled = false;
press: function () {
    if (bActionCalled) return;
    bActionCalled = true;
    that._callAction(sAction);
    oDialog.close();
}
```

### 🟡 #4 — `setTimeout` 타이머 ID 미보관 (중복 타이머 가능성)

| 항목 | 내용 |
|------|------|
| **Severity** | **Low** |
| **Location** | `DetailMain.controller.js` / `_checkMyReview()` / line ~76 |
| **Category** | Memory Leak & Retained Reference |

**Issue**:
```javascript
if (!oUserModel) {
    this._checkRetryCount = (this._checkRetryCount || 0) + 1;
    if (this._checkRetryCount <= 5) {
        setTimeout(this._checkMyReview.bind(this), 200);  // 타이머 ID 저장 안 함
    }
    return;
}
```

`_checkMyReview`가 빠르게 연속 호출되면(예: `_onRouteMatched` + `afterSubmit`), 여러 개의 setTimeout이 동시에 등록될 수 있다. 최대 5회 제한은 있지만, 여러 호출이 각각 5회씩 타이머를 등록할 수 있다.

**Technical Cause**: 타이머 ID를 `this._retryTimer` 등에 저장하고, 새 재시도 전에 기존 타이머를 해제하지 않음.

**Potential Runtime Impact**: 드문 케이스지만, `_loadDetail`이 완료되기 전에 다른 컨텐츠로 이동하면 이전 컨텐츠의 타이머가 계속 실행될 수 있음.

**Estimated Frequency**: Rare (빠른 연속 내비게이션 시)

**Confidence**: Low

**Recommended Fix**:
```javascript
clearTimeout(this._retryTimer);
this._retryTimer = setTimeout(this._checkMyReview.bind(this), 200);
```

### 🟢 #5 — `fnDeepCopy` 재귀 깊이 제한 없음

| 항목 | 내용 |
|------|------|
| **Severity** | **Low** |
| **Location** | `BaseController.js` / `fnDeepCopy()` / line ~133 |
| **Category** | CPU Hotspot / Stack Overflow |

**Issue**:
```javascript
fnDeepCopy : function(oData){
    for(var key in oData){
        if(oData.hasOwnProperty(key)){
            oNewObj[key] = this.fnDeepCopy(oData[key]);  // 무제한 재귀
        }
    }
}
```

순환 참조가 있는 객체를 복사하면 **무한 재귀 → Stack Overflow** 발생.

**Technical Cause**: 순환 참조 감지 로직 없음. `Date`만 특별 처리.

**Potential Runtime Impact**: 현재 코드베이스에서는 순환 참조 객체가 전달될 가능성이 낮음 (주로 OData 응답 flat 객체). 그러나 추후 OData expand된 중첩 데이터에서 발생 가능.

**Estimated Frequency**: Rare

**Confidence**: Low

**Recommended Fix**:
```javascript
fnDeepCopy: function(oData, aSeen = []) {
    if (aSeen.includes(oData)) return oData;
    aSeen.push(oData);
    // ... 재귀 시 aSeen 전달
}
```
