# OTT 권한 체계 맵핑 — roleGroup > role > user > menu

> **분석 담당**: 안효서
> **백엔드 담당**: core 팀 (UserManagement / MenuManagement 서비스)

> [!summary] 한눈에
>
> - **목적**: 기존 소스에서 권한 체계를 읽어낸 맵핑 — 배포 환경 approuter에 화면을 붙이는 다음 작업의 설계 기준.
>
> - **핵심 체인**: `User → RoleGroup → Role → Menu`(사용자 로그인 시 볼 메뉴·접근 데이터를 결정하는 줄기).
>
> - **구현 기준**: 2026-07-03.

> [!note]- 읽기 전 — 소스·서비스
>
> - **소스**: `zen-ott-onboarding`(core = `cap-node/core`, ott = `cap-node/ott`, 공용 라이브러리 = `cap-app/library_ott`, approuter = `cap-app/approuter_ott`)
>
> - **서비스**: `UserManagement` / `MenuManagement`(core)
>
> - **참고**: [[OTT-포털-흰화면-메뉴미출력-이슈로그]] — 빈 배열 = 시드 문제(권한 체계 동작 원리의 실사례)

## 0. 한눈에 보기

사용자 한 명이 로그인했을 때, 어떤 메뉴를 볼 수 있고 어떤 데이터에 손을 댈 수 있는지를 결정하는 줄기는 하나다.

```text
User ──▶ RoleGroup ──▶ Role ──▶ Menu
```

- 사용자는 **역할 그룹(RoleGroup)** 에 속하고

- 역할 그룹은 여러 **역할(Role)** 을 묶고

- 역할은 여러 **메뉴(Menu)** 와 연결된다

- 메뉴가 곧 화면(UI5 앱) 이다

이 줄기를 따라 사용자에게 도달하는 메뉴만 화면에 뜬다. 그리고 데이터 단위의 통제는 CDS의 `$user` 제약과 XSUAA의 롤이 나눠 갖는다. **프론트는 권한을 분기하지 않는다 — 백엔드가 다 결정한다.**

---

## 1. 데이터 계층 — 다섯 줄기의 연결

`cap-node/core/db/cds/` 아래 엔티티들이 줄기를 이룬다.

```text
User
  └─(User_RoleGroup)─▶ RoleGroup
                        └─(Role_RoleGroup)─▶ Role
                                              ├─(Role_Menu)─▶ Menu ──▶ MenuFunction
                                              └─(Role_MenuFunction)
```

| 엔티티              | 키                               | 역할                            | SCIM 연계 필드                      |
| ---------------- | ------------------------------- | ----------------------------- | ------------------------------- |
| `User`           | `user_id`                       | 로그인 사용자                       | `scim_id`                       |
| `User_RoleGroup` | `user_id` + `role_group_code`   | 사용자 ↔ 역할 그룹 연결                | `scim_id`, `scim_role_group_id` |
| `RoleGroup`      | `role_group_code`               | 역할을 묶는 단위                     | `scim_role_group_id`            |
| `Role_RoleGroup` | `role_group_code` + `role_code` | 역할 그룹 ↔ 역할 연결                 | —                               |
| `Role`           | `role_code`                     | 권한의 단위                        | —                               |
| `Role_Menu`      | `role_code` + `menu_code`       | 역할 ↔ 메뉴 연결, `use_flag` 로 켜고 끔 | —                               |
| `Menu`           | `menu_code`                     | 화면(UI5 앱) 단위, 부모 메뉴로 트리       | —                               |
| `MenuFunction`   | —                               | 메뉴 안의 기능(버튼) 단위 권한            | —                               |

**Menu 에는 UI5 앱을 연결하는 네 필드가 있다**
— 이게 메뉴를 실제 화면으로 이어준다.

- `menu_app_id` — UI5 앱 식별자

- `menu_repo_path` — html5-apps-repo 경로

- `menu_app_path` — 앱 내부 경로

- `menu_route_path` — 라우트 경로

## 2. 인증 — 두 축이 역할을 나눠 갖는다

권한 통제는 XSUAA 와 CDS 두 축이 나눠 맡는다.

### 2-1. XSUAA — 역할 컬렉션 단위

`cap-node/ott/xs-security.json`:

```text
scope          : $XSAPPNAME.Admin
role-template  : Admin
role-collection: OTT_Settlement_Admin
```

- BTP Cockpit 에서 사용자에게 `OTT_Settlement_Admin` role-collection 을 할당한다.

- 할당된 사용자의 JWT 에 `Admin` scope 이 실린다.

### 2-2. CDS `$user` — 데이터 행 단위

CDS 의 `$user` 의사변수는 로그인한 사용자의 `user_id` 로 치환된다. `@(restrict: ...)` 로 본인 데이터만 걸러낸다.

```cds
@(restrict: [{ grant: 'READ', where: 'user_id = $user' }])
```

서비스/엔티티 단위 통제는 `requires`:

```cds
@(requires: 'Admin')              // Admin scope 필요
@(requires: 'authenticated-user') // 로그인만 하면 됨
```

**두 축의 분업**
— XSUAA 는 "이 사람이 이 역할을 가지는가", CDS `$user` 는 "이 행이 이 사람의 것인가". 한 줄기 체계에 두 질문이 겹쳐 작동한다.

## 3. 서비스 — 백엔드가 권한을 계산한다

`/srv-api/odata/v2|v4/ott-core/...` 경로. 핵심은 **프론트가 아니라 백엔드가 사용자의 역할 줄기를 따라 메뉴를 걸러준다** 는 점.

### 3-1. UserManagement

| 엔티티 | 통제 | 비고 |
|---|---|---|
| `UserSet` | — | 사용자 CRUD. `before('CREATE')` 이메일 중복 검증만 |
| `UserWithRoles` | — | `string_agg` 로 role_group_codes 한 줄 |
| **`UserSessionInfo`** | `user_id = $user` | 로그인 사용자 본인 1건. **handler 없음** — CDS restrict 로 자동 동작 |

`UserSessionInfo` 는 보드(기능6) 의 본인확인에 쓰인다. 프론트가 이걸 읽어 "이 글이 내 글인가"를 판단하는 게 아니라, 백엔드 restrict 가 본인 것만 내보낸다.

### 3-2. MenuManagement

| 엔티티 | 통제 | 비고 |
|---|---|---|
| `MenuSet` | — | 메뉴 마스터 |
| **`MenusRoleAppliedList`** | `user_id = $user` | **핵심** — 6개 조인으로 역할 기반 메뉴 자동 필터 |
| `MenuFunctionAppliedList` | `user_id = $user` | 메뉴별 기능(버튼) 권한. `row_number` 파티션으로 중복 제거, `coalesce(rmf.use_flag, mf.default_use_flag)` 로 우선순위 처리 |
| `FavoriteRoleAppliedList` | `user_id = $user` | 즐겨찾기 |

**MenusRoleAppliedList 의 작동**
— 다음 여섯 단계 조인을 한 쿼리로 돌린다:

```text
menu → role_menu → role → role_rolegroup → rolegroup → user_rolegroup → user
```

여기에 `where menu_use_flag = true and role_menu.use_flag = true` 조건. 즉 로그인 사용자의 `user_id` 에서 줄기를 타고 올라가, 닿는 메뉴만 distinct 로 뽑는다. 프론트 MenuLoader 는 이 결과를 그대로 받아 Menu 모델에 세팅할 뿐이다. **권한 분기 로직이 프론트에 없다.**

### 3-3. RoleManagement / RoleGroupManagement

역할·역할 그룹 CRUD 서비스. 이번 분석에선 아직 읽지 않았다 — 다음 단계에서 확인 예정.

## 4. 프론트 주입 — common_ott.lib 가 공통 로더를 돌린다

공용 라이브러리의 실체는 `cap-app/library_ott/`, 네임스페이스 `common_ott.lib` 이다. `BaseController.onInit` 가 다섯 로더를 병렬로 돌린다.

| 로더 | 모델 | 소스 | 용도 |
|---|---|---|---|
| **UserSessionLoader** | `User` | V2 `/srv-api/odata/v2/ott-core/UserManagement` read `/UserSessionInfo` | 본인 정보 (공유 staticModel) |
| **MenuLoader** | `Menu` | V4 `/srv-api/odata/v4/ott-core/MenuManagement/` bindList `/MenusRoleAppliedList` | 역할 기반 메뉴 |
| CommonCodeLoader | — | — | 공통 코드 |
| MessageLoader | — | — | 메시지 |
| MenuFunctionLoader | — | — | 메뉴별 기능 권한 |

각 로더는 `attachEventOnce('ready')` 로 병렬 실행, 결과를 view 와 Core 에 세팅한다.

**메뉴 식별자 주입**
— BaseController 가 이 한 줄로 approuter 가 전달한 메뉴 ID 를 읽는다:

```js
var sMenuId = getOwnerComponent().getComponentData().MenuId;
```

approuter 가 UI5 앱을 열 때 `componentData.MenuId` 에 현재 메뉴 코드를 심어준다. **이 주입 메커니즘은 다음 단계에서 확인한다** — approuter 커스텀 로직인지, portal_ott URL 쿼리스트링인지 아직 못 봤다.

## 5. approuter_ott 라우트 — 요청을 어디로 보낼까

`cap-app/approuter_ott/xs-app.json`. 모든 라우트 `authenticationType: xsuaa`, `sessionTimeout: 60`.

| source 패턴 | 목적지 | 비고 |
|---|---|---|
| `^.*/srv-api/(odata/v[24]/ott-core/.*)$` | `onOttCoreService` | core OData. csrfProtection false |
| `^.*/srv-api/(.*)$` | `onOttService` | ott 8083 catch-all |
| `/common_ott.lib/(.*)` | html5-apps-repo-rt `common_ottlib-1.0.0` | 공용 라이브러리 |
| `/common.lib/(.*)` | html5-apps-repo-rt `commonlib-1.0.0` | 공용 라이브러리(legacy) |
| `^.*/portal_ott/(.*)$` | html5-apps-repo-rt `comportalott-1.0.0` | welcomeFile `/index.html` |
| `^.*/sysmgt_ott/(.*)$` | html5-apps-repo-rt `comsysmgtott-1.0.0` | 시스템 관리 |
| `(.*)` (catch-all) | `comportalott-1.0.0` | 결국 portal_ott 로 귀결 |

**로컬 dev**
— `cap-app/approuter_ott/dev/xs-app.json` 에는 `localDir` 라우트가 따로 있다(보드 `/board/`, 정산 `/settlement/` 패턴). OData 는 배포와 같은 catch-all → ott 8083. 배포에선 이 localDir 라우트가 html5-apps-repo-rt 라우트로 바뀐다.

## 6. 정산·보드가 배포 환경에서 걸리는 곳

두 기능이 배포 환경에서 만나는 통제 지점을 짚는다.

### 6-1. 정산(기능5)

```text
@(requires: 'Admin')
   → scope $XSAPPNAME.Admin
   → role-collection OTT_Settlement_Admin (BTP Cockpit 할당)
```

- Admin role-collection 이 없으면 → 403

- 같은 이유로 `MenusRoleAppliedList` 에 정산 메뉴가 안 나온다 → 메뉴도 안 뜸

### 6-2. 보드(기능6)

- 서비스는 `authenticated-user` — 로그인만 하면 접근

- **본인확인**
  은 `UserSessionInfo`(`user_id = $user`) 로. 글의 `author_user_id` === `$user` 분기는 백엔드 `before('UPDATE'/'DELETE')` 의 `verifyPostAuthor` 가 처리

### 6-3. 메뉴 노출 조건 — 두 단계가 다 필요

메뉴에 정산·보드를 띄우려면:

1. **Menu 마스터 등록**
   — `menu_code` / `menu_app_id` / `menu_route_path`(또는 `menu_app_path`) 세팅

2. **Role_Menu 매핑**
   — 해당 역할에 `use_flag = true` 로 연결

둘 중 하나라도 빠지면 `MenusRoleAppliedList` 에 안 나온다. approuter 라우트만 열어둔 상태론 메뉴 진입이 안 된다.

## 7. 다음 단계에서 확인할 것

배포 approuter 작업 설계로 넘어가기 전, 아직 못 읽은 다섯 가지.

1. **portal_ott 메뉴 트리/런치패드 구조**
   — 메뉴 클릭 → UI5 앱 진입, `componentData.MenuId` 주입 방식

2. **approuter 의 MenuId 주입 메커니즘**
   — html5-apps-repo-rt 서빙 시 `componentData` 에 메뉴 ID 를 어떻게 심는지

3. **RoleManagement / RoleGroupManagement 서비스**
   — 역할·그룹 CRUD

4. **Menu.csv 시드**
   — 정산·보드 메뉴 등록 여부. **등록 안 되어 있으면 시드 추가가 배포의 출발점**. 라우트만 있어도 메뉴가 안 뜨므로

5. **배포 MTA 구조**
   — standalone vs approuter_ott 통합. 팀 합의 필요. 로컬 dev localDir 라우트는 배포에서 html5-apps-repo-rt 라우트로 전환

내 판단: **4번(Menu.csv 시드) 부터** 보는 게 빠르다. 등록이 안 되어 있으면 배포 아무것도 안 뜨니, 이게 전체 작업의 출발점이 된다.

## 8. 한 줄 요약

> 사용자 → 역할 그룹 → 역할 → 메뉴. 백엔드가 `$user` 와 6개 조인으로 이 줄기를 따라 메뉴를 걸러주고, 프론트는 받기만 한다. 정산은 `Admin` role-collection, 보드는 `authenticated-user` + 본인확인. 메뉴 노출은 Menu 마스터 등록 + Role_Menu 매핑이 둘 다 있어야 한다.
