
## 1. 개요

기능 1·2는 OTT 플랫폼의 **컨텐츠 탐색**과 **상세 보기 + 리뷰**를 담당합니다.

| 기능 | 책임 화면 | 핵심 API |
|---|---|---|
| 기능 1 | 메인 페이지: 컨텐츠 목록, 장르/태그 필터, 검색 | `MainService` (`/odata/v4/main`) |
| 기능 2 | 상세 페이지: 컨텐츠 정보, 리뷰/평점, 출연진 | `DetailService` (`/odata/v4/detail`) |

이 문서는 계획 단계 산출물로서 **무엇을 설계했고 어떤 범위로 개발할지**를 정리합니다. 실제 CDS 정의, handler 처리 방식, 로컬 인증 설정, SQL 및 호출 예시는 [[02_개발단계/기능1_2_서비스_아키텍처_및_커스텀로직]]에서 관리합니다.

## 2. 계획 단계 범위

### 2.1 포함 범위

- 기능 1·2의 화면/API 책임 분리
- 주요 엔티티와 관계 구조 정의
- 메인/상세 서비스의 데이터 노출 범위 정의
- 리뷰 등록/삭제와 평점 집계가 필요한 업무 흐름 정의
- 팀 협의가 필요한 설계 결정 사항 정리

### 2.2 개발 단계 이관 범위

- 실제 CDS entity/service 코드
- 서비스 구현 경로와 custom action 구현 상세
- 사용자 식별, 데이터 생성, 권한/오류 처리 방식
- 평점/리뷰 수 재계산 SQL
- 로컬 mocked 인증 설정
- OData 호출 및 검증 예시

---

## 3. 설계 원칙

- **표준 OData 사용**: 검색/필터/정렬은 `$filter`, `$search`, `$orderby` 등 OData 표준 기능을 우선 사용합니다.
- **N:M 정규화**: 컨텐츠의 장르/태그는 각각 별도 마스터와 연결 테이블로 분리합니다.
- **Service 분리**: 메인 화면과 상세 화면이 필요로 하는 데이터 범위가 달라 `MainService`와 `DetailService`를 별도로 둡니다.
- **Composition vs Association**: 리뷰/출연진/장르/태그는 컨텐츠에 종속되는 구조로, 시청 기록은 독립적으로 참조되는 구조로 설계합니다.
- **공통 감사 필드 자동 관리**: 생성/수정 일시와 사용자는 CAP의 `managed` Aspect로 관리하는 것을 기준으로 합니다.

## 4. 엔티티 개념 설계

네임스페이스: `com.cap.ott`

| 엔티티 | 목적 | 주요 필드/관계 |
|---|---|---|
| `Users` | 서비스 사용자 및 구독 상태 관리 | 사용자 ID, 이름, 구독 플랜/상태, 연령대, 성별, 구독 플랜 참조 |
| `Contents` | OTT 컨텐츠 기본 정보 관리 | 컨텐츠 ID, 제목, 유형, 평균 평점, 리뷰 수, 파트너 참조 |
| `Genre` | 장르 마스터 | 장르 ID, 장르명 |
| `Tag` | 태그 마스터 | 태그 ID, 태그명 |
| `ContentGenre` | 컨텐츠-장르 N:M 연결 | 컨텐츠, 장르 복합 키 |
| `ContentTag` | 컨텐츠-태그 N:M 연결 | 컨텐츠, 태그 복합 키 |
| `Review` | 컨텐츠 리뷰 및 평점 | 리뷰 ID, 컨텐츠, 사용자, 평점, 리뷰 내용 |
| `ContentCast` | 상세 화면 출연진 정보 | 출연진 ID, 컨텐츠, 이름, 역할, 배역명 |

### 4.1 주요 관계

- `Contents`는 장르, 태그, 리뷰, 출연진 정보를 하위 데이터로 가집니다.
- 장르/태그는 마스터 엔티티와 N:M 연결 테이블로 분리합니다.
- 리뷰는 컨텐츠와 사용자에 연결되며, 등록/삭제 시 컨텐츠의 평균 평점과 리뷰 수에 영향을 줍니다.
- `Users`의 연령대/성별은 기능 3의 트렌드 분석과 공유되는 기준 필드입니다.
- `Contents`의 파트너 참조는 기능 4·5 정산 영역과 연결되는 확장 지점입니다.

## 5. 서비스/API 설계

### 5.1 MainService

| 항목 | 설계 내용 |
|---|---|
| 경로 | `/odata/v4/main` |
| 사용 화면 | 메인 페이지 |
| 주요 책임 | 컨텐츠 목록 조회, 장르/태그 기준 필터, 제목 검색, 세션 사용자 조회 |
| 컨텐츠 노출 범위 | 컨텐츠 ID, 제목, 컨텐츠 유형, 장르, 태그 중심의 최소 정보 |
| 사용자 노출 범위 | 사용자 ID, 이름, 구독 상태, 연령대, 성별 등 메인 화면에 필요한 정보 |
| 제외 정보 | 상세 평점, 리뷰, 출연진, 시청 기록 등 상세 화면 전용 데이터 |

### 5.2 DetailService

| 항목 | 설계 내용 |
|---|---|
| 경로 | `/odata/v4/detail` |
| 사용 화면 | 상세 페이지 |
| 주요 책임 | 컨텐츠 상세 조회, 장르/태그 expand, 리뷰 조회/등록/삭제, 출연진 조회 |
| 컨텐츠 노출 범위 | 컨텐츠 전체 필드와 리뷰/출연진/장르/태그 관계 |
| 사용자 노출 범위 | 리뷰 작성자 식별에 필요한 최소 사용자 정보 |
| 주요 action | `submitReview`, `deleteReview` |

### 5.3 권한 기준

- 두 서비스 모두 인증된 사용자만 접근하는 것을 기준으로 합니다.
- 로컬 개발과 운영 인증 방식은 환경별 설정으로 분리합니다.

## 6. API 계약 요약

| 업무 | 서비스 | 방식 | 비고 |
|---|---|---|---|
| 컨텐츠 목록 조회 | `MainService.Contents` | OData read | 메인 화면 카드/목록 기준 |
| 제목 검색 | `MainService.Contents` | OData filter/search | `$filter` 기본, `$search` 확장 여부 협의 필요 |
| 장르 필터 | `MainService.Contents` | OData expand + filter | N:M 관계 기반 필터 |
| 컨텐츠 상세 조회 | `DetailService.Contents` | OData read + expand | 장르, 태그, 리뷰, 작성자 정보 포함 |
| 리뷰 등록 | `DetailService.submitReview` | custom action | 평점 범위 검증 및 집계 갱신 필요 |
| 리뷰 삭제 | `DetailService.deleteReview` | custom action | 작성자 본인 여부 확인 및 집계 갱신 필요 |
| 세션 사용자 조회 | `MainService.getSessionUser` | custom action | 로그인 사용자 기본 정보 반환 |

## 7. 팀 협의 필요 사항

| 주제 | 현재 상태 | 결정 필요 |
|---|---|---|
| `review_id` 채번 | 임시 채번 방식 사용 | sequence / UUID-short / prefix 정책 중 확정 |
| 검색 | 제목 contains 기반 검색 우선 | `$search` 적용 여부, `@cds.search` 확장 여부 |
| HANA SQL 검증 | 로컬 SQLite 기준 검증 | HDI Container에서 평점 재계산 방식 검증 |
| CSV 데이터 | 기본 시드 데이터 등록 | `ContentGenre`/`ContentTag` 참조 정합성 확인 |
| UI5 정렬/필터 | 세부 UX 미확정 | 메인 화면 기획 확정 후 `$orderby`, 다중 필터 방식 결정 |
