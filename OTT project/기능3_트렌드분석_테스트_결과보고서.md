## 3. 테스트 중 발생한 문제 및 수정 내역

### 문제 #1 — 404 Not Found: 경로 prefix 누락

**현상**:
```
HTTP/1.1 404 Not Found
Cannot GET /odata/v4/ott/trend/TrendAnalysis/SubscriberGroupSet
```

**원인**: 기존 Mypage/Admin 서비스는 `@(path: 'ott/Mypage')`, `@(path: 'ott/Admin')`처럼 `ott/` prefix가 있었으나, TrendAnalysis/AnomalyAlert 서비스는 `@(path: 'trend/TrendAnalysis')`, `@(path: 'trend/AnomalyAlert')`로 prefix가 누락되어 라우팅 불일치 발생.

**수정** (`srv/cds/trend/` 내 2개 파일):
```cds
// Before
@(path: 'trend/TrendAnalysis')
@(path: 'trend/AnomalyAlert')

// After
@(path: 'ott/trend/TrendAnalysis')
@(path: 'ott/trend/AnomalyAlert')
```

### 문제 #2 — 500 SQLITE_CONSTRAINT_NOTNULL: FK 컬럼명 불일치

**현상**:
```json
{
  "error": {
    "message": "NOT NULL constraint failed: com_cap_ott_GroupTrend.group_id",
    "code": "SQLITE_CONSTRAINT_NOTNULL"
  }
}
```

**원인**: CDS 모델에서 `group : Association to SubscriberGroup on group.group_id = group_id`처럼 **on 절로 명시적 FK 매핑**을 했기 때문에, CAP이 생성한 FK 컬럼명은 `group_id` 그대로입니다. 그러나 핸들러에서는 `group_group_id`(암시적 매핑 시 생성되는 이름)로 INSERT를 시도하여 NOT NULL 제약 위반 발생.

**CDS 모델 분석**:
```cds
entity GroupTrend {
    key trend_id   : UUID;
        group_id   : String(30) not null;   // ← FK 컬럼명이 group_id
        group      : Association to SubscriberGroup
                     on group.group_id = group_id;  // on 절 있음 → 암시적 이름 생성 안 함
}
```

> **CAP 네이밍 규칙**: `on` 절로 명시적 매핑 시 FK 컬럼명은 필드명 그대로 사용. `on` 절 없을 때만 `associationName_keyName` 형태로 자동 생성.

**수정** (`TrendAnalysis.handler.ts` 내 7곳):

| 위치 | Before | After |
|------|--------|-------|
| GroupTrend INSERT (빈 그룹) | `group_group_id` | `group_id` |
| GroupTrend INSERT (일반) | `group_group_id` | `group_id` |
| ContentGroupStat INSERT | `group_group_id` | `group_id` |
| prevTrend SELECT WHERE | `group_group_id` | `group_id` |
| anomaly CHURN_SPIKE | `group_group_id` | `group_id` |
| anomaly WATCH_DROP | `group_group_id` | `group_id` |
| anomaly WATCH_SURGE | `group_group_id` | `group_id` |

### 문제 #3 — Server Crash: `req.error()` 미반환

**현상** (2-2 analysisMonth 누락 테스트 시):
```
[calculateTrends] 대상 그룹 수: 10, 분석월: undefined
TypeError: Cannot read properties of undefined (reading 'split')
    at getPreviousMonth (...TrendAnalysis.handler.ts:214:26)
```

**원인**: `req.error(400, ...)` 호출 후 `return`이 없어 코드가 계속 실행되어 `getPreviousMonth(undefined)`에서 `undefined.split()` 호출로 크래시 발생.

**수정**:
```ts
// Before
if (!analysisMonth) {
    req.error(400, 'analysisMonth 파라미터가 필요합니다. (형식: YYYY-MM)');
}

// After
if (!analysisMonth) {
    return req.error(400, 'analysisMonth 파라미터가 필요합니다. (형식: YYYY-MM)');
}
```

> 참고: Settlement 핸들러는 `return req.reject()` 패턴을 사용하고 있었음. 두 방식 모두 `return`이 필요.

## 5. 주의사항 및 교훈

### 5-1. CDS Association의 FK 컬럼명 규칙

| Association 정의 방식 | 생성되는 FK 컬럼명 | 예시 |
|------------------------|-------------------|------|
| `on group.group_id = group_id` (명시적) | 필드명 그대로 | `group_id` |
| `Association to SubscriberGroup` (암시적) | `associationName_keyName` | `group_group_id` |

> **핵심**: `on` 절 유무에 따라 FK 컬럼명이 달라지므로, INSERT/SELECT 시 주의 필요.

### 5-2. `req.error()` / `req.reject()` 반드시 `return`과 함께 사용

에러 객체를 생성할 뿐 실행을 중단하지 않으므로, 후속 코드가 계속 실행되어 예기치 않은 크래시로 이어질 수 있음.

### 5-3. 모든 서비스 path prefix 통일

마이크로서비스가 여러 개일 경우, OData URL namespace를 일관되게 유지해야 혼선을 방지할 수 있음. (`ott/` prefix 통일)
