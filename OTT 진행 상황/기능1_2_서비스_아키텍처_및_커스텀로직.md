
## 1. 목적

이 문서는 OTT 기능 1·2의 개발 단계 산출물입니다. 계획 단계 문서에서 분리한 실제 구현 상세를 보관합니다.

- CDS entity 구현 기준
- `MainService`, `DetailService` projection 및 action 구성
- handler 비즈니스 로직
- 로컬 인증 설정
- 평점/리뷰 수 재계산 SQL
- OData 호출 및 검증 예시

계획 단계의 범위와 설계 의도는 [[01_계획단계/OTT-기능1-2-엔티티, 서비스 설계]]를 기준으로 합니다.

## 2. CDS Entity 구현 기준

네임스페이스: `com.cap.ott`

### 2.1 Users

```cds
entity Users {
    key user_id               : String(10);
        user_name             : String(100);
        subscription_plan     : String(20);   // BASIC / STANDARD / PREMIUM
        subscription_status   : String(20);   // active / cancelled / suspended
        subscription_start_date : Date;
        subscription_end_date : Date;
        age_group             : String(10);   // 10s / 20s / 30s / 40s / 50s
        gender                : String(1);    // M / F
        plan                  : Association to ott.SubscriptionPlans
                                  on plan.code = subscription_plan;
}
```

- `age_group`, `gender`는 기능 3(트렌드 분석)과 공유하는 필드입니다.
- `plan`은 기능 4·5(구독/정산)와 연결되는 카탈로그 참조입니다.
- `managed` Aspect를 적용해 `createdAt`, `modifiedAt`, `createdBy`, `modifiedBy`를 자동 관리합니다.

### 2.2 Contents

```cds
entity Contents {
    key content_id   : String(10);
        title        : String(200);
        content_type : String(20);          // MOVIE / SERIES / FEATURE / ANIMATION
        avg_rating   : Decimal(2,1) default 0;
        rating_count : Integer      default 0;

        partner      : Association to ott.Partners;   // 기능 4·5 정산용

        genres       : Composition of many ContentGenre on genres.content = $self;
        tags         : Composition of many ContentTag   on tags.content   = $self;
        reviews      : Composition of many Review       on reviews.content = $self;
        casts        : Composition of many ContentCast  on casts.content  = $self;
        histories    : Association to many ViewingHistory on histories.content = $self;
}
```

- `avg_rating`과 `rating_count`는 리뷰 등록/삭제 시 handler에서 재계산합니다.
- `partner`는 기능 4·5 확장 영역입니다.
- `managed` Aspect를 적용합니다.

### 2.3 Genre / Tag

```cds
entity Genre {
    key genre_id : String(10);
        name     : String(50);
        contents : Association to many ContentGenre on contents.genre = $self;
}

entity Tag {
    key tag_id : String(10);
        name     : String(50);
        contents : Association to many ContentTag on contents.tag = $self;
}
```

- `managed` Aspect를 적용합니다.

### 2.4 ContentGenre / ContentTag

```cds
entity ContentGenre {
    key content : Association to Contents;
    key genre   : Association to Genre;
}

entity ContentTag {
    key content : Association to Contents;
    key tag     : Association to Tag;
}
```

- 복합 키로 `content`와 `genre`/`tag` association을 사용합니다.
- 단순 연결 테이블이므로 `managed` Aspect는 미적용합니다.

### 2.5 Review

```cds
entity Review {
    key review_id   : String(10);
        content     : Association to Contents;
        user        : Association to Users;
        rating      : Decimal(2,1);           // 0.0 ~ 5.0
        review_text : String(1000);
        // createdAt, modifiedAt, createdBy, modifiedBy는 managed Aspect가 자동 생성
}
```

- `managed` Aspect를 적용합니다.
- `review_id` 채번 방식은 아직 팀 정책 확정 전입니다. 현재는 `R` + timestamp 뒤 9자리를 임시로 사용합니다.

### 2.6 ContentCast

```cds
entity ContentCast {
    key cast_id        : String(10);
        content        : Association to Contents;
        name           : String(100);
        role           : String(50);   // 주연, 조연, 감독, 성우
        character_name : String(100);
}
```

- 상세 페이지의 출연진 정보입니다.
- `managed` Aspect를 적용합니다.

## 3. Service 구현 기준

### 3.1 MainService

```cds
@path: '/odata/v4/main'
@impl: 'srv/src/config/MainService.handler'
service MainService @(requires: 'authenticated-user') {

    entity Genre, Tag, ContentGenre, ContentTag;

    entity Contents as projection on db.Contents {
        content_id,
        title,
        content_type,
        genres,
        tags
    };

    entity Users as projection on db.Users {
        user_id,
        user_name,
        subscription_plan,
        subscription_status,
        age_group,
        gender
    };

    action getSessionUser() returns Users;
}
```

- 메인 목록용으로 최소 필드만 노출합니다.
- `avg_rating`, `rating_count`, `partner`, `reviews`, `casts`, `histories`는 노출하지 않습니다.
- `Users`의 구독 시작/종료일은 제외합니다.

### 3.2 DetailService

```cds
@path: '/odata/v4/detail'
@impl: 'srv/src/config/DetailService.handler'
service DetailService @(requires: 'authenticated-user') {

    entity Contents     as projection on db.Contents;
    entity Review       as projection on db.Review;
    entity ContentCast  as projection on db.ContentCast;
    entity Genre        as projection on db.Genre;
    entity Tag          as projection on db.Tag;
    entity ContentGenre as projection on db.ContentGenre {
        *,
        genre   : redirected to DetailService.Genre,
        content : redirected to DetailService.Contents
    };
    entity ContentTag   as projection on db.ContentTag {
        *,
        tag     : redirected to DetailService.Tag,
        content : redirected to DetailService.Contents
    };

    entity Users as projection on db.Users {
        user_id,
        user_name,
        age_group,
        gender
    };

    action submitReview(
        content_id  : String(10),
        rating      : Decimal(2,1),
        review_text : String(1000)
    ) returns Review;

    action deleteReview(review_id: String(10)) returns Boolean;
}
```

- 상세 화면용으로 컨텐츠 전체 필드와 리뷰/출연진을 노출합니다.
- `Contents?$expand=genres($expand=genre),tags($expand=tag)`를 지원하기 위해 `ContentGenre`, `ContentTag`, `Genre`, `Tag` projection을 service 내에 추가하고 `redirected to`로 바인딩합니다.
- `Users`는 리뷰 작성자 정보를 노출할 때 필요한 필드만 선택합니다.

### 3.3 로컬 인증 설정

로컬 개발용 `.cdsrc.json`에는 mocked 사용자를 설정합니다.

```json
{
  "auth": {
    "kind": "mocked",
    "users": {
      "U001": { "roles": ["authenticated-user"] },
      "U002": { "roles": ["authenticated-user"] }
    }
  }
}
```

## 4. Handler / 비즈니스 로직

### 4.1 getSessionUser (MainService)

현재 로그인한 사용자의 기본 정보를 반환합니다.

```cds
action getSessionUser() returns Users;
```

- `req.user.id`로 사용자를 식별합니다.
- `Users` projection에서 본인 정보를 조회합니다.
- 인증 실패 시 401, 사용자 미존재 시 404를 반환합니다.

### 4.2 submitReview (DetailService)

```cds
action submitReview(
    content_id  : String(10),
    rating      : Decimal(2,1),
    review_text : String(1000)
) returns Review;
```

- `content`와 `user`를 association 객체 형태로 INSERT합니다.
  - `{ content: { content_id }, user: { user_id: userId }, ... }`
- 등록 후 해당 컨텐츠의 `avg_rating`과 `rating_count`를 raw SQL로 재계산합니다.
- rating은 0 ~ 5 범위를 검증합니다.
- `createdAt`/`modifiedAt`/`createdBy`/`modifiedBy`는 `managed` Aspect가 자동으로 관리합니다. handler에서 수동으로 채우지 않습니다.

### 4.3 deleteReview (DetailService)

```cds
action deleteReview(review_id: String(10)) returns Boolean;
```

- 삭제 대상이 존재하는지 확인합니다. 없으면 404를 반환합니다.
- `existing.user_user_id !== req.user.id`면 403을 반환합니다.
- 삭제 후 해당 컨텐츠의 `avg_rating`과 `rating_count`를 raw SQL로 재계산합니다.

## 5. 평점/리뷰 수 재계산 SQL

```sql
UPDATE COM_CAP_OTT_CONTENTS
SET avg_rating   = (SELECT AVG(rating) FROM COM_CAP_OTT_REVIEW WHERE content_id = ?),
    rating_count = (SELECT COUNT(*)  FROM COM_CAP_OTT_REVIEW WHERE content_id = ?)
WHERE content_id = ?
```

- CAP의 자동 aggregate UPDATE는 관계형 DB에서 FK 해석 문제가 있어 raw SQL을 사용합니다.
- HANA HDI Container에서도 동일 구조를 사용하며, 배포 전 별도 검증이 필요합니다.

## 6. 주요 OData 호출 예시

### 6.1 메인 — 컨텐츠 목록

```http
GET /odata/v4/main/Contents?$top=20
```

### 6.2 메인 — 제목 검색

```http
GET /odata/v4/main/Contents?$filter=contains(title,'검색어')
```

### 6.3 메인 — 장르 필터

```http
GET /odata/v4/main/Contents
  ?$expand=genres($expand=genre)
  &$filter=genres/any(g: g/genre/genre_id eq 'G0001')
```

### 6.4 상세 — 컨텐츠 상세 + 장르/태그/리뷰

```http
GET /odata/v4/detail/Contents('C0001')
  ?$expand=genres($expand=genre),tags($expand=tag),reviews($expand=user)
```

### 6.5 리뷰 등록

```http
POST /odata/v4/detail/submitReview
Content-Type: application/json

{
  "content_id": "C0001",
  "rating": 4.5,
  "review_text": "재밌었습니다."
}
```

### 6.6 리뷰 삭제

```http
POST /odata/v4/detail/deleteReview
Content-Type: application/json

{ "review_id": "R123456789" }
```

## 7. 개발 검증 체크리스트

| 항목 | 검증 내용 |
|---|---|
| CDS compile | entity/service 정의가 CAP compile을 통과하는지 확인 |
| MainService 목록 | `/odata/v4/main/Contents`가 최소 필드 중심으로 응답하는지 확인 |
| DetailService expand | 장르/태그/리뷰 expand가 정상 동작하는지 확인 |
| 리뷰 등록 | `submitReview` 후 리뷰가 생성되고 평균 평점/리뷰 수가 갱신되는지 확인 |
| 리뷰 삭제 | 작성자 본인만 삭제 가능하고 삭제 후 집계가 갱신되는지 확인 |
| 로컬 인증 | mocked 사용자 `U001`, `U002` 기준으로 action이 동작하는지 확인 |
| HANA 검증 | HDI Container에서 raw SQL 집계가 동일하게 동작하는지 확인 |

## 8. 남은 개발 결정 사항

| 주제 | 현재 상태 | 필요 조치 |
|---|---|---|
| `review_id` 채번 | `R` + timestamp 뒤 9자리 임시 사용 | 팀 정책 확정 후 handler 반영 |
| 검색 방식 | `$filter=contains(title,'keyword')` 기본 | `$search`, `@cds.search` 적용 여부 결정 |
| HANA SQL | SQLite 로컬 테스트 기준 | HDI Container 검증 |
| CSV 데이터 | 기본 시드 등록 | N:M 참조 정합성 확인 |
| UI5 정렬/필터 | 미확정 | 화면 기획 확정 후 `$orderby`, 다중 필터 구현 |
