## 0. 프로젝트 아키텍처 — 최종

### 0.1 풀스택 _ott 독립

모든 OTT 모듈은 `_ott` 접미사를 사용하며, 원본 템플릿은 **일체 수정하지 않는다**.

```
cap-node/
├── core/          ← 원본 (절대 손 안 댐)
├── ott/           ← OTT 비즈니스 백엔드 (DetailService, MembershipService, MainService)
└── core_ott/      ← OTT 공통 백엔드 (MenuManagement, UserManagement, CodeManagement, MessageManagement)

cap-app/
├── library/       ← 원본 (절대 손 안 댐)
├── library_ott/   ← OTT 공통 UI5 라이브러리 (namespace: common_ott.lib)
├── approuter/     ← 원본
├── approuter_ott/ ← OTT 진입점 + 라우팅
├── portal_ott/    ← OTT 포털
└── sysmgt_ott/    ← OTT 기능 모듈
    └── webapp/
        ├── detailPage/   ← 기능2: 컨텐츠 상세
        └── myPage/       ← 기능4: 마이페이지
```

### 0.4 아키텍처 의도

| 설계 결정 | 이유 |
|-----------|------|
| **MyMembership ↔ PlanChange 분리** | 조회(Membership)와 액션(PlanChange)의 책임 분리. SAP Fiori List Report → Object Page 패턴과 동일. 하나의 화면에 모든 기능을 넣으면 UI가 과도하게 길어지고 상태 분기 로직이 복잡해짐 |
| **`DynamicPage` 제거, `ObjectPageLayout` 루트 사용** | 두 컨트롤러는 서로 다른 스크롤 컨테이너. 중첩 시 spacer 무한 확장 버그 발생. `ObjectPageLayout`만으로 `snappedContent`·`expandedContent`·`actions`를 네이티브 지원하므로 래퍼 불필요 |
| **`sap.ui.table.Table` → `sap.m.Table`** | 출연진 데이터는 3~5행으로 소규모. `sap.ui.table.Table`의 기본 `visibleRowCount=10`이 빈 행을 생성하고 height 계산을 복잡하게 함. 단순 표는 `sap.m.Table`이 적합 |
| **i18n 완전 분리** | 모든 UI 텍스트를 `i18n.properties`에서 관리. 언어 추가 시 파일 하나만 추가하면 됨 |
| **V2 OData 콜백 패턴** | `sap.ui.model.odata.v2.ODataModel`의 `callFunction`은 Promise를 반환하지 않음. `success`/`error` 콜백 필수 |
| **리뷰 등록 후 서버 재조회** | 수동 배열 조작(prepend + avg 재계산) 대신 `_loadDetail()`로 서버에서 전체 데이터를 다시 가져옴. 사용자명 expand 누락, 평점 계산 오류를 근본적으로 방지 |

### 1.5 핵심 로직

#### A. 1인 1리뷰 제한 + FeedListItemAction (7/14)

```javascript
// UserSessionLoader 비동기 → 최대 5회(1초) 재시도
// _checkMyReview에서 각 리뷰 행에 _actions 배열 동적 생성
_checkMyReview: function () {
    // ... User 모델 대기 로직 (기존과 동일) ...
    for (var i = 0; i < aReviews.length; i++) {
        var isMine = aReviews[i].user_user_id === myUserId;
        if (isMine) oMyReview = aReviews[i];
        // ★ FeedListItemAction: 본인=edit+delete, 타인=빈배열(···메뉴 숨김)
        oViewModel.setProperty("/Detail/reviews/results/" + i + "/_actions", isMine ? [
            { Text: "수정", Icon: "sap-icon://edit", Key: "edit" },
            { Text: "삭제", Icon: "sap-icon://delete", Key: "delete" }
        ] : []);
    }
    oViewModel.setProperty("/HasReviewed", !!oMyReview);
}
```

- User 모델 키: `/user_id` (NOT `/userId`)
- ★ `_actions` 배열이 FeedListItem의 `actions` aggregation에 바인딩
- `FeedListItemAction`은 `sap.m.Button`이 아닌 별도 타입. `templateShareable: false` 필수

#### B. 리뷰 등록 → FeedInput.post → 서버 재조회 (7/14)

```javascript
// ★ onSubmitReview → onFeedPost (FeedInput의 post 이벤트)
// ★ review_text는 oEvent.getParameter("value")로 획득
onFeedPost: function (oEvent) {
    if (this._bSubmitting) return;
    var reviewText = oEvent.getParameter("value");
    // ...
    var doSubmit = function () {
        oModel.callFunction("/submitReview", {
            success: function () {
                that._loadDetail(contentId);  // 서버 재조회
                that._bSubmitting = false;
            }
        });
    };
    if (editingReviewId) {
        // 수정: deleteReview → submitReview 순차 실행
        oModel.callFunction("/deleteReview", { success: doSubmit });
    } else { doSubmit(); }
}
```

- FeedInput은 자체 등록 버튼 내장 → 별도 버튼 불필요
- `.toFixed(2)`로 평점 소수점 둘째 자리까지 표시

#### C. 수정 모드 — FeedListItemAction "edit" 키 (7/14)

```javascript
// ★ onEditReview 분리 함수 → onFeedAction 통합 핸들러
onFeedAction: function (oEvent) {
    var key = oEvent.getSource().getKey();  // "edit" or "delete"
    var oItem = oEvent.getParameter("item");
    var oReview = /* binding context에서 추출 */;
    if (key === "edit") {
        oViewModel.setProperty("/NewReview", {
            rating: oReview.rating,
            review_text: oReview.review_text,
            editing_review_id: oReview.review_id
        });
    } else if (key === "delete") {
        this.onDeleteReview({ getSource: function() { return oItem; } });
    }
}
```

- FeedListItem의 `···` 메뉴에서 edit/delete 분기
- `onDeleteReview`는 `onFeedAction`이 내부 호출

### 1.6 해결된 주요 이슈

| # | 이슈 | 원인 | 해결 |
|---|------|------|------|
| 1 | **무한 스크롤** (스크롤바 계속 증가) | `DynamicPage` 안에 `ObjectPageLayout` 중첩 → spacer div 무한 확장 | `ObjectPageLayout`을 루트로 변경 |
| 2 | **별점 클릭 시 원상 복귀** | `NewReview` 객체가 ViewModel에 초기화되지 않음 | `_onRouteMatched`에서 `{rating:0, review_text:""}` 초기화 |
| 3 | **별점 클릭 시 입력 텍스트 소멸** | 동일한 `NewReview` 경로 의존성 | 초기화로 해결 (동일 원인) |
| 4 | **V2 `callFunction` `.then` 오류** | `sap.ui.model.odata.v2.ODataModel`은 Promise 미반환 | `success`/`error` 콜백 패턴으로 전면 교체 |
| 5 | **User 모델 null (hasUserModel: false)** | `UserSessionLoader` 비동기 로딩이 페이지 로드보다 늦음 | 최대 5회 200ms 간격 재시도 + `_checkRetryCount` 초기화 |
| 6 | **User 모델 `userId` undefined** | 프로퍼티명이 `user_id`(스네이크)인데 `userId`(카멜)로 접근 | `getProperty("/user_id")`로 수정 |
| 7 | **리뷰 등록 후 UI 미갱신** | V2 `callFunction` Promise 미지원으로 callback 미실행 | callback → `_loadDetail` 서버 재조회 |
| 8 | **중복 리뷰 허용** | `HasReviewed` 체크 로직 부재 | `_checkMyReview()` + 작성 폼 `visible` 조건 |
| 9 | **리뷰 섹션 드롭다운 빈 항목** | SubSection에 title 미지정 | `label.reviewList` 추가 |
| 10 | **검색 결과 "[object Object]"** | 배열을 직접 바인딩 | expression binding: `{= … .length}` |
| 11 | **i18n 전면 부재** | 모든 텍스트 하드코딩 | 53개 키 생성, 뷰·컨트롤러 전면 교체 |
| 12 | **flat 레이아웃** | `DynamicPage` + div 나열 | `ObjectPageLayout` + `ObjectPageSection` + `SimpleForm` + `ResponsiveGridLayout` + `GridData span` |
| 13 | **Excel 멀티시트 미지원** | UI5 1.136 CDN `Spreadsheet`가 `sheets` 미지원 | 단일시트 → 2개 버튼 분리 |
| 14 | **리뷰 등록 후 사용자명 빈칸 + 평점 오류** | 수동 배열 조작 시 user expand 누락, avg 계산 버그 | `_loadDetail()` 서버 재조회로 대체 |
| 15 | **평균 평점이 전혀 다른 값 (4.0→3.0이어야 함)** | 🔥 SQL `WHERE content_id = ?` → 실제 FK 컬럼명은 `content_content_id`. 필터링 실패로 전체 리뷰 AVG 반영 | `content_content_id`로 수정 (`DetailService.handler.ts`) |
| 16 | **제공사 정보 출력 불가** | `DetailService.cds`에 `entity Partners` 미노출 | `Partners` 엔티티 추가 + OData expand에 `partner` 추가 |
| 17 | **`_loadDetail` Race Condition** | C001→C002 빠른 내비게이션 시 stale 응답이 최신 데이터 덮어씀 | `_loadToken` 토큰 패턴 |
| 18 | **Controller 파괴 후 콜백 접근** | setTimeout/MessageBox 콜백이 파괴된 Controller 참조 | `onExit()` + `_bDestroyed` 가드 |
| 19 | **리뷰 제출 중복 클릭** | Submit 연타 시 deleteReview→submitReview 체인 꼬임 | `_bSubmitting` 플래그 |

### 2.6 해결된 주요 이슈

| # | 이슈 | 원인 | 해결 |
|---|------|------|------|
| 1 | **V2 `callFunction` `.then` 오류** | `ODataModel` V2는 Promise 미지원 | `success`/`error` 콜백 전환 (3곳) |
| 2 | **PlanChange 화면 미표시** | `layout:GridBox`가 UI5 1.136 CDN에 없음 | `FlexBox` (`wrap="Wrap"`) |
| 3 | **`wrap="true"` 경고** | `FlexBox.wrap`은 enum | `wrap="Wrap"` |
| 4 | **`borderDesign` 경고** | `VBox`에 없는 속성 | 제거 |
| 5 | **`core:CustomData` 위치 오류** | 직접 자식으로 불가 | `<customData>` 태그로 감쌈 |
| 6 | **i18n 전면 부재** | 모든 텍스트 하드코딩 | dialog 구조 재편 포함 키 생성 |
| 7 | **`DynamicPage` 래퍼** | 무한 스크롤 위험 | `ObjectPageLayout` 루트로 변경 |
| 8 | **멤버십 폼 2열 높이 불균형** | `ResponsiveGridLayout`이 `float` 기반이라 셀 높이 차이 시 배치 이탈 | CSS `flex` 강제 전환 + `align-items: flex-start` |
| 9 | **구독 해지/재개 팝업 아이콘·텍스트 짤림** | `MessageBox.confirm`은 아이콘 '?' 고정, width 제한 | ★ `MessageBox.warning`으로 표준화 → 아이콘⚠️, `contentWidth: "30rem"`, 짧은 제목 + 긴 본문 구조 |
| 10 | **Dialog 메모리 누수** | 커스텀 `Dialog` 사용 시 `destroy()` 누락 | `MessageBox.warning`으로 교체하여 자동 생명주기 관리 + `_bDestroyed` 가드 |
| 11 | **Controller 파괴 후 MessageBox 콜백 접근** | `onClose` 콜백이 파괴된 Controller 참조 | `onExit()` + `_bDestroyed` 가드 |
| 12 | **플랜 변경 중복 제출** | Submit 연타 시 중복 API 호출 | `_bSubmitting` 플래그 |

### 4.3 V2 OData 콜백 패턴 — 전체 적용

`sap.ui.model.odata.v2.ODataModel.callFunction()`은 **Promise를 반환하지 않는다**. 모든 `callFunction` 호출을 `success`/`error` 콜백으로 통일:

```javascript
// ❌ 잘못된 패턴
oModel.callFunction("/action", { method: "POST", urlParameters: {...} })
    .then(function() {...}).catch(function() {...});

// ✅ 올바른 패턴 (V2)
oModel.callFunction("/action", {
    method: "POST",
    urlParameters: {...},
    success: function(oData) { /* 처리 */ },
    error: function(oErr) { /* 처리 */ }
});
```

**적용 위치**: detailPage 3곳 (`onSubmitReview`, `onDeleteReview`, `onDeleteMyReview`), myPage 3곳 (`_callAction`, `onSubmitPlan`)

## 6. 주요 교훈 (Lessons Learned)

| # | 교훈 | 상세 |
|---|------|------|
| 1 | **_ott 패턴 일관성** | 원본 템플릿은 절대 수정하지 않고 반드시 `_ott` 복사본에 작업 |
| 2 | **V2 OData 콜백 필수** | `callFunction`은 Promise 미지원. `success`/`error` 콜백 사용 |
| 3 | **User 모델 프로퍼티명** | `user_id` (스네이크). `userId` (카멜) 아님. `sap.ui.getCore().getModel("User").getProperty("/user_id")` |
| 4 | **User 모델 비동기 로딩** | `UserSessionLoader`는 비동기. 페이지 로드 시 null 가능 → 재시도 패턴 필요 |
| 5 | **`DynamicPage` + `ObjectPageLayout` 중첩 금지** | 두 컨트롤러 모두 자체 스크롤 관리 → spacer 무한 확장 |
| 6 | **`ObjectPageLayout` 루트 사용** | `ObjectPageDynamicHeaderTitle`만으로 `expandedContent`·`snappedContent`·`actions` 모두 지원 |
| 7 | **UI5 1.136 CDN 제약** | `sap/ui/layout/GridBox.js` 미제공, `f:Card` default aggregation 미지원, `sap.ui.layout.cssgrid` 미제공 → CSS로 우회 |
| 8 | **View ↔ Controller `this` 컨텍스트** | 콜백 내 `this` 보존 위해 `var that = this` 또는 `.bind(this)` 패턴 필수 |
| 9 | **approuter `catch-all` 확인** | `(.*)` destination이 실제 의도한 서비스를 가리키는지 반드시 확인 |
| 10 | **board 벤치마킹** | 작동하는 코드를 1:1 비교하는 것이 가장 빠른 디버깅 방법 |
| 11 | **`useBatch: false`** | V2 OData에서 배치 모드는 불필요한 복잡성만 추가 |
| 12 | **`async: true`** | SAPUI5 1.136에서 비동기 부트스트랩 필수 |
| 13 | **`ResponsiveGridLayout`은 `float` 기반** | 셀 높이 차이 시 배치 이탈. `display: flex`로 강제 전환하여 2열 정렬 안정화 |
| 14 | **`Dialog.close()` ≠ 메모리 해제** | UI5 컨트롤은 `destroy()`를 호출해야 GC 대상이 됨 |
| 15 | **CAP Association FK 컬럼명** | `Association to Foo`는 `foo_ID` 형태의 FK 컬럼 생성. SQL raw query에서 `content_id` → `content_content_id` 실수 방지 |
| 16 | **서버 재조회가 수동 조작보다 안전** | `_loadDetail()` 한 줄이 배열 prepend + avg 재계산 40줄보다 정확하고 버그에 강함 |
| 17 | **`MessageBox.warning` 표준화** | `actions` + `emphasizedAction` + `contentWidth` + `onClose` 패턴으로 Fiori 디자인 가이드라인 준수 |
| 18 | **Controller 생명주기 (`onExit`)** | UI5 SPA에서 페이지 전환 시 Controller 파괴 → `onExit()`에서 타이머 해제, `_bDestroyed` 플래그로 모든 비동기 콜백 가드 |
| 19 | **Race Condition 방지 (토큰 패턴)** | 비동기 요청마다 `_loadToken` 증가 → 응답 도착 시 토큰 비교하여 stale 응답 무시 |
| 20 | **중복 제출 방지** | `_bSubmitting` 플래그로 Submit 버튼 연타 방어 + 성공/실패 콜백에서 복원 |
| 21 | **`fnDeepCopy` 순환 참조** | 재귀적 deep copy 시 `WeakSet`으로 방문 객체 추적 → stack overflow 방지 |
| 22 | **ID 생성기 충돌 방지** | `Date.now()` 기반 ID는 동시 요청 시 충돌 → `crypto.randomUUID()` 사용 |
| 23 | **FeedListItem은 Fiori 표준 피드 패턴** | `sender`·`text`·`icon`·`info`·`actions` 5요소. `FeedListItemAction`은 Button이 아닌 별도 타입 |
| 24 | **FeedInput은 자체 등록 버튼 내장** | `post` 이벤트로 값 획득 (`oEvent.getParameter("value")`). `growing` 자동확장, `maxLength`+`showExceededText` 글자수 |
| 25 | **`convertLinksToAnchorTags`는 enum** | boolean(`true`) ❌ → 문자열(`"All"`) ✅ |
| 26 | **`iconSize` vs `transform:scale`** | 전자는 공간 확보, 후자는 겹침 → iconSize 사용 |
| 27 | **Popover 패딩은 `VerticalLayout` + `sapUiContentPadding`** | `Text`만 넣으면 패딩 없음. `addStyleClass`는 메서드 |
| 28 | **`sap.m.Page` CSS override 어려움** | `ObjectPageLayout`으로 전환하여 일관된 스타일 적용 |
