## 2. 요청 생명주기 (Request Lifecycle)

### 2.1 사용자가 포털에 접속했을 때

```
1. 사용자 브라우저 → AppRouter 에 접속
   URL: https://<app-url>/

2. AppRouter
   ├── auth: XSUAA 로그인 페이지로 리디렉션 (인증되지 않은 경우)
   ├── 인증 성공 → JWT 토큰 발급
   └── routing: index.html 서빙
       └── index.html → location.href="/srv-api/odata/v4/frame/AuthCheck"

3. /srv-api/odata/v4/frame/AuthCheck
   ├── AppRouter → /srv-api/* → onCapService destination으로 프록시
   ├── CAP core 서버 → FrameService.AuthCheck() 실행
   │   └── 로깅 ("###############################onAuthCheck####...")
   │   └── res.redirect('/portal/index.html') → 포털로 리디렉션
   └── 브라우저 → /portal/index.html 로 이동

4. Portal UI 로딩
   ├── index.html 로드
   ├── Component.js init() 실행
   │   ├── UserSessionLoader 로드 (GET /srv-api/odata/v2/core/UserManagement/UserSessionInfo)
   │   │   └── 사용자 언어·날짜서식·숫자서식·테마 설정
   │   └── MenuLoader 로드 (GET /srv-api/odata/v4/core/MenuManagement/MenusRoleAppliedList)
   │       └── 사용자 권한에 따른 메뉴 리스트 조회
   │       └── 동적으로 Route, Target, ComponentUsage 생성
   └── Layout.view.xml 렌더링
       └── ToolHeader (로고·검색·사용자명·설정 버튼)
       └── 동적 상단 메뉴바
       └── 메인 컨텐츠 영역 (Home.view.xml 기본)
```

### 2.2 CAP 서버 기동 시 생명주기 이벤트

```
cds.server() 호출
  ├── 'loaded'     → "CDS loaded event fired."
  ├── 'bootstrap'  → Express 미들웨어 등록 (요청 로깅)
  │                   app.use((req,res,next) => { console.log(`User: ${x-username}`) })
  ├── 'connect'    → DB 연결 (hana) "CDS connect event fired for service: db"
  ├── 'serving'    → 각 서비스 마운트 시작 (UserManagement, RoleManagement, ...)
  ├── 'served'     → 모든 서비스 마운트 완료
  │                   커스텀 엔드포인트 등록:
  │                   - GET /i18n.properties       → DB에서 한국어 메시지 조회
  │                   - GET /i18n_:lang.properties → DB에서 특정 언어 메시지 조회
  └── 'listening' → HTTP 서버 대기 시작
```

## 5. AppRouter 라우팅 맵

AppRouter는 URL 패턴을 기반으로 요청을 적절한 목적지로 보낸다.

| URL 패턴 | 목적지 | 설명 |
|---|---|---|
| `^.*/srv-api/(.*)$` | onCapService | CAP core 백엔드 API |
| `^.*/mdm-api/(.*)$` | onMDMService | CAP mdm 백엔드 API |
| `^.*/plan-api/(.*)$` | onPlanService | CAP plan 백엔드 API |
| `^.*/erp-api/(.*)$` | onCloudConnector | SAP ERP (Cloud Connector 경유) |
| `^/user-api(.*)` | sap-approuter-userapi | 사용자 정보 (AppRouter 내장) |
| `^.*/scim-api/(.*)$` | onSCIMApi | SCIM ID 관리 |
| `/common.lib/(.*)` | html5-apps-repo-rt | 공통 UI5 라이브러리 |
| `^.*/portal/(.*)$` | html5-apps-repo-rt | 포털 UI (comportal) |
| `^.*/sysmgt/(.*)$` | html5-apps-repo-rt | 시스템 관리 UI (comsysmgt) |
| `^.*/mdm/(.*)$` | html5-apps-repo-rt | 자재 관리 UI (commdm) |
| `^.*/dp/(.*)$` | html5-apps-repo-rt | 수요 계획 UI (comdp) |
| 그 외 모든 요청 | html5-apps-repo-rt | Welcome 페이지 (comwelcome) |

## 6. 인증 흐름

```
1. 사용자 접속 → AppRouter
   ├── 세션 없음 → XSUAA 로그인 페이지로 리디렉션
   ├── ID/PW 입력 → XSUAA 인증
   └── JWT 토큰 발급 (쿠키에 저장)

2. API 호출 시
   ├── AppRouter: JWT 토큰 검증
   ├── x-username 헤더 추가 (사용자 ID)
   └── CAP 서버: @(requires: 'authenticated-user') 검증
       └── 모든 서비스에 적용됨 (익명 접근 불가)

3. 데이터 접근 제어
   ├── @(restrict: [{ grant: 'READ', where: 'user_id = $user' }])
   │   └── UserSessionInfo: 로그인한 사용자 본인 데이터만 조회 가능
   │   └── MenusRoleAppliedList: 로그인한 사용자의 권한 메뉴만 조회
   │   └── FavoriteRoleAppliedList: 로그인한 사용자의 즐겨찾기만 조회
   └── MenuFunctionAppliedList: 사용자 권한별 UI 기능 사용 여부
```

## 7. 개발 시 주의할 점

1. **CAP core는 포트 8080, OTT는 포트 8083**으로 분리되어 있다. 로컬 개발 시 포트 충돌에 주의한다.

2. **OData v2와 v4가 혼용**된다: core 서비스는 `@cap-js-community/odata-v2-adapter`를 통해 v2로 노출되고, menu/role 등 일부는 v4로 직접 노출된다. URL 경로가 다르다 (`/odata/v2/...` vs `/odata/v4/...`).

3. **동적 i18n**은 core의 `server.ts`에서 DB 기반으로 제공된다 (`/i18n.properties`). UI5의 `CoreResourceModel`이 이걸 읽어서 i18n 모델로 사용한다.

4. **메뉴는 동적으로 생성**된다: `MenuLoader` → `MenusRoleAppliedList` 조회 → 사용자 권한에 맞는 메뉴만 트리 구조로 구성 → `Component.js`에서 동적 Route/Target 생성.

5. **한국어가 기본 언어**다: 코드 주석, i18n 기본값 등이 한국어로 작성되어 있다.
