# 6-1. 시나리오별 비용 비교와 TCO 판단

## 결론

**SAP 공개 목록가로 Integration Suite의 구독료는 계산할 수 있다. 그러나 “개발자 몇 명이 더 싸다”는 결론은 공개 가격만으로 낼 수 없다.** 직접 연동과 Integration Suite 모두 개발·운영 인력이 필요하고, 내부 인건비·기존 플랫폼·요구사항 규모가 조직마다 다르기 때문이다.

따라서 이 문서는 다음 두 값을 분리한다.

1. **공식 목록가**: SAP 가격 페이지에서 확인한 객관적 기준값
2. **내부 TCO 추정값**: 조직이 입력한 개발자 월비용과 작업량으로 계산하는 비교값

## 1. 계산에 쓰는 공식 목록가

2026-07-23에 확인한 SAP 한국 가격 페이지의 1테넌트 월 목록가와 연 환산값이다. 계약 기간은 3~36개월로 표기되어 있으며, 실제 계약 가격·세금·할인은 달라질 수 있다.

| 에디션 | 월 목록가 | 연 목록가 | 대표적인 사용 범위 |
|---|---:|---:|---|
| Starter | KRW 2,643,486 | KRW 31,721,832 | 제한적 Cloud Integration, 사용자 정의 iFlow 10개 한도 표기 |
| Standard | KRW 8,168,372 | KRW 98,020,464 | API Management, B2B 기능, Open Connectors, 이벤트 기능이 필요한 경우 |
| Enhanced | KRW 11,763,513 | KRW 141,162,156 | Standard 범위와 추가 Alert Notification, Cloud Transport Management, Document AI, advanced event mesh가 필요한 경우 |

> **산식**: 연 목록가 = 공개 월 목록가 × 12. 위 금액은 1테넌트 기준이며 개발·테스트·운영 테넌트 분리, Edge Integration Cell, 추가 메시지, 기타 BTP 서비스는 포함하지 않는다.

## 2. “개발자 몇 명” 대신 개발자 투입개월로 비교하는 이유

한 명의 개발자가 1년 내내 투입되는지, 몇 달만 투입되는지는 시나리오마다 다르다. 그래서 사람 수보다 **개발자 투입개월(인월)** 로 비교해야 한다.

```text
직접 연동 1년 TCO
= (초기 개발 인월 + 연간 변경·운영 인월 + 장애 대응 인월) × 개발자 월 총원가
  + 기존 플랫폼의 증분 라이선스·인프라 비용

Integration Suite 1년 TCO
= IS 연 목록가 + IS 설계·구현·운영 인월 × 개발자 월 총원가
  + 추가 테넌트·옵션·초과 메시지 비용
```

**개발자 월 총원가**에는 급여만이 아니라 4대 보험, 복리후생, 장비, 관리 간접비, 외주 단가 등 조직이 실제 부담하는 비용을 넣는다. 단순 연봉을 12로 나눈 값으로 비교하면 실제 의사결정 비용을 과소평가하기 쉽다.

### 구독료만 개발자 투입개월로 바꾸어 보면

아래 표는 “IS 구독료만큼의 예산”이 개발자 몇 인월에 해당하는지를 보여 준다. IS 구현·운영 인력은 아직 넣지 않은 값이므로, **도입 타당성의 출발점일 뿐 손익분기 결론은 아니다.**

| 에디션 | 연 목록가 | 개발자 월 총원가 800만 원일 때 | 1,000만 원일 때 | 1,200만 원일 때 |
|---|---:|---:|---:|---:|
| Starter | 약 3,172만 원 | 3.97 인월 | 3.17 인월 | 2.64 인월 |
| Standard | 약 9,802만 원 | 12.25 인월 | 9.80 인월 | 8.17 인월 |
| Enhanced | 약 1억 4,116만 원 | 17.65 인월 | 14.12 인월 | 11.76 인월 |

예를 들어 개발자 월 총원가를 1,000만 원으로 두면, Standard의 연 목록가는 약 9.8 인월이다. 직접 구현과 연간 운영에 필요한 총 작업량이 9.8 인월보다 작다면 **구독료만 놓고는** 직접 방식이 더 저렴할 가능성이 높다. 반대로 9.8 인월을 넘는다고 해서 자동으로 IS가 더 싸지는 않는다. IS에서도 설계·개발·운영 인월이 발생하기 때문이다.

## 3. 시나리오별 인력비 손익분기 예시

이 절은 “개발자를 몇 명 붙일 것인가”를 **연간 인월과 연간 총원가**로 바꿔 비교한다. 아래 인월은 SAP의 공식 인력 산정치나 업계 평균이 아니다. 앞의 시나리오를 비교하기 위해 둔 작업량 가정이다.

```text
손익분기 연간 총원가
= 12 × (IS 연 목록가 + IS 방식의 기타 증분비용 - 직접 방식의 기타 증분비용)
  ÷ (직접 방식 연간 인월 - IS 방식 연간 인월)
```

이 산식은 **직접 방식의 인월이 IS 방식보다 큰 경우에만** 쓴다. 손익분기 연간 총원가보다 인력의 실제 총원가가 높으면, 해당 가정 아래에서는 IS 방식이 비용 면에서 유리해진다. 반대로 낮으면 직접 방식이 유리하다. 양쪽 방식에 공통으로 드는 비용은 계산에서 제외한다.

> **연봉과 구분할 것**: 아래 값은 기본급이 아니라 회사가 부담하는 연간 총원가다. 기본급 연봉으로 환산하려면 `기본급 연봉 = 연간 총원가 ÷ 회사의 총원가 배수`를 사용한다. 이 배수는 고용 형태·복리후생·간접비·외주 여부에 따라 다르므로 문서에서 임의의 공통값을 정하지 않는다.

| 시나리오 | 직접 방식 가정 | IS 방식 가정 | 적용 IS 목록가 | 손익분기 연간 총원가 | 1억 2,000만 원 총원가에서의 비교 |
|---|---:|---:|---:|---:|---|
| A. 내부 Fiori 앱 한 개가 OData 소비 | 3 인월 | 3 인월 | Starter 약 3,172만 원 | 해당 없음 | 인월 절감이 없으므로 직접 방식이 유리 |
| B. 외부 파트너·여러 앱에 API 공개 | 10 인월 | 5 인월 | Standard 약 9,802만 원 | 약 2억 3,525만 원 | 직접 1억 원 / IS 약 1억 4,802만 원 → 직접 방식이 낮음 |
| C. 두 ERP 간 변환·재처리 연동 | 6 인월 | 3 인월 | Starter 약 3,172만 원 | 약 1억 2,689만 원 | 직접 6,000만 원 / IS 약 6,172만 원 → 거의 비슷하나 직접 방식이 낮음 |
| D. 여러 거래처와 B2B/EDI 운영 | 18 인월 | 8 인월 | Standard 약 9,802만 원 | 약 1억 1,762만 원 | 직접 1억 8,000만 원 / IS 약 1억 7,802만 원 → 거의 손익분기 |
| E. 하나의 이벤트를 여러 시스템이 구독 | 14 인월 | 7 인월 | Standard 약 9,802만 원 | 약 1억 6,804만 원 | 직접 1억 4,000만 원 / IS 약 1억 6,802만 원 → 직접 방식이 낮음 |

### 표를 읽는 방법

1. **A는 IS가 인력을 줄이지 않는 상황**이다. 단일 OData 소비에 중간 계층을 더하면 IS 구현·운영 인월도 생기므로, 구독료까지 추가한 IS가 비용상 이길 근거가 약하다.
2. **B는 API 정책·개발자 포털·분석을 IS로 공통화해 5 인월을 줄인다는 가정**이다. 개발자 연간 총원가가 약 2억 3,525만 원보다 낮다면 첫해 비용만으로는 직접 방식이 낮다. 실제로는 API 수와 소비자 증가에 따라 직접 방식의 인월이 커지는지 재산정해야 한다.
3. **C는 Starter 구독료와 3 인월 절감이 거의 맞닿는 상황**이다. 1억 2,000만 원 기준으로 차이가 약 172만 원에 불과하므로, 비용 외에 재처리·모니터링·향후 연동 확장성을 함께 본다.
4. **D는 파트너 수가 늘수록 직접 방식 인월이 누적되는 상황**이다. 위 가정에서는 1억 2,000만 원에서 거의 손익분기다. VAN·EDI 전문 서비스가 이미 있다면 그 서비스의 증분비용을 `직접 방식의 기타 증분비용`에 넣어 다시 계산한다.
5. **E는 이벤트 소비자 수에 따라 결과가 크게 달라지는 상황**이다. 소비자가 더 늘어 직접 방식이 14 인월을 초과하거나, IS 운영 인월이 7 인월보다 작으면 손익분기 연간 총원가는 낮아진다.

### 실제 조직 값으로 바꾸는 절차

1. 직접 방식과 IS 방식의 초기 구현, 연간 변경, 운영, 장애 대응을 각각 인월로 적는다.
2. 각 방식의 인월을 합산한다. 인력이 다른 역할이라면 ABAP, BTP, 통합, 보안, 운영 인력의 인월을 모두 합산한다.
3. 월 총원가가 아니라 **연간 총원가**를 정한다. 내부 인력과 외주 인력은 별도로 계산한다.
4. 현재 보유한 API Gateway, ESB, VAN, 클라우드 서비스가 있다면 신규 도입가가 아니라 **증분비용**만 넣는다.
5. 12개월과 36개월을 각각 계산한다. 초기 구축 인월은 첫해에만, 운영 인월은 매년 넣는다.

## 4. 시나리오별 비교 예시

아래의 인월은 실제 견적이 아니라 비교 방법을 보여 주기 위한 **가정 예시**다. `개발자 월 총원가 = 1,000만 원`으로만 통일했다. 각 조직은 자체 작업분해(WBS)와 인건비로 바꿔 계산해야 한다.

### 시나리오 A — 내부 Fiori 앱 한 개가 S/4HANA OData를 소비

| 항목 | 직접 OData | Integration Suite 경유 |
|---|---|---|
| 요구 | 단일 소비자, 표준 OData, 외부 공개 없음 | 같은 요구에 중간 계층을 추가 |
| 필요한 기능 | S/4HANA OData, 기존 BTP 인증·권한 | 단순 변환만이면 Starter 가능 범위 검토. API Management가 필요하면 Standard 필요 |
| 구독료 기준 | 추가 IS 구독료 없음 | Starter 연 약 3,172만 원 또는 API Management가 필요하면 Standard 연 약 9,802만 원 |
| 비용 해석 | 직접 구현·운영이 3.17 인월 미만이면 Starter 구독료보다 낮다 | 단일 API만으로는 구독료를 상쇄하기 어렵다 |

**판단**: 기존 S/4HANA OData와 BTP 보안으로 요구를 충족한다면 직접 연동이 기본 선택이다. IS는 소비자 증가, 데이터 변환, 공통 정책, 다중 backend, 운영 추적 요구가 생길 때 재검토한다.

### 시나리오 B — 외부 파트너와 여러 앱에 API를 공개

| 항목 | 개별 개발 또는 기존 API Gateway | Integration Suite Standard |
|---|---|---|
| 요구 | 여러 소비자, OAuth·정책·분석·개발자 온보딩 | API 수명주기와 정책을 중앙 관리 |
| 공식 기능 근거 | 기존 플랫폼 보유 여부는 조직별 사실 | Standard에 API Management와 개발자 포털 생성이 포함됨 |
| 공개 목록가 | 기존 플랫폼의 증분비용 확인 필요 | 연 약 9,802만 원 = 개발자 월 총원가 1,000만 원 기준 9.80 인월 |
| 가정 예시 | 초기 5 인월 + 연간 변경·운영 5 인월 = 10 인월 → 1억 원 | 구독료만 약 9,802만 원, 여기에 IS 구현·운영 인월 추가 |

**판단**: 위 가정에서는 직접 방식의 작업비와 Standard 구독료가 비슷해 보인다. 그러나 IS 구현·운영 비용까지 넣으면 첫해에 IS가 더 비쌀 수 있다. 반대로 API 수와 소비자가 빠르게 늘어나거나 기존 Gateway가 없다면, 정책·포털·분석을 각각 구현하는 비용이 매년 누적되므로 2~3년 TCO를 별도로 계산해야 한다.

### 시나리오 C — 두 SAP ERP의 업무 데이터를 변환·재처리하며 연결

| 항목 | 직접 IDoc/RFC/OData 또는 파일 교환 | Integration Suite Starter 범위 검토 |
|---|---|---|
| 요구 | 인터페이스 1~2개, 변경 적음 | 데이터 변환, 재시도, 모니터링, 향후 SaaS 확장 |
| 공개 목록가 | 추가 IS 구독료 없음 | Starter 연 약 3,172만 원 = 1,000만 원 기준 3.17 인월 |
| 가정 예시 | 초기 2 인월 + 연간 운영·변경 1 인월 = 3 인월 → 3,000만 원 | 구독료만 약 3,172만 원, IS 구현·운영 인월 별도 |

**판단**: 이 예시의 첫해 비용만 보면 직접 방식이 낮다. 다만 재처리·오류 추적·매핑 변경이 실제로 반복되면 운영 인월이 빠르게 증가할 수 있다. 인터페이스가 10개 한도를 넘거나 API/B2B/이벤트 기능이 필요하면 Starter만으로 판단하지 말고 Standard 범위와 비용을 다시 계산한다.

### 시나리오 D — 여러 거래처와 B2B/EDI를 운영

| 항목 | 거래처별 개별 연결 | Integration Suite Standard |
|---|---|---|
| 요구 | 파트너별 메시지 표준, 매핑, 인증서, 오류 처리 | B2B 전자 교환 라이브러리, Integration Advisor, 거래 파트너 관리 |
| 공개 목록가 | VAN·EDI 전문 서비스 또는 자체 개발의 별도 견적 필요 | 연 약 9,802만 원 = 1,000만 원 기준 9.80 인월 |
| 가정 예시 | 공통 기반 1 인월 + 거래처 3곳 × 각 3 인월 = 10 인월 → 1억 원 | 구독료만 약 9,802만 원, IS 구현·운영 인월 별도 |

**판단**: 거래처 3곳에 대한 단발성 구축비만으로 IS가 자동으로 저렴해지는 것은 아니다. 그러나 거래처 수·표준 종류·인증서 갱신·오류 대응이 증가하면 개별 연결의 운영 인월이 누적된다. 이미 VAN/EDI 전문 서비스를 계약했다면 그 계약의 증분 단가와 Standard를 비교해야 한다.

### 시나리오 E — 한 업무 이벤트를 여러 시스템이 구독

| 항목 | 생산자가 각 소비자를 직접 호출 | Integration Suite Standard |
|---|---|---|
| 요구 | 여러 소비자, 비동기 처리, 소비자 추가·제거 | 이벤트 발행·소비와 운영 가시성 |
| 공개 목록가 | 추가 IS 구독료 없음 또는 기존 메시징 비용 | 연 약 9,802만 원 = 1,000만 원 기준 9.80 인월 |
| 가정 예시 | 공통 2 인월 + 소비자 6곳 × 각 1.5 인월 = 11 인월 → 1억 1,000만 원 | 구독료만 약 9,802만 원, IS 구현·운영 인월 별도 |

**판단**: 소비자 수가 적고 동기 호출로 충분하면 직접 방식이 단순하다. 소비자가 늘면 생산자 변경, 실패 재처리, 추적이 반복될 수 있다. 이때는 소비자별 개발비만이 아니라 생산자 수정과 장애 분석에 드는 인월을 모두 직접 연동 TCO에 넣는다.

## 5. 실제 견적을 위한 최소 입력표

| 입력값 | 직접 방식 | Integration Suite 방식 |
|---|---|---|
| 초기 구현 인월 | 인터페이스별 개발·테스트·보안 | iFlow/API/이벤트/파트너 구성·테스트 |
| 연간 변경 인월 | 변경 요청, 소비자 추가, 매핑 수정 | 콘텐츠·정책·매핑 변경 |
| 연간 운영 인월 | 모니터링, 재처리, 인증서, 장애 분석 | IS Monitor, 재처리, 보안·인증서, 장애 분석 |
| 플랫폼 비용 | 기존 Gateway/ESB/VAN/클라우드의 증분비용 | 선택 에디션, 테넌트, 옵션, 초과 메시지 |
| 위험 비용 | 장애·감사·보안 통제의 예상 대응비 | 같은 항목을 IS 운영 모델 기준으로 추정 |

이 표에 12개월과 36개월 값을 각각 넣는다. **초기 구축비가 큰 경우에는 1년 비교만 하면 직접 방식이 과도하게 유리하거나 불리하게 보일 수 있다.**
