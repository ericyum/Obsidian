# index.html 완전 이해

> **대상**: `detailPage/webapp/index.html`
> **역할**: 앱의 **시작점**. UI5 엔진을 CDN에서 로드하고, 리소스 경로를 등록하고, Component 자동 생성을 트리거한다. 이 파일 없이는 아무것도 시작되지 않는다.

---

## 전체 코드

```html
<!DOCTYPE html>
<html>
<head>
    <meta http-equiv="X-UA-Compatible" content="IE=edge" />
    <meta charset="utf-8" />
    <title>UI5 Application: detailPage</title>
    <script
        id="sap-ui-bootstrap"
        src="https://ui5.sap.com/1.136/resources/sap-ui-core.js"
        data-sap-ui-theme="sap_fiori_3"
        data-sap-ui-resourceroots='{
            "detailPage": "./",
            "common_ott.lib": "/common_ott.lib/"
        }'
        data-sap-ui-compatVersion="edge"
        data-sap-ui-async="true"
        data-sap-ui-frameOptions="trusted"
        data-sap-ui-onInit="module:sap/ui/core/ComponentSupport"
    ></script>
</head>
<body class="sapUiBody sapUiSizeCompact" id="content">
    <div
        data-sap-ui-component
        data-name="detailPage"
        data-id="container"
        data-settings='{"id" : "detailPageApp"}'
    ></div>
</body>
</html>
```

27줄. HTML 자체는 거의 비어 있다. 모든 것은 `<script>` 태그 하나와 `<div>` 하나에 집약되어 있다.

## 📋 한눈에 보는 핵심 요소

| 요소 | 하는 일 |
|------|---------|
| `<script src="...sap-ui-core.js">` | SAP CDN에서 UI5 코어 엔진 로드 (1.136 버전) |
| `data-sap-ui-resourceroots` | UI5 모듈 로더에게 **네임스페이스 → URL 매핑** 제공 |
| `data-sap-ui-async="true"` | ⚡ **절대 필수** — 비동기 모듈 로딩 활성화 |
| `data-sap-ui-onInit="module:...ComponentSupport"` | 페이지 로드 완료 시 **자동으로 Component 생성** |
| `<div data-sap-ui-component>` | Component가 렌더링될 **빈 자리(placeholder)** |
| `data-name="detailPage"` | 어떤 Component를 만들지 지정 → `detailPage.Component` |
| `data-settings='{"id":"detailPageApp"}'` | Component 인스턴스에 전달할 설정값 |

## ① CDN 부트스트랩 — `sap-ui-core.js`

```html
<script
    id="sap-ui-bootstrap"
    src="https://ui5.sap.com/1.136/resources/sap-ui-core.js"
    ...
></script>
```

UI5의 엔진을 SAP 공식 CDN에서 가져온다. 이 스크립트가 로드되면 `sap.ui.define`, `sap.ui.Component` 등 모든 UI5 API가 사용 가능해진다.

> `id="sap-ui-bootstrap"`은 UI5가 이 `<script>` 태그를 찾는 식별자다. 이름이 정확히 일치해야 한다. 그냥 아무 `<script>`에 `data-sap-ui-*`를 붙인다고 작동하지 않는다.

버전 `1.136` — 이 버전에 따라 사용 가능한 컨트롤이 결정된다. 과거에 `f:GridList` + `layout:GridBox`가 안 됐던 이유도 1.136 CDN이 `GridBox`를 미제공했기 때문이다. `ObjectPageLayout` 등 `sap.uxap`은 1.136에서 사용 가능.

## ② `data-sap-ui-resourceroots` — 네임스페이스 → URL 매핑

```html
data-sap-ui-resourceroots='{
    "detailPage": "./",
    "common_ott.lib": "/common_ott.lib/"
}'
```

**UI5 모듈 로더의 근본적인 경로 변환 규칙을 정의한다.** 이 속성이 없으면 UI5는 어떤 모듈도 찾지 못한다.

### 어떻게 작동하는가

```
모듈명 "detailPage.Component" 를 로드해야 할 때:

① 네임스페이스 확인: "detailPage" → resourceroots에 있음 → 매핑값 "./"
② 네임스페이스 부분을 "./" 로 교체, 나머지는 점(.)→슬래시(/) 변환 + .js 추가
③ 최종 URL: "./Component.js"
```

| 모듈명                                        | resourceroots 변환                       | 최종 URL                                         |
| ------------------------------------------ | -------------------------------------- | ---------------------------------------------- |
| `detailPage.Component`                     | `"detailPage": "./"`                   | `./Component.js`                               |
| `detailPage.view.App`                      | `"detailPage": "./"`                   | `./view/App.view.xml`                          |
| `detailPage.i18n.i18n`                     | `"detailPage": "./"`                   | `./i18n/i18n.properties`                       |
| `common_ott.lib.controller.BaseController` | `"common_ott.lib": "/common_ott.lib/"` | `/common_ott.lib/controller/BaseController.js` |

### 두 개의 resourceroots가 필요한 이유

| 네임스페이스 | 실제 파일 위치 | 왜 필요한가 |
|:---|------|------|
| `"detailPage": "./"` | detailPage/webapp/ 아래 | 자기 자신의 뷰·컨트롤러·i18n을 찾기 위해 (기본값) |
| `"common_ott.lib": "/common_ott.lib/"` | library_ott/src/ 아래 (완전히 다른 폴더) | 공통 라이브러리 BaseController 등을 찾기 위해 |

> 💡 **Component.js와의 관계**: `index.html`의 `resourceroots`에 이미 `"common_ott.lib"`가 등록되어 있으므로, Component.js의 `registerModulePath`는 **사실상 중복**이다. 하지만 Component.js에도 넣어둔 이유는 **방어적 코딩** — 다른 앱에서 index.html 설정을 깜빡해도 Component.js가 자체 등록하기 때문이다.

## ③ `data-sap-ui-async="true"` — ⚡ 결정적 한 줄

```html
data-sap-ui-async="true"
```

**이 속성이 없으면 UI5는 완전히 다르게 동작한다.** `async: true`가 의미하는 것:

| `async: true` | `async: false` (또는 누락) |
|:---|:---|
| 모듈을 **비동기**로 로드 (AMD 표준) | 모듈을 **동기**로 로드 |
| 필요한 모듈만 그때그때 fetch | 모든 의존성을 한 번에 로드 |
| 페이지 로딩 빠름 | 페이지 로딩 느림 |
| **현대 UI5의 표준** | 구버전 방식, 곧 deprecated |

실제로 과거에 `async: true`가 누락되어 앱이 통째로 깨진 적이 있다. "절대 빼먹지 말 것" 리스트 1순위.

## ④ `data-sap-ui-onInit="module:sap/ui/core/ComponentSupport"` — 자동 생성 마법

```html
data-sap-ui-onInit="module:sap/ui/core/ComponentSupport"
```

UI5 코어 로딩이 **완료된 후** 자동으로 `sap/ui/core/ComponentSupport` 모듈을 실행하라는 지시다.

### ComponentSupport가 하는 일

```
1. DOM 전체를 스캔
2. data-sap-ui-component 속성이 붙은 <div>를 찾는다
3. 그 div의 data-name 속성을 읽는다 → "detailPage"
4. "detailPage.Component" 모듈을 로드한다
5. data-settings의 설정으로 new detailPage.Component({id: "detailPageApp"}) 호출
6. 생성된 Component 인스턴스를 해당 <div> 안에 렌더링
```

즉 개발자가 직접 `new Component()`를 호출할 필요가 전혀 없다. **선언적 자동 생성** 패턴이다.

```
index.html 로드 → sap-ui-core.js 로드 → resourceroots 등록
    → async 모듈 시스템 준비 완료
    → ComponentSupport 발동
        → <div data-name="detailPage"> 발견
        → detailPage/Component.js 로드
        → new detailPage.Component({id: "detailPageApp"})
        → init() 실행
```

## ⑤ `<div data-sap-ui-component>` — 빈 자리

```html
<div
    data-sap-ui-component
    data-name="detailPage"
    data-id="container"
    data-settings='{"id" : "detailPageApp"}'
></div>
```

| 속성                      | 값                         | 의미                                                  |
| ----------------------- | ------------------------- | --------------------------------------------------- |
| `data-sap-ui-component` | (속성 존재 자체)                | 이 div는 Component가 들어갈 자리다 — ComponentSupport가 찾는 표식 |
| `data-name`             | `"detailPage"`            | 로드할 Component 이름 → `detailPage.Component`           |
| `data-id`               | `"container"`             | 이 div 자체의 DOM ID                                    |
| `data-settings`         | `{"id": "detailPageApp"}` | Component 생성자에 전달할 설정. `id`는 Component 인스턴스의 ID가 된다 |

### 세 가지 ID 구분 (중요!)

`data-settings`의 `id`는 다른 ID들과 혼동하지 말아야 한다:

| ID | 위치 | 대상 |
|----|------|------|
| `data-settings.id = "detailPageApp"` | index.html | **Component 인스턴스**의 ID |
| manifest의 `rootView.id = "App"` | manifest.json | **루트 뷰(View) 객체**의 ID |
| manifest의 `routing.config.controlId = "app"` | manifest.json | 뷰 내부 **`<App>` 컨트롤**의 ID |

세 ID는 각각 다른 대상을 가리킨다. 전부 다르다.

### DOM 중첩 구조 — "러시아 인형"

그렇다면 이 작은 `<div>` 안에 실제로 무엇이 들어가는가? Component 인스턴스, App 뷰, DetailMain 뷰가 차례로 중첩되어 들어간다.

```
<body>
  <div id="container" data-sap-ui-component>   ← index.html에 있던 빈 div
    └─ <div id="detailPageApp">                ← Component 인스턴스가 만든 래퍼
         └─ <div id="App">                     ← rootView (App.view.xml → mvc:View)
              └─ <div id="app">                ← sap.m.App 컨트롤
                   └─ <div id="app-pages">     ← pages aggregation 컨테이너
                        └─ <div id="DetailMain"> ← Router가 동적으로 삽입한 뷰
                             └─ ... 실제 컨텐츠 (ObjectPageLayout) ...
```

**계층별 설명**:

| 계층 | ID | 출처 | 언제 생성되는가 |
|:---|:---|:---|:---|
| ① 빈 div | `container` | index.html 정적 HTML | 페이지 로드 시 이미 존재 |
| ② Component 래퍼 | `detailPageApp` | ComponentSupport가 자동 생성 | Component 인스턴스 생성 시 |
| ③ 루트 뷰 | `App` | manifest.rootView → App.view.xml | 부모 init() 실행 시 |
| ④ App 컨트롤 | `app` | App.view.xml 안의 `<App id="app">` | App.view.xml 렌더링 시 |
| ⑤ pages 컨테이너 | `app-pages` | sap.m.App 내부 자동 생성 | App 컨트롤 생성 시 |
| ⑥ 타겟 뷰 | `DetailMain` | Router.initialize() → DetailMain.view.xml | URL 해시 매칭 시 |

**핵심**: index.html의 그 작은 `<div>` 하나가 최종적으로는 전체 화면을 담는 뿌리가 된다. 마트료시카(러시아 인형)처럼 Component → App 뷰 → DetailMain 뷰가 차례로 중첩된다.

## ⑥ 기타 속성들

```html
<meta http-equiv="X-UA-Compatible" content="IE=edge" />
```
Internet Explorer에서 최신 렌더링 엔진 사용하도록 강제. (사실상 레거시 지원용)

```html
<meta charset="utf-8" />
```
문자 인코딩 UTF-8. 한글 i18n 파일들을 위해 필수.

```html
data-sap-ui-theme="sap_fiori_3"
```
UI5 테마. `sap_fiori_3`는 Fiori 3 디자인 시스템을 적용한다. 모든 UI 컨트롤의 기본 외형을 결정.

```html
data-sap-ui-compatVersion="edge"
```
최신 호환성 모드 사용. 구버전 API 호환성 레이어 없이 가장 최신 동작을 따른다.

```html
data-sap-ui-frameOptions="trusted"
```
X-Frame-Options 헤더 값. `trusted`는 같은 도메인 내 iframe 임베딩을 허용한다.

```html
<body class="sapUiBody sapUiSizeCompact" id="content">
```
- `class="sapUiBody"` — UI5가 body에 요구하는 표준 클래스. UI5 글로벌 스타일 적용.
- `sapUiSizeCompact` — 기본 컨트롤 사이즈를 Compact(작게)로 설정. Fiori 벨리즈 테마에서 주로 사용.

## 🔄 index.html 중심의 전체 생명주기

```
① 브라우저가 index.html 요청
   → 서버는 HTML 파일만 반환 (해시 # 이후는 전송 안 됨)
   → index.html이 유일한 진입점. 이 파일 없인 아무 일도 안 일어난다

② <script src="...sap-ui-core.js"> 로드 시작 (SAP CDN)
   → UI5 코어 엔진 초기화
   → sap.ui.define, sap.ui.Component 등 모든 API 사용 가능

③ data-sap-ui-* 속성들 처리 (부트스트랩 설정)
   ├─ resourceroots 등록 ("detailPage": "./", "common_ott.lib": "/common_ott.lib/")
   ├─ async 모드 활성화
   ├─ 테마 "sap_fiori_3" 적용
   └─ compatVersion "edge" 설정

④ DOM 로드 완료 → data-sap-ui-onInit 발동
   → ComponentSupport 모듈 로드

⑤ ComponentSupport가 DOM 스캔
   → data-sap-ui-component 속성 가진 <div> 발견

⑥ data-name="detailPage" → detailPage/Component.js 모듈 로드
   → new detailPage.Component({id: "detailPageApp"})
   → Component.init() 실행
       ├─ registerModulePath("common_ott.lib", "/common_ott.lib/")
       │   └─ (resourceroots와 중복이지만 방어적 등록)
       │
       ├─ UIComponent.prototype.init()
       │   ├─ manifest.json 파싱 → 모든 설정 메모리 로드
       │   ├─ 모델 생성 (i18n, detail⚡$metadata, viewModel)
       │   ├─ rootView(App.view.xml) 로드 → 빈 셸 렌더링
       │   └─ style.css 로드
       │
       └─ Router.initialize()
           ├─ ⑴ 현재 해시(#/Detail/C001) 즉시 처리
           ├─ ⑵ 향후 해시 변경 감시 리스너 등록
           ├─ DetailMain.view.xml 내부 HTTP 요청 → 로드
           ├─ DetailMain.controller.js 인스턴스 생성 → onInit()
           └─ App의 pages에 뷰 삽입 → 화면 표시 완료
```

## ❓ 자주 묻는 질문

### Q1: `data-sap-ui-resourceroots`와 Component.js의 `registerModulePath`는 무슨 관계인가?

**기능적으로 완전히 동일하다.** 둘 다 UI5 모듈 로더에게 "네임스페이스 X는 URL Y에서 찾아라"라고 알려준다. `index.html`에 `resourceroots`로 먼저 등록되고, Component.js의 `init()`에서 `registerModulePath`로 다시 등록된다. **중복이지만 문제는 없다.** 오히려 한쪽이 빠져도 다른 한쪽이 커버하는 방어적 구조다.

### Q2: 왜 `<div>`에 Component를 렌더링하는 방식인가? `body`에 직접 하면 안 되나?

`ComponentSupport` 패턴의 장점은:
- 하나의 HTML에 여러 Component를 각각 다른 `<div>`에 렌더링할 수 있다.
- Component가 렌더링될 DOM 위치를 선언적으로 지정할 수 있다.
- 테스트나 FLP(Launchpad) 통합 시 유연하게 대응 가능하다.

### Q3: CDN이 아니라 로컬에 UI5를 설치해서 쓸 수도 있나?

가능하다. CDN 대신 로컬 경로를 쓰려면 `src`를 `"./resources/sap-ui-core.js"` 등으로 변경하고 UI5 라이브러리를 직접 프로젝트에 포함시키면 된다. 하지만 CDN 방식이 가장 간단하고 SAP의 최적화된 엣지 서버를 활용할 수 있어 일반적으로 권장된다.

### Q4: `data-sap-ui-onInit` 대신 수동으로 Component를 생성할 수도 있나?

가능하다. `data-sap-ui-onInit`를 빼고, index.html 하단에 별도 `<script>`를 두어 `sap.ui.getCore().attachInit(function() { new detailPage.Component(...) })` 식으로 수동 생성할 수 있다. 하지만 ComponentSupport 패턴이 SAP의 현대적 권장 방식이다.

| # | 포인트 |
|---|--------|
| 1 | `index.html`의 역할은 **시작점** — UI5 엔진을 CDN에서 로드하고, 리소스 경로를 등록하고, Component 자동 생성을 트리거한다 |

| 5 | `data-name="detailPage"` → `detailPage.Component`를 찾아 인스턴스 생성. 여기서부터 manifest.json → init() → Router의 연쇄 반응 시작 |
| 6 | `data-settings='{"id":"detailPageApp"}'`는 Component 인스턴스 ID. rootView.id("App")나 controlId("app")와 다름 |
| 7 | 해시(`#/Detail/C001`)는 서버에 전송되지 않음 — index.html 반환 후 브라우저에서 Component.init() → Router가 해시를 읽어 처리한다 |
| 8 | `resourceroots`의 `common_ott.lib` 등록 = Component.js의 `registerModulePath`와 기능적 중복 — 방어적 코딩 |

## 🔗 3대 파일 연결 관계 (최종)

```
index.html                    Component.js                manifest.json
──────────                    ─────────────               ─────────────
resourceroots ──────────▶     registerModulePath          (관련 없음)
                              (중복 등록, 방어적)

ComponentSupport ───────▶     new Component() → init()
                                        │
                              부모 init() ───────────▶    manifest.json 파싱
                              Router.init() ─────────▶    routes + targets 참조
```
