# 6. DetailMain.controller.js 완전 이해

**날짜**: 2026-07-06
**범위**: `detailPage/webapp/controller/DetailMain.controller.js` — 컨텐츠 상세 페이지의 핵심 비즈니스 로직
**이전 단계**: [[5. DetailMain.view.xml_완전_이해]], [[4. BaseController.js_완전_이해]], [[2. Component.js_완전_이해]]

## 개요: 이 파일의 정체

`DetailMain.controller.js`는 **detailPage의 두뇌**다. 뷰(DetailMain.view.xml)가 "어떻게 보일지"를 담당한다면, 이 파일은 "어떻게 동작할지"를 담당한다. 사용자가 URL을 통해 페이지에 진입하는 순간부터, 데이터를 가져오고, 리뷰를 쓰고 지우고, 엑셀을 다운받는 모든 동작이 여기에 정의되어 있다.

**코드량**: 약 230줄
**연결된 뷰**: `detailPage.view.DetailMain` (view.xml의 `controllerName` 속성)
**사용 모델**: `viewModel` (JSONModel), `detail` (OData V2), `User` (Core)
**상속**: `DetailMain` → `detailPage.BaseController` (껍데기) → `library_ott.BaseController` (진짜 로직)

| 요소 | 설명 |
|------|------|
| `sap.ui.define` | UI5의 AMD 모듈 정의 함수 |
| **배열** (1번째 인자) | 의존성 목록. 이 모듈들을 **먼저** 로드한 후 팩토리 함수 실행 |
| **팩토리 함수** (2번째 인자) | 의존성이 모두 로드되면 실행. 인자 순서가 배열 순서와 1:1 매핑 |
| `"use strict"` | 엄격 모드. 선언되지 않은 변수 사용 금지 등 JS 오류 조기 발견 |
| `return BaseController.extend(...)` | 이 파일의 "수출품". UI5 컴포넌트가 이것을 가져다 씀 |

## 1. 상속 체인 — DetailMain이 물려받은 것들

```
sap.ui.core.mvc.Controller
    └─ common_ott.lib.controller.BaseController  ← 진짜 로직이 있는 부모
        │   ├─ onInit() → UserSessionLoader, CommonCodeLoader 등 초기화
        │   ├─ getResourceBundle() → i18n 편의 메서드
        │   ├─ readOdata() → Promise 기반 OData read 래퍼
        │   └─ fnDeepCopy(), fnMasterCodeValue() 등 유틸
        │
        └─ detailPage.controller.BaseController  ← 빈 껍데기 (확장 포인트)
            │   (아무 코드도 없음 — 미래에 detailPage 공통 로직을 넣을 자리)
            │
            └─ detailPage.controller.DetailMain  ← 실제 컨트롤러 (이 파일)
```

### 상속이 주는 혜택

`DetailMain`은 `extend("detailPage.controller.DetailMain")`만으로도 **자동으로** 다음을 사용할 수 있다:

| 물려받은 것 | 어디서 왔나 | DetailMain에서 어떻게 쓰나 |
|-------------|-----------|--------------------------|
| `this.getResourceBundle()` | library_ott BaseController | `this.i18n`에 저장해서 다국어 텍스트 조회 |
| `this.readOdata({...})` | library_ott BaseController | `_loadDetail()`에서 OData 호출. Promise 반환하므로 `.then()` 체이닝 가능 |
| `this.i18n` | library_ott BaseController onInit | `this.i18n.getText("키")` 로 i18n 텍스트 조회 |
| User/Core 모델 | library_ott BaseController onInit | `sap.ui.getCore().getModel("User")`로 로그인 사용자 정보 접근 |

## 2. 포매터 — `formatReviewCount`

```javascript
formatReviewCount: function (nCount) {
    return "(" + this.getResourceBundle().getText("label.review") + " " +
           (nCount || 0) +
           this.getResourceBundle().getText("label.countUnit") + ")";
}
```

**언제 호출되나**: 뷰의 Expression Binding에서 참조될 때 자동 호출

```xml
<!-- DetailMain.view.xml -->
<Text text="{path: 'viewModel>/Detail/rating_count', formatter: '.formatReviewCount'}" />
```

**동작**:
```
nCount = 4  →  "(리뷰 4개)"
nCount = 0  →  "(리뷰 0개)"    ← nCount || 0 으로 undefined/null 방어
nCount = undefined → "(리뷰 0개)"
```

> **주의**: 포매터는 컨트롤러의 메서드이므로 `this`가 DetailMain 인스턴스를 가리킨다. 그래서 `this.getResourceBundle()`을 쓸 수 있다. 단, 바인딩의 `formatter`에 전달될 때는 UI5가 `this`를 자동으로 바인딩해준다.

## 3. 생명주기 메서드

### 3.1 `onInit()` — 탄생

```javascript
onInit: function () {
    BaseController.prototype.onInit.call(this);  // ① 부모 초기화
    this.i18n = this.getResourceBundle();         // ② i18n 편의 참조
    this.getOwnerComponent().getRouter()          // ③ 라우터 이벤트 구독
        .getRoute("DetailMain")
        .attachPatternMatched(this._onRouteMatched, this);
},
```

| 줄 | 코드 | 의미 |
|:--:|------|------|
| ① | `BaseController.prototype.onInit.call(this)` | `super.onInit()`. 부모(BaseController)의 onInit 실행 → UserSessionLoader, CommonCodeLoader 등 모든 비동기 로더 초기화 |
| ② | `this.i18n = this.getResourceBundle()` | `getResourceBundle()`은 library_ott BaseController에 정의된 편의 메서드. i18n 번들을 로컬 변수에 저장해서 `this.i18n.getText("키")`로 간편하게 사용 |
| ③ | `attachPatternMatched(...)` | **라우터 이벤트 구독**. URL 패턴이 `DetailMain`과 매칭될 때마다 `_onRouteMatched`를 호출하도록 등록. 두 번째 인자 `this`는 콜백 내부의 `this`가 DetailMain을 가리키도록 바인딩 |

> **핵심**: `onInit()`은 **딱 한 번만** 호출된다 (컨트롤러 인스턴스 생성 시). 이후 같은 컨트롤러로 다른 content_id를 조회할 때는 `_onRouteMatched()`만 다시 호출된다.

### 3.2 `onExit()` — 소멸

```javascript
onExit: function () {
    this._bDestroyed = true;            // 모든 콜백에서 참조하는 파괴 플래그
    if (this._checkTimer) {
        clearTimeout(this._checkTimer); // User 모델 대기 타이머 정리
        this._checkTimer = null;
    }
},
```

**왜 필요한가**: 사용자가 페이지를 떠난 후에도 비동기 콜백(타이머, AJAX 응답)이 실행되면 **이미 사라진 뷰**를 조작하려다 오류가 발생할 수 있다.

`_bDestroyed` 플래그는 이 파일의 **거의 모든 콜백**에서 확인한다:
```javascript
if (that._bDestroyed) return;  // "이미 죽었으면 아무것도 하지 마"
```

## 4. 라우트 처리 — `_onRouteMatched`

```javascript
_onRouteMatched: function (oEvent) {
    var contentId = oEvent.getParameter("arguments").content_id;

    // ★ NewReview 초기화 (없으면 RatingIndicator가 이전 값으로 남음)
    this.getOwnerComponent().getModel("viewModel").setProperty("/NewReview", {
        rating: 0,
        review_text: ""
    });

    // ★ retry count 초기화 (이전 콘텐츠의 잔여 카운트 방지)
    this._checkRetryCount = 0;

    this._loadDetail(contentId);  // 실제 데이터 로딩 시작
},
```

### URL에서 content_id 추출

```
URL: #/Detail/C001
              └─ content_id = "C001"
```

`manifest.json`의 `pattern: "Detail/{content_id}"`에 의해 `{content_id}` 부분이 파라미터로 추출된다.

### 초기화가 필요한 이유

사용자가 C001 → C002로 이동할 때 **컨트롤러는 재사용**된다. 이전 데이터가 남아있으면:

| 초기화하지 않으면 | 발생하는 문제 |
|-----------------|------------|
| `NewReview` 초기화 안 함 | C001에서 별점 5점 준 상태로 C002 이동 → RatingIndicator에 5점이 그대로 남음 |
| `_checkRetryCount` 초기화 안 함 | C001에서 User 모델 대기 5회 실패 → C002에서 바로 포기 |

## 5. 데이터 로딩 — `_loadDetail` (가장 핵심!)

```javascript
_loadDetail: function (contentId) {
    var oModel = this.getView().getModel("detail");          // OData V2 모델
    var oViewModel = this.getOwnerComponent().getModel("viewModel"); // JSON 모델
    var oView = this.getView();
    var that = this;

    // ★ Race condition 방지 토큰
    this._loadToken = (this._loadToken || 0) + 1;
    var token = this._loadToken;

    oView.setBusy(true);  // 로딩 인디케이터 ON

    this.readOdata({
        model: oModel,
        path: "/Contents('" + contentId + "')",
        param: {
            urlParameters: {
                "$expand": "partner,genres($expand=genre),tags($expand=tag),casts,reviews($expand=user)"
            }
        }
    }).then(function (oData) {
        if (token !== that._loadToken) return;  // ★ stale 응답 무시
        oViewModel.setProperty("/Detail", oData);
        if (token === that._loadToken) {
            that._checkMyReview();
        }
        oView.setBusy(false);
    }.bind(this)).catch(function () {
        oView.setBusy(false);
        MessageToast.show(this.i18n.getText("message.loadFail"));
    }.bind(this));
},
```

### 5.1 사용하는 두 모델

| 변수 | 모델 | 타입 | 용도 |
|------|------|------|------|
| `oModel` | `"detail"` | OData V2 | **HTTP 통신 담당**. `this.readOdata()`에 전달되어 실제 API 호출 |
| `oViewModel` | `"viewModel"` | JSONModel | **데이터 저장소**. 응답 데이터를 `setProperty("/Detail", oData)`로 저장 |

> **왜 모델이 두 개인가?** OData V2 모델(detail)은 read/write/callFunction 같은 통신 기능을 제공하지만, 데이터를 setProperty로 직접 조작하기엔 불편하다. 그래서 통신은 detail 모델로 하고, 받은 데이터는 viewModel(JSONModel)에 담아서 뷰와 바인딩한다.

### 5.2 `$expand` — 한 번에 다 가져오기

```
/Contents('C001') 만 호출하면:
→ { title: "...", content_type: "...", ... }
  (연결된 장르, 태그, 출연진, 리뷰는 안 따라옴)

/Contents('C001')?$expand=genres($expand=genre),reviews($expand=user) 호출하면:
→ {
    title: "...",
    genres: { results: [{ genre: { name: "액션" } }] },     ← 장르까지!
    reviews: { results: [{ user: { user_name: "홍길동" } }] } ← 리뷰 + 작성자까지!
  }
```

| $expand 항목 | 의미 | 왜 필요한가 |
|-------------|------|-----------|
| `partner` | 컨텐츠 제공사 정보 | 기본 정보 섹션에 제공사명 표시 |
| `genres($expand=genre)` | 장르 목록 → 각 장르의 상세 | 장르 칩에 장르명 표시 |
| `tags($expand=tag)` | 태그 목록 → 각 태그의 상세 | 태그 칩에 태그명 표시 |
| `casts` | 출연진 목록 | 출연진 테이블 |
| `reviews($expand=user)` | 리뷰 목록 → 각 리뷰의 작성자 | 리뷰 목록 + 작성자명 + 내 리뷰 확인 |

### 5.3 Token — Race Condition 방어

**문제 상황**: 사용자가 C001 → C002 → C003을 빠르게 연속 클릭하면?

```
요청1: /Contents('C001') → 응답 300ms 후 도착 예정
요청2: /Contents('C002') → 응답 200ms 후 도착 예정
요청3: /Contents('C003') → 응답 100ms 후 도착 예정
```

네트워크 상황에 따라 응답 순서가 요청 순서와 다를 수 있다. C003의 응답이 먼저 와서 viewModel을 업데이트했는데, 그 후 C001의 늦은 응답이 도착하면 **엉뚱한 데이터로 덮어쓰는 버그**가 발생한다.

**해결**:
```javascript
this._loadToken = (this._loadToken || 0) + 1;  // 요청마다 번호 증가
var token = this._loadToken;                    // 현재 요청의 번호 기억

// 응답 도착 시:
if (token !== that._loadToken) return;           // 더 최신 요청이 있으면 이 응답은 버림
```

```
요청1: token=1 → _loadToken=1
요청2: token=2 → _loadToken=2
요청3: token=3 → _loadToken=3

응답2 도착 (token=2, _loadToken=3) → 2 !== 3 → 버림 ✗
응답3 도착 (token=3, _loadToken=3) → 3 === 3 → 처리 ✓
응답1 도착 (token=1, _loadToken=3) → 1 !== 3 → 버림 ✗
```

### 5.4 `this.readOdata()` — Promise 기반 OData 호출

`readOdata`는 **library_ott BaseController**에 정의된 래퍼 함수다. OData V2의 `model.read()`를 Promise로 감싸서 `.then()` / `.catch()` 체이닝을 가능하게 한다.

> **참고**: V2 OData의 `model.callFunction()`은 Promise를 지원하지 않아서 `success`/`error` 콜백을 직접 써야 한다. 하지만 `model.read()`는 `readOdata` 래퍼 덕분에 `.then()`으로 편하게 쓸 수 있다.

### 5.5 `.bind(this)` — 콜백 안에서 `this` 유지

```javascript
.then(function (oData) {
    this.i18n.getText(...)  // 여기서 this는?
}.bind(this))
```

JavaScript에서 콜백 함수의 `this`는 기본적으로 호출 컨텍스트에 따라 달라진다. `.bind(this)`로 **DetailMain 인스턴스를 고정**해야 `this.i18n`에 접근할 수 있다.

### 5.6 `setBusy()` — 로딩 인디케이터

```javascript
oView.setBusy(true);   // "로딩 중..." 표시
// ...데이터 로딩...
oView.setBusy(false);  // 로딩 완료, UI 표시
```

## 6. 내 리뷰 확인 — `_checkMyReview`

```javascript
_checkMyReview: function () {
    if (this._bDestroyed) return;                          // ① 파괴 가드

    var oViewModel = this.getOwnerComponent().getModel("viewModel");
    var oUserModel = sap.ui.getCore().getModel("User");

    if (!oUserModel) {                                     // ② User 모델 아직 없음 → 재시도
        this._checkRetryCount = (this._checkRetryCount || 0) + 1;
        if (this._checkRetryCount <= 5) {
            this._checkTimer = setTimeout(
                this._checkMyReview.bind(this), 200
            );
        }
        return;
    }
    this._checkRetryCount = 0;                             // ③ 성공 — 카운트 리셋

    var myUserId = oUserModel.getProperty("/user_id");
    var aReviews = oViewModel.getProperty("/Detail/reviews/results") || [];

    var oMyReview = null;
    for (var i = 0; i < aReviews.length; i++) {            // ④ 전체 리뷰 중 내 것 찾기
        if (aReviews[i].user_user_id === myUserId) {
            oMyReview = aReviews[i];
            break;
        }
    }
    oViewModel.setProperty("/MyReview", oMyReview);         // ⑤ 찾은 리뷰 저장 (없으면 null)
    oViewModel.setProperty("/HasReviewed", !!oMyReview);    // ⑥ 있었으면 true, 없으면 false
},
```

### 6.1 User 모델 비동기 재시도

```
UserSessionLoader (비동기)         _checkMyReview()
───────────────                    ────────────────
로딩 중...      ─────────────────→  oUserModel = null
                                     → 200ms 후 재시도 #1
로딩 중...      ─────────────────→  oUserModel = null
                                     → 200ms 후 재시도 #2
로딩 완료!      ─────────────────→  oUserModel = {...}
                                     → myUserId = "U001"
                                     → 리뷰 목록에서 찾기 성공!
```

**왜 재시도가 필요한가**: `UserSessionLoader`는 HTTP 요청을 통해 사용자 정보를 가져오는 **비동기** 로더다. `_loadDetail()`의 OData 요청과 UserSessionLoader 중 누가 먼저 완료될지 알 수 없다. User 모델이 준비되지 않았으면 최대 5회(총 1초) 기다린다.

#### 6.1a User 모델은 어디서 오는가 — `UserSessionLoader` 깊이 이해하기

`sap.ui.getCore().getModel("User")`로 가져오는 이 모델은 `manifest.json` 어디에도 선언되어 있지 않다. **런타임에 동적으로 생성**되기 때문이다.

**흐름 요약**:

```
DetailMain.onInit()
  └─ BaseController.prototype.onInit.call(this)  // 부모 초기화
       └─ library_ott BaseController.onInit()
            ├─ new UserSessionLoader()  ← OData 요청 발사 (/UserSessionInfo)
            └─ .attachEventOnce('ready', function(oEvent) {
                   let oModel = oEvent.getParameter("model");
                   oCore.setModel(oModel, "User");  ← 코어에 등록
               })
```

**`UserSessionLoader` 내부 구조** (핵심만):

```javascript
// library_ott/src/util/UserSessionLoader.js
let staticModel;  // ← 모듈 레벨 변수 (싱글톤)

return EventProvider.extend("common_ott.lib.util.UserSessionLoader", {
    metadata: {
        events: {
            ready: { parameters: { model: { type: "object" } } }  // 커스텀 이벤트 선언
        }
    },

    constructor: function() {
        if (!staticModel) {
            staticModel = new JSONWithReadModel();           // ① 최초 1회만 모델 생성
            this._fetchUserSession().then(function(oData) {   // ② OData 호출
                staticModel.setReady(oData.results[0]);       // ③ 데이터 주입 → JSONWithReadModel ready 발생
            });
        }
    },

    _finalizeSession: function(sessionData) {
        if (!staticModel.isReady()) {
            staticModel.setReady(sessionData);  // 데이터 주입 (최초 시)
        }
        this.fireEvent("ready", { model: staticModel });  // ④ UserSessionLoader의 ready 발생
    }
});
```

**이벤트 릴레이 구조**:

```
_fetchUserSession() → OData 응답 도착
  ↓
_finalizeSession(sessionData)
  ├─ staticModel.setReady(sessionData)      ← JSONWithReadModel에 데이터 주입
  └─ this.fireEvent("ready", { model: staticModel })  ← UserSessionLoader의 ready 발생
        ↓
     BaseController의 attachEventOnce('ready', ...)
        ↓
     oCore.setModel(oModel, "User")        ← 코어에 "User"라는 이름으로 등록
```

**핵심 포인트**:

| 개념 | 설명 |
|------|------|
| `staticModel` | 모듈 레벨 변수. 모든 `UserSessionLoader` 인스턴스가 **하나의 모델을 공유** (싱글톤) |
| `attachEventOnce` | BaseController가 `ready` 이벤트를 **딱 한 번만** 수신. 중복 등록 방지 |
| 코어 등록 (`oCore.setModel`) | `sap.ui.getCore().getModel("User")`로 **전역 접근** 가능하게 함 |
| 뷰 등록 (`this.getView().setModel`) | XML 뷰에서 `{User>/user_id}` 형태로 **바인딩** 가능하게 함 |

> **왜 코어와 뷰 양쪽에 등록하나?** 둘 다 **같은 `staticModel` 객체**를 가리킨다. 코어 등록은 JS 코드(`sap.ui.getCore().getModel("User")`)에서, 뷰 등록은 XML 바인딩(`{User>/user_id}`)에서 사용하기 위함이다.

### 6.2 `onExit`에서 타이머 정리

```javascript
// onExit()
if (this._checkTimer) {
    clearTimeout(this._checkTimer);  // 대기 중인 타이머 취소
    this._checkTimer = null;
}
```

사용자가 타이머가 돌고 있는 중에 페이지를 떠나면, `onExit`에서 `clearTimeout`으로 취소한다. 안 그러면 사라진 컨트롤러에서 `_checkMyReview`가 실행되어 오류 발생.

| `/MyReview` | 내 리뷰 객체 (또는 `null`) | 수정 버튼 클릭 시 prefill 데이터로 사용 |

## 9. 리뷰 등록/수정 — `onSubmitReview` (가장 복잡한 함수)

```javascript
onSubmitReview: function () {
    if (this._bSubmitting) return;                          // ① 중복 제출 방지

    var oModel = this.getView().getModel("detail");
    var oViewModel = this.getOwnerComponent().getModel("viewModel");
    var contentId = oViewModel.getProperty("/Detail/content_id");
    var rating = oViewModel.getProperty("/NewReview/rating");
    var reviewText = oViewModel.getProperty("/NewReview/review_text");
    var editingReviewId = oViewModel.getProperty("/NewReview/editing_review_id");

    if (!rating || !reviewText) {                           // ② 입력값 검증
        MessageToast.show(this.i18n.getText("message.noReviewInput"));
        return;
    }

    this._bSubmitting = true;                                // ③ 제출 시작 — 잠금
    var that = this;

    var doSubmit = function () {                             // ④ 실제 등록 함수
        oModel.callFunction("/submitReview", {
            method: "POST",
            urlParameters: {
                content_id: contentId,
                rating: rating,
                review_text: reviewText
            },
            success: function () {
                if (that._bDestroyed) return;
                oViewModel.setProperty("/NewReview", { rating: 0, review_text: "" });
                that._loadDetail(contentId);                 // ★ 서버에서 전체 재조회
                MessageToast.show(that.i18n.getText(
                    editingReviewId ? "message.reviewUpdateSuccess" : "message.reviewSuccess"
                ));
                that._bSubmitting = false;
            },
            error: function () {
                if (that._bDestroyed) return;
                MessageToast.show(that.i18n.getText("message.reviewFail"));
                that._bSubmitting = false;
            }
        });
    };

    if (editingReviewId) {                                   // ⑤ 수정 모드: 기존 리뷰 먼저 삭제
        oModel.callFunction("/deleteReview", {
            method: "POST",
            urlParameters: { review_id: editingReviewId },
            success: function () { doSubmit(); },            // 삭제 성공 → 새로 등록
            error: function () {
                if (that._bDestroyed) return;
                MessageToast.show(that.i18n.getText("message.reviewEditFail"));
                that._bSubmitting = false;
            }
        });
    } else {
        doSubmit();                                          // ⑥ 신규 등록: 바로 등록
    }
},
```

### 9.1 중복 제출 방지

```javascript
if (this._bSubmitting) return;    // 이미 제출 중이면 무시
...
this._bSubmitting = true;         // 제출 시작 — 잠금
...
that._bSubmitting = false;        // 제출 완료 — 해제
```

사용자가 "등록" 버튼을 빠르게 연타해도 첫 번째 요청만 처리되고 나머지는 무시된다.

### 9.2 V2 OData `callFunction` — Promise 미지원

```javascript
// ★ callFunction은 Promise를 반환하지 않는다!
// 반드시 success/error 콜백을 직접 전달해야 함
oModel.callFunction("/submitReview", {
    method: "POST",
    urlParameters: { ... },
    success: function () { ... },  // ← 콜백으로 처리
    error: function () { ... }
});
```

> **주의**: `this.readOdata()`는 library_ott BaseController가 Promise로 감싼 래퍼지만, `callFunction`은 그런 래퍼가 없다. 그래서 `.then()`이 아닌 `success`/`error` 콜백을 직접 써야 한다.

### 9.3 수정 = 삭제 + 신규 등록

```javascript
if (editingReviewId) {
    // 1단계: 기존 리뷰 삭제
    callFunction("/deleteReview", { review_id: editingReviewId })
    // 2단계: 삭제 성공 → 새 리뷰 등록
    → doSubmit()  // callFunction("/submitReview", ...)
}
```

CAP 백엔드에 리뷰 수정 API가 따로 없어서, **삭제 후 재등록** 방식으로 구현했다. 삭제와 등록이 순차적으로 실행되어야 하므로 콜백을 중첩해서 순서를 보장한다.

### 9.4 서버 재조회 vs 수동 조작

**과거에는 이렇게 했다 (버그):**
```javascript
// 수동으로 배열에 추가 + 평점 재계산 → 사용자명 누락, 평점 버그
aReviews.unshift(oResult);
oViewModel.setProperty("/Detail/avg_rating", newAvg);
```

**현재:**
```javascript
that._loadDetail(contentId);  // 한 줄로 모든 데이터 정확하게 갱신
```

서버에서 전체 데이터를 다시 가져오면 `$expand=reviews($expand=user)`로 사용자명도 따라오고, 평균 평점도 서버가 정확하게 계산한 값으로 교체된다.

### 10.1 `onDeleteReview` — 목록의 🗑️ 버튼

```javascript
onDeleteReview: function (oEvent) {
    var oContext = oEvent.getSource().getBindingContext("viewModel");
    var reviewId = oContext.getProperty("review_id");

    MessageBox.warning(this.i18n.getText("message.deleteConfirm"), {

    });

```

### 10.2 `oEvent.getSource().getBindingContext()` — 이벤트 발생 위치에서 데이터 추출

```javascript
var oContext = oEvent.getSource().getBindingContext("viewModel");
var reviewId = oContext.getProperty("review_id");
```

이 코드가 하는 일:

1. `oEvent.getSource()` → 클릭된 🗑️ 버튼
2. `.getBindingContext("viewModel")` → 그 버튼이 리스트의 몇 번째 행에 있는지 → 해당 행의 viewModel 컨텍스트
3. `.getProperty("review_id")` → 그 행의 review_id 값

```
리뷰 목록에서 2번째 항목의 🗑️를 클릭:
→ 해당 행의 컨텍스트 = /Detail/reviews/results[1]
→ review_id = "R002"
```

> **참고 — `onDeleteMyReview`는 제거됨**: 7/2 리팩토링에서 "내 리뷰" 하위 섹션이 사라지면서, 더 이상 뷰에서 호출되지 않는 `onDeleteMyReview` 함수는 컨트롤러에서 삭제되었다. 현재 모든 리뷰 삭제는 리뷰 목록 항목의 🗑️ 버튼 → `onDeleteReview` 단일 경로로만 처리된다.

## 13. 방어적 패턴 모음

### 13.1 `that = this` — 콜백 내 `this` 보존

```javascript
var that = this;
// 콜백 안에서:
if (that._bDestroyed) return;
```

JavaScript 콜백 함수 내부의 `this`는 호출 방식에 따라 달라진다. `that`에 `this`를 미리 저장해두면 언제나 DetailMain 인스턴스를 안전하게 참조할 수 있다. `.bind(this)`와 같은 목적이지만, `_bDestroyed` 확인처럼 콜백 체인 전체에서 일관되게 써야 할 때는 `that`이 더 편리하다.

### 13.2 `_bDestroyed` — 파괴 감지

```javascript
// onExit
this._bDestroyed = true;

// 모든 콜백의 첫 줄
if (that._bDestroyed) return;
```

사용자가 페이지를 떠난 후에도 진행 중이던 AJAX 응답이나 타이머 콜백이 실행되면, 이미 파괴된 뷰/컨트롤러를 조작하려다 오류가 발생한다. 모든 콜백 시작 부분에서 `_bDestroyed`를 확인하여 안전하게 중단한다.

### 13.3 `_bSubmitting` — 중복 제출 방지

```javascript
if (this._bSubmitting) return;
this._bSubmitting = true;
// ... API 호출 ...
that._bSubmitting = false;
```

버튼 연타를 방지하는 간단한 락(lock). `onSubmitReview`에서만 사용된다.

### 13.4 `_loadToken` — Race Condition 방지

```javascript
this._loadToken = (this._loadToken || 0) + 1;
var token = this._loadToken;
// 응답 도착 시:
if (token !== that._loadToken) return;
```

연속된 빠른 페이지 이동에서 오래된 응답이 최신 데이터를 덮어쓰는 것을 방지한다.

### 13.5 `_checkRetryCount` — 비동기 의존성 대기

```javascript
this._checkRetryCount = (this._checkRetryCount || 0) + 1;
if (this._checkRetryCount <= 5) {
    this._checkTimer = setTimeout(this._checkMyReview.bind(this), 200);
}
```

`UserSessionLoader`처럼 준비 시간이 불확실한 비동기 의존성을 최대 5회(1초)까지 기다린다. UserSessionLoader는 `ready` 이벤트 기반으로 동작하며, 모델 준비가 완료되면 `sap.ui.getCore().setModel(oModel, "User")`로 코어에 등록된다 (자세한 내용은 [[#6.1a User 모델은 어디서 오는가 — `UserSessionLoader` 깊이 이해하기]] 참조).

### 13.6 `_checkTimer` — 타이머 정리

```javascript
// onExit
if (this._checkTimer) {
    clearTimeout(this._checkTimer);
    this._checkTimer = null;
}
```

`onExit`에서 진행 중인 타이머를 정리하지 않으면 파괴된 컨트롤러에서 콜백이 실행되어 오류가 발생한다.

## 14. 전체 생명주기 & 데이터 흐름

```
① URL 변경: #/Detail/C001
│
② Router → _onRouteMatched(oEvent)
│   ├─ contentId = "C001"
│   ├─ /NewReview 초기화
│   └─ _loadDetail("C001")
│
③ _loadDetail
│   ├─ loadToken 발급 (race condition 방지)
│   ├─ setBusy(true) — 로딩 인디케이터
│   └─ readOdata( "/Contents('C001')?$expand=... ")
│
④ Promise 응답 도착
│   ├─ token 검사 → 통과
│   ├─ viewModel.setProperty("/Detail", oData)  ← 백엔드 데이터 저장
│   └─ _checkMyReview()
│       ├─ User 모델 확인 → 없으면 재시도 (최대 5회)
│       ├─ 전체 리뷰 중 user_user_id === myUserId 찾기
│       ├─ viewModel.setProperty("/MyReview", ...)
│       └─ viewModel.setProperty("/HasReviewed", ...)
│
⑤ UI 자동 갱신 (데이터 바인딩)
│   ├─ 헤더: title, avg_rating, rating_count
│   ├─ 기본 정보: content_id, partner, 장르, 태그
│   ├─ 출연진: Table (이름·역할·캐릭터명)
│   └─ 리뷰: List + MessageStrip (HasReviewed 여부)
│
⑥ setBusy(false) — 로딩 완료
│
⑦ 사용자 인터랙션 대기
    ├─ ✏️ 수정 → onEditReview → 폼 prefill
    ├─ ⭐ 등록 → onSubmitReview → API 호출 → _loadDetail 재조회
    ├─ 🗑️ 삭제 → onDeleteReview → MessageBox → API 호출 → 재조회
    ├─ 📊 엑셀 → onExportInfo / onExportCast → Spreadsheet.build()
    └─ ← 뒤로 → onNavBack → history.go(-1)
```

## 15. 핵심 암기 사항

| # | 내용 |
|---|------|
| 1 | **3단계 상속**: DetailMain → detailPage.BaseController(껍데기) → library_ott.BaseController(진짜) |
| 2 | **`readOdata`는 Promise 지원**: `_loadDetail`에서 `.then()`으로 편하게 사용. library_ott 래퍼 덕분 |
| 3 | **`callFunction`은 Promise 미지원**: 반드시 `success`/`error` 콜백 직접 전달. V2 OData 제약 |
| 4 | **Token 패턴**: 연속 페이지 이동 시 오래된 응답이 최신 데이터를 덮어쓰는 것 방지 |
| 5 | **서버 재조회 > 수동 조작**: `_loadDetail()` 한 줄이 수동 배열 조작보다 정확하다 |
| 6 | **수정 = 삭제 + 재등록**: CAP에 수정 API가 없어서 2단계로 구현. 콜백 중첩으로 순서 보장 |
| 7 | **`_bDestroyed` + `that = this`**: 모든 비동기 콜백에서 페이지 이탈 후 안전하게 중단 |
| 8 | **`_bSubmitting`**: 버튼 연타 방지 락. submit 시작 시 true, 완료 시 false |
| 9 | **User 모델 비동기 재시도**: `setTimeout`으로 최대 5회(1초). `_checkRetryCount`로 제한 |
| 10 | **`_checkTimer` 정리**: `onExit`에서 `clearTimeout` — 안 하면 파괴된 컨트롤러에서 콜백 실행 |
| 11 | **`getBindingContext`**: 목록 항목의 이벤트에서 해당 행의 데이터를 추출하는 표준 패턴 |
| 12 | **`setBusy(true/false)`**: 데이터 로딩 중 UI 차단. try/finally 대신 then/catch 양쪽에서 false |
| 13 | **i18n 메시지 키**: `message.reviewSuccess`, `message.reviewUpdateSuccess` 등 수정/신규 구분 |
| 14 | **Spreadsheet**: `build().finally(destroy)` — 메모리 누수 방지 |
