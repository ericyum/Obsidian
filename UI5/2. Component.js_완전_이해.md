# Component.js 완전 이해

> **대상**: `detailPage/webapp/Component.js`
> **역할**: 앱의 진짜 **시동 키**. manifest.json이 "설계도"라면, Component.js는 그 설계도를 읽고 엔진을 거는 시동 장치다. 단 11줄이지만 앱 생명주기의 모든 분기점을 제어한다.

---

## 전체 코드

```javascript
sap.ui.define([
    "sap/ui/core/UIComponent"
], function(UIComponent) {
    "use strict";
    return UIComponent.extend("detailPage.Component", {
        metadata: {
            manifest: "json"
        },
        init: function() {
            jQuery.sap.registerModulePath("common_ott.lib", "/common_ott.lib/");
            UIComponent.prototype.init.apply(this, arguments);
            this.getRouter().initialize();
        }
    });
});
```

11줄. 하지만 이 11줄이 하는 일은 방대하다.

---

## 1줄씩 해부

### ① `sap.ui.define` — 모듈을 정의한다

```javascript
sap.ui.define([의존성배열], 팩토리함수)
```

SAP UI5의 모듈 시스템은 **AMD(Asynchronous Module Definition)** 기반이다. 브라우저가 이 파일을 만나면:

```
1. 의존성 배열 확인 → ["sap/ui/core/UIComponent"] 필요
2. UIComponent 모듈을 먼저 로드 (없으면 서버에서 fetch)
3. 로드 완료되면 팩토리 함수 실행
4. 팩토리 함수의 반환값(새 클래스)을 "detailPage.Component"라는 이름으로 등록
```

| 구성 요소 | 설명 |
|-----------|------|
| `["sap/ui/core/UIComponent"]` | 이 모듈이 의존하는 유일한 모듈. 모든 UI 컴포넌트의 부모 클래스 |
| `function(UIComponent)` | 의존성이 주입되는 팩토리 함수. 배열 순서대로 인자가 매핑됨 — 즉 첫 번째 의존성 `"sap/ui/core/UIComponent"`가 `UIComponent`라는 인자로 들어온다 |
| `return UIComponent.extend(...)` | 새 클래스를 만들어 반환 = 이 파일의 "수출품" |

> 💡 다른 모듈이 `sap.ui.define(["detailPage/Component"], ...)` 하면 이 팩토리 함수의 반환값을 받게 된다.

---

### ② `UIComponent.extend("detailPage.Component", { ... })` — 클래스를 상속한다

```javascript
UIComponent.extend("detailPage.Component", {
    metadata: { ... },
    init: function() { ... }
})
```

| 요소 | 값 | 의미 |
|------|-----|------|
| 부모 클래스 | `sap.ui.core.UIComponent` | 모든 UI5 앱의 루트 클래스. manifest 파싱, 라우터 생성, 모델 관리 등 기본 기능 제공 |
| 새 클래스명 | `"detailPage.Component"` | **manifest.json의 `sap.app.id: "detailPage"`와 반드시 일치해야 함.** |
| 두 번째 인자 | `{ metadata, init }` | 부모 클래스에 **추가/재정의**할 멤버들. 여기서 `metadata`는 새로 추가, `init`은 부모 것을 오버라이딩 |

#### 네임스페이스 일치 규칙

```
manifest.json  →  "sap.app.id": "detailPage"
Component.js   →  UIComponent.extend("detailPage.Component")
파일 경로      →  detailPage/webapp/Component.js
                            └────────────┘
                         sap.app.id = 이 폴더가 루트
```

셋이 일치해야 UI5가 Component.js를 찾을 수 있다. Java에서 패키지명과 폴더 구조가 일치해야 하는 것과 같은 원리.

---

### ③ `metadata: { manifest: "json" }` — 설계도를 연결한다

```javascript
metadata: {
    manifest: "json"
}
```

**이 한 줄이 하는 일:**

```
"manifest": "json"
       │
       ▼
UI5가 Component.js와 같은 폴더에서 manifest.json을 찾는다
       │
       ▼ (실제 파싱은 부모 init()에서 일어남 — 지금은 "어디서 찾을지만" 등록)
manifest.json을 파싱해서 메모리에 올린다
       │
       ├─ sap.app.id → 네임스페이스 등록
       ├─ dataSources → 백엔드 연결 정보 등록
       ├─ dependencies.libs → 필요한 라이브러리 로드 시작
       ├─ models → 모델 생성 (preload: true면 이때 HTTP 요청)
       ├─ rootView → 루트 뷰 정보 저장
       └─ routing → 라우트 설정 저장
```

> 💡 `metadata`는 UI5 클래스 시스템의 핵심 장치다. 단순 주석이 아니라 **런타임에 실제로 읽히고 실행되는 선언적 설정**이다. `manifest: "json"`이라고 적으면 UI5가 "이 컴포넌트는 manifest.json을 쓴다"고 인지하고, 부모 `init()`이 호출될 때 해당 파일을 파싱한다.

---

### ④ `init()` — 시동을 건다

`init()`은 `new detailPage.Component()`가 호출될 때 **단 한 번** 실행되는 생성자다. 세 가지 일을 **반드시 이 순서대로** 한다. 순서가 바뀌면 앱이 깨진다.

---

#### 4-a. `registerModulePath` — 외부 라이브러리 경로 등록

```javascript
jQuery.sap.registerModulePath("common_ott.lib", "/common_ott.lib/");
```

**UI5 모듈 로더의 URL 변환 테이블에 항목 하나를 추가한다.**

| 인자 | 값 | 의미 |
|------|-----|------|
| 첫 번째: 모듈 네임스페이스 | `"common_ott.lib"` | 코드에서 `sap.ui.define(["common_ott/lib/..."])` 할 때 쓰는 이름 |
| 두 번째: URL 경로 | `"/common_ott.lib/"` | 이 네임스페이스를 이 URL 프리픽스로 변환하라 |

**구체적인 변환 예시:**

`BaseController.js`에서 이런 코드가 있다:

```javascript
// detailPage/webapp/controller/BaseController.js
sap.ui.define([
    "common_ott/lib/controller/BaseController"  // ← 이 모듈명을...
], function(Controller) { ... });
```

이때 UI5 모듈 로더의 변환 과정:

```
① 코드 안의 모듈명: "common_ott/lib/controller/BaseController"
         │
         ▼ registerModulePath 변환 규칙 적용
② 네임스페이스 "common_ott.lib" → URL 프리픽스 "/common_ott.lib/" 로 교체
         │
         ▼ 나머지 경로를 그대로 붙이고 .js 확장자 추가
③ 최종 URL: "/common_ott.lib/controller/BaseController.js"
         │
         ▼ 브라우저가 이 URL로 HTTP GET 요청
         │  (approuter가 이 경로를 받아 library_ott/src/ 의 실제 파일로 연결)
         │
④ 실제 파일: cap-app/library_ott/src/controller/BaseController.js
```

**왜 필요한가?** `common_ott.lib`는 `detailPage/webapp/` **안에 없다**. UI5 기본 규칙대로라면 `webapp/common_ott/lib/...`에서 찾으려 하지만, 실제 파일은 완전히 다른 폴더인 `library_ott/src/`에 있다. 이 괴리를 `registerModulePath`가 메꿔준다.

```
❌ UI5 기본 추측: detailPage/webapp/common_ott/lib/controller/BaseController.js (없음!)
✅ registerModulePath 후: /common_ott.lib/controller/BaseController.js → 실제 파일 찾기 성공
```

> 💡 `common_ott.lib`는 detailPage뿐 아니라 myPage, portal_ott 등 **여러 _ott 앱이 공유**하는 라이브러리다. manifest.json의 `resourceRoots`로도 동일한 설정이 가능하지만, 공통 라이브러리이므로 Component.js의 init()에 코드로 중앙 등록하는 방식이 관리상 더 깔끔하다.

---

#### 4-b. `UIComponent.prototype.init.apply(this, arguments)` — 부모 초기화 (⚡ 여기서 모든 게 일어난다)

```javascript
UIComponent.prototype.init.apply(this, arguments);
```

이 한 줄은 **부모 클래스인 `sap.ui.core.UIComponent`의 `init()`을 호출**하는 JavaScript 패턴이다 (= `super.init()`). SAP UI5는 ES6 클래스가 아닌 자체 `extend()` 시스템을 사용하므로 `super` 키워드를 쓸 수 없다.

##### 부모 init() 내부에서 일어나는 일 (순서대로)

```
UIComponent.prototype.init()
│
├─ ① manifest.json 파싱 & 메모리 로드
│    ├─ metadata.manifest: "json" → 같은 폴더의 manifest.json을 읽는다
│    ├─ sap.app.id → "detailPage" 네임스페이스 등록
│    ├─ dependencies.libs → sap.m, sap.ui.core, sap.f, sap.ui.layout, sap.uxap 로드
│    └─ dataSources → "detailService" 연결 정보 저장 (⚠️ 아직 통신 안 함)
│
├─ ② 모델 생성 (manifest의 models 섹션 처리)
│    ├─ i18n → ResourceModel, bundleName 기반으로 i18n.properties 파일을 HTTP로 가져옴
│    ├─ detail → ODataModel V2, dataSource="detailService"에 연결
│    │   └─ ⚡ preload: true → $metadata HTTP 요청 발생!
│    └─ viewModel → 빈 JSONModel {} (클라이언트 상태용)
│
├─ ③ rootView 생성
│    ├─ viewName: "detailPage.view.App" → webapp/view/App.view.xml을 HTTP로 가져옴
│    └─ id: "App" → <mvc:View id="App"> 생성
│
└─ ④ 리소스 로드
     └─ css: "css/style.css" 로드
```

##### preload 시점 상세

```
dataSources만 선언:           통신 0회 (주소만 기억)
model + dataSource 연결:      통신 0회 (연결만)
model + preload: true:        ⚡ 이때 $metadata 요청! (OData 모델 구조 파악)
뷰 바인딩 평가:               실제 데이터 요청 (Contents('C001') 등)
```

> ⚠️ **중요**: ③에서 App.view.xml이 렌더링되지만, 그 안의 DetailMain은 **아직 로드되지 않는다.** App.view.xml은 `<App id="app">`라는 빈 셸만 만든다. 실제 컨텐츠(DetailMain)는 다음 단계인 라우터가 URL 해시를 보고 결정한다.

##### 왜 `UIComponent.prototype.init.apply(this, arguments)` 인가?

SAP UI5는 ES6 클래스가 아닌 고유의 `extend()` 시스템을 사용하므로 `super.init()`을 쓸 수 없다. 대신 부모의 prototype에 직접 접근해서 `apply`로 호출한다:

```javascript
// ❌ ES6 스타일 (UI5에서는 동작 안 함)
super.init();

// ✅ UI5 스타일
UIComponent.prototype.init.apply(this, arguments);
```

`apply(this, arguments)`는 "부모의 init을 현재 인스턴스(`this`)의 컨텍스트에서, 전달받은 인자 그대로 실행하라"는 뜻이다.

#### 4-c. `this.getRouter().initialize()` — 라우팅 활성화

```javascript
this.getRouter().initialize();
```

라우터를 가동한다. **이 한 줄은 두 가지 일을 동시에 한다:**

```
this.getRouter().initialize()
│
├─ ① 즉시 실행: 지금 당장 URL 해시(#)를 읽고 타겟 뷰를 로드한다
│     현재 URL의 해시가 #/Detail/C001 이면 바로 DetailMain을 띄운다
│
└─ ② 상시 대기 등록: window.onhashchange 이벤트 리스너를 등록한다
      이후 사용자가 다른 해시로 이동할 때마다 자동으로 반응한다
```

**비유**: TV를 켜면서 "지금 채널 확인" + "리모컨 수신 켜기"를 동시에 하는 것과 같다.

##### ① 즉시 실행: 현재 해시 처리

```
URL 예시: .../index.html#/Detail/C001
                      └─────────────┘
                      window.location.hash

getRouter().initialize()
│
├─ 현재 해시 읽기 → "#/Detail/C001"
│
├─ manifest의 routes에서 패턴 매칭
│   pattern: "Detail/{content_id}" → 매칭 성공!
│   content_id = "C001" 추출
│
├─ target "DetailMain" 결정
│   viewPath: "detailPage.view" + viewName: "DetailMain"
│   → 모듈명: "detailPage.view.DetailMain"
│   → 점(.)을 슬래시(/)로, 뒤에 .view.xml 붙임
│   → 경로: webapp/view/DetailMain.view.xml
│   → ⚡ 브라우저가 내부적으로 HTTP GET 요청을 보내 파일을 가져온다
│
├─ DetailMain.controller.js 인스턴스 생성
│   └─ onInit() 실행 → 이벤트 핸들러 등록, UserSessionLoader 등 비동기 시작
│
└─ DetailMain 뷰를 App의 pages aggregation에 삽입
    └─ slide 애니메이션과 함께 화면에 표시
```

##### ② 상시 대기 등록: 앞으로의 해시 변경 감지

`initialize()`가 `window.onhashchange` 이벤트에 리스너를 등록한다. 이후 사용자가 링크를 클릭하거나 브라우저 뒤로가기를 누르면:

```
사용자가 다른 링크 클릭 → 해시 변경 (#/Detail/C002)
    │
    ▼
hashchange 이벤트 발생 → 라우터가 자동 감지
    │
    ▼
새로운 해시로 라우트 매칭 → 필요한 경우 새 뷰 로드
(단, 같은 라우트에 content_id만 바뀌면 뷰/컨트롤러는 재사용, _onRouteMatched()만 재호출)
```

##### 해시가 없는 경우 → `bypassed` 발동

```
URL: .../index.html  (해시 없음)
      │
      ▼ 매칭할 route가 없음 → bypassed
      │
      ▼ bypassed.target: "DetailMain"
      │
      ▼ DetailMain을 기본으로 표시
```

##### 중요: 해시(#)는 서버가 절대 볼 수 없다

HTTP 프로토콜의 규칙상, 브라우저는 URL의 **해시(#) 부분을 서버에 전송하지 않는다.**

```
사용자가 입력한 전체 URL:
http://localhost:5000/sysmgt_ott/detailPage/webapp/index.html#/Detail/C001

브라우저가 서버에 실제로 보내는 HTTP 요청:
GET /sysmgt_ott/detailPage/webapp/index.html    ← 여기까지. # 이후는 제외됨

서버(approuter)가 아는 것:
/sysmgt_ott/detailPage/webapp/index.html

서버(approuter)가 모르는 것:
#/Detail/C001   ← 이건 클라이언트(브라우저)만 알고 있다
```

따라서 전체 흐름은 **두 단계의 분업**이다:

```
[1단계: 서버 사이드]
  GET index.html → approuter가 index.html 반환 → 역할 끝
        │
[2단계: 클라이언트 사이드]
  index.html 로드 → UI5 부트스트랩 → Component.init()
    → 부모 init() → manifest 파싱, App.view.xml 셸 생성
    → getRouter().initialize()
        → window.location.hash 읽기 → "#/Detail/C001" 발견
        → 라우트 매칭 → DetailMain.view.xml 내부 HTTP 요청
        → 받은 XML을 App의 pages에 삽입 → 화면 완성
```

## 🧬 상속 체인 — Component.js가 연결하는 전체 계층

### 컴포넌트 계층

```
sap.ui.core.UIComponent
    └─ detailPage.Component  ← 우리가 분석 중인 파일
```

### 컨트롤러 계층 (Component.js가 간접적으로 연결)

```
sap.ui.core.mvc.Controller
    └─ common_ott.lib.controller.BaseController  ← library_ott (진짜 로직)
        │   ├─ onInit() → UserSessionLoader, CommonCodeLoader, etc.
        │   ├─ getResourceBundle() → this.getOwnerComponent().getModel("i18n")...
        │   ├─ readOdata() → Promise 기반 OData read 래퍼
        │   ├─ fnDeepCopy() → 깊은 복사 유틸
        │   └─ fnMasterCodeValue() → 공통 코드 매핑
        │
        └─ detailPage.controller.BaseController  ← detailPage (빈 껍데기)
            │   └─ 아무것도 재정의하지 않음. 순수 상속만.
            │
            └─ detailPage.controller.DetailMain  ← 실제 컨트롤러
                ├─ onInit() → _onRouteMatched 구독 등
                ├─ _onRouteMatched() → 데이터 로드 + UI 초기화
                └─ onSubmitReview(), onDeleteReview(), etc.
```

> 💡 `detailPage.controller.BaseController`는 **빈 껍데기**다. 템플릿(원본 codeManagement)도 동일하게 빈 BaseController를 두는 구조라서 그대로 따른 것. 실제 로직은 전부 `common_ott.lib.controller.BaseController`에 있다. 이 BaseController를 가져오기 위해 `registerModulePath`가 반드시 필요하다.

## ❓ 자주 묻는 질문

### Q1: Component.js에 `new`는 누가 호출하나?

UI5 코어가 `index.html`의 `data-sap-ui-component` 설정을 보고 자동으로 인스턴스를 생성한다. 개발자가 직접 `new`할 필요는 없다.

---

### Q2: `getRouter().initialize()` — 요청이 들어올 때 발동하는 건가, 상시 대기 모드를 켜는 건가?

**둘 다다.** `initialize()`는:

1. **지금 당장** 현재 URL의 해시를 읽어서 라우트 매칭 → 타겟 뷰를 즉시 로드한다.
2. **동시에** `window.onhashchange` 리스너를 등록해서, 앞으로 해시가 변경될 때마다 자동으로 반응하는 상시 대기 상태를 켠다.

한 번만 호출하면 된다. 이후로는 리스너가 계속 살아서 해시 변경을 감지한다.

---

### Q3: `DetailMain.view.xml`은 어떻게 가져오는가? 내부적으로 HTTP 요청을 보내는가?

**그렇다.** `getRouter().initialize()`가 DetailMain 타겟을 결정한 뒤, UI5 모듈 로더가 내부적으로 `XMLHttpRequest`(또는 `fetch`)를 사용해 HTTP GET 요청을 보낸다:

```
모듈명: "detailPage.view.DetailMain"
    → 점→슬래시 변환, .view.xml 확장자 추가
    → 최종 URL: .../webapp/view/DetailMain.view.xml
    → 브라우저가 내부적으로 GET 요청
    → 응답으로 받은 XML 문자열을 UI5 XML 파서가 DOM으로 변환
    → App 컨트롤의 pages aggregation에 삽입
```

이 요청 역시 `index.html` 정적 파일 요청과 동일한 경로로 전송되며, approuter가 실제 파일 위치로 연결해준다.

---

### Q4: 사용자가 `#/Detail/C001`을 포함한 전체 URL로 접속했을 때, 해시를 먼저 처리하는 건 서버인가 클라이언트인가?

**클라이언트다.** HTTP 프로토콜상 브라우저는 URL의 `#` 이후 부분을 **서버에 절대 전송하지 않는다.** 따라서:

1. **서버(approuter)**는 `index.html`까지만 받아서 해당 파일을 반환한다. `#/Detail/C001`은 알지도 못한다.
2. **클라이언트(브라우저)**에서 `index.html` 로드 후 Component.init() → `getRouter().initialize()`가 실행되면, 그때 `window.location.hash`를 읽어 `#/Detail/C001`을 인지하고 라우팅을 시작한다.

해시는 100% 클라이언트 사이드 장치다.

---

### Q5: `UIComponent.prototype.init.apply(this, arguments)` 대신 `super.init()`은 왜 안 쓰나?

SAP UI5는 ES6 클래스가 아니라 고유의 `extend()` 시스템을 사용한다. 따라서 JavaScript의 `super` 키워드를 쓸 수 없다. 대신 부모의 prototype에 직접 접근해서 `apply`로 호출하는 게 정석 패턴이다.

---

### Q6: manifest.json에 `resourceRoots`를 안 쓰고 왜 `registerModulePath`를 코드로 하나?

`common_ott.lib`는 detailPage만 쓰는 게 아니라 myPage, portal_ott 등 **여러 _ott 앱이 공유**하는 라이브러리다. manifest.json에 `resourceRoots`를 설정하는 것도 가능하지만, 공통 라이브러리 경로는 manifest마다 분산 선언하는 것보다 Component.js init()에서 한 번에 코드로 등록하는 게 중앙 관리에 유리하다.

---

### Q7: `this.getRouter()`는 어디서 오는가?

`UIComponent.prototype.init()` 내부에서 manifest의 `routing.config`를 읽고 라우터 인스턴스를 생성해서 컴포넌트에 붙인다. 그래서 `init()` 안에서도 부모 init() **호출 이후**에만 `this.getRouter()`가 유효하다. 호출 순서가 중요한 이유다.

---

### Q8: Component.js가 이렇게 짧은데도 앱이 돌아가는 이유는?

UI5의 철학은 **"선언적 설정"** 이다. 복잡한 동작은 manifest.json에 **데이터로 선언**하고, Component.js는 단지 그 선언을 **실행하는 트리거** 역할만 한다. 11줄이면 충분한 이유다.

## 🔗 Component.js와 manifest.json의 연결 관계

```
manifest.json                         Component.js
─────────────                         ─────────────
sap.app.id: "detailPage"    ──▶      extend("detailPage.Component")  (네임스페이스 일치)
dataSources: {...}          ──▶      metadata: { manifest: "json" }  (설계도 연결)
models: [{...}]             ──▶      UIComponent.prototype.init()    (모델 생성)
rootView: {...}             ──▶      ↑                              (루트 뷰 생성)
routing: {...}              ──▶      this.getRouter().initialize()   (라우터 가동)
```
