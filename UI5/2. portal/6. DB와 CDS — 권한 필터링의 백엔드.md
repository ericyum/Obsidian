# 사용자 권한 기반 동적 메뉴 시스템 완전 분석

> **작성일**: 2026-07-10 · **업데이트**: 2026-07-13 (Q&A 기반 심화 보강 — v3.0)
> **목적**: portal_ott에서 manifest.json의 정적 targets가 아닌, DB 기반으로 사용자 권한에 따라 메뉴가 동적으로 생성되는 전체 메커니즘을 이해한다.

---

## 1. 문제 인식

`manifest.json`에는 `sap.ui5.routing.targets`에 정적인 target들이 정의되어 있다. 그러나 portal_ott의 메뉴는 **사용자마다 다르게** 보인다:

- `U001` (OTT_USER_GROUP) → 5개 메뉴 (OTT_MAIN, DETAIL, MYPAGE, BOARD 등)
- `admin1` (OTT_USER_GROUP + OTT_ADMIN_GROUP) → 7개 메뉴 (위 + SETTLEMENT, TREND)

이 차이는 manifest.json만으로 설명되지 않는다. **어딘가에 사용자 권한에 따라 메뉴를 동적으로 구성하는 코드**가 있다.

---

## 2. 정답: 모든 건 CDS View `@restrict` 한 줄에 있다

```cds
@(restrict: [
    { grant: 'READ', where: 'user_id = $user' }
])
view MenusRoleAppliedList as select distinct ...
```

**이 한 줄이 전부다.** JavaScript 코드 어디에도 `if (role === 'Admin')` 같은 분기문은 없다.

---

## 3. CDS View의 두 가지 선언 방식

`MenuManagement-service.cds`에는 두 종류의 엔티티 선언이 공존한다. 이 차이를 이해하는 것이 핵심이다.

### 3.1 방식 1: `entity ~ as projection on` — 단일 테이블 창문

```cds
entity MenuSet as projection on core_menu {
    key menu_code,
    menu_name,
    menu_app_id
};
```

```
    core_menu 테이블
┌──────────────────────────┐
│ menu_code │ menu_name... │
│ menu_app_id             │
│ menu_route_path         │  ← 이건 빼고
│ menu_repo_path          │  ← 이것도 빼고
│ ...                     │
└──────────────────────────┘
           │
           │  projection on  (일부 컬럼만 통과)
           ▼
┌──────────────────────────┐
│ menu_code │ menu_name    │
│ menu_app_id             │  ← 3개 컬럼만 OData로 노출
└──────────────────────────┘
```

**특징**:
- **단일 테이블**에서 컬럼 일부만 골라내는 필터
- `projection on` 뒤에는 **오직 하나의 엔티티 이름**만 올 수 있음
- 원본 테이블에 CUD(생성·수정·삭제) 전파 가능

### 3.2 방식 2: `view ~ as select ... from ... join` — 다중 테이블 특수 창문

```cds
view MenusRoleAppliedList as 
    select distinct
        menu.menu_code,
        menu.menu_name,
        core_user.user_id
    from core_menu as menu
    inner join core_role_menu as role_menu on ...
    inner join core_user as core_user on ...
```

```
 core_menu          core_role_menu       core_user
┌──────────┐      ┌──────────────┐     ┌──────────┐
│OTT_MAIN  │──────│OTT_USER,MAIN │     │  U001    │
│OTT_DETAIL│     │OTT_USER,DETAIL│─────│ admin1   │
│SETTLEMENT│     │Admin,SETTLE   │     └──────────┘
└──────────┘      └──────────────┘
       │                 │                 │
       └─────────── JOIN ─────────────────┘
                       │
                       ▼
          ┌──────────────────────────┐
          │ menu_code │ user_id      │
          │ OTT_MAIN  │ admin1       │
          │ OTT_DETAIL│ admin1       │
          │ SETTLEMENT│ admin1       │  ← 여러 테이블을 합친 결과
          └──────────────────────────┘
```

**특징**:
- **여러 테이블을 JOIN**해서 새로운 결과 집합을 만듦
- DISTINCT, GROUP BY, 계산된 필드(CASE WHEN) 등 복잡한 SQL 사용 가능
- **읽기 전용**(READ ONLY) — 여러 테이블을 합친 결과라 CAP이 수정 대상을 특정할 수 없음

### 3.3 왜 `entity as projection on`으로는 안 되는가?

메뉴 필터링에 필요한 것은:
1. User → RoleGroup → Role → Menu 체인을 JOIN
2. `$user`로 현재 사용자의 메뉴만 필터링

`projection on`은 **단일 테이블만** 바라볼 수 있으므로, 6개 테이블을 연결하는 JOIN이 **원천적으로 불가능**하다.

```cds
// ❌ 문법 오류! projection on 뒤에는 하나의 엔티티만 올 수 있음
entity MenuSet as projection on core_menu, core_role_menu { ... }

// ✅ 유일한 방법: view as select ... join
view MenusRoleAppliedList as select ... from core_menu inner join core_role_menu ...
```

### 3.4 `using`의 역할 — 단순한 import

```cds
using { com.cap.ott.core.User as core_user } from '../../db/cds/User-model';
using { com.cap.ott.core.Menu as core_menu } from '../../db/cds/Menu-model';
using { com.cap.ott.core.Role_Menu as core_role_menu } from '../../db/cds/Role_Menu-model';
```

`using`은 JavaScript의 `import`와 완전히 같다: "이 파일에서 `core_user`, `core_menu`라는 이름으로 해당 엔티티를 참조할 수 있게 해줘."

**하지만 `using`으로 아무리 많은 엔티티를 import해도, `projection on`은 단일 테이블만 선택할 수 있다.** 여러 테이블을 엮으려면 `view as select ... join`이 유일한 방법이다.

| | `entity as projection on` | `view as select ... join` |
|---|---|---|
| 단일 테이블에서 컬럼 선택 | ✅ | ✅ |
| 여러 테이블 JOIN | ❌ | ✅ |
| DISTINCT, GROUP BY | ❌ | ✅ |
| `@restrict` 적용 | ✅ | ✅ |
| CUD(생성·수정·삭제) | 가능 (원본 테이블에 전파) | ❌ **읽기 전용** |
| 계산된 필드(CASE WHEN) | ❌ (제한적) | ✅ |

## 4. 전체 메커니즘: 6단계

### 4.1 DB: 관계형 권한-메뉴 구조

#### 왜 7개 테이블이 필요한가?

본질은 **3개의 다대다 관계**다. 각 다대다 관계를 풀려면 중간 테이블이 하나씩 필요하다.

```
User ──다대다── RoleGroup ──다대다── Role ──다대다── Menu
```

| 관계 | 중간 테이블 | 의미 |
|------|-----------|------|
| User ↔ RoleGroup | **User_RoleGroup** | "U001은 OTT_USER_GROUP 소속" |
| RoleGroup ↔ Role | **Role_RoleGroup** | "OTT_USER_GROUP은 OTT_USER 역할을 가짐" |
| Role ↔ Menu | **Role_Menu** | "OTT_USER 역할은 OTT_MAIN 메뉴를 볼 수 있음" |

**핵심**: 이렇게 RoleGroup, Role이라는 **추상화 계층**을 두면, "사용자별 메뉴 직접 할당"이 아니라 "역할 기반"으로 유연하게 관리할 수 있다. 새 메뉴가 생기면 Role_Menu에 한 줄 추가하는 것으로 모든 해당 역할 사용자에게 자동 적용된다.

### 4.2 CDS View: JOIN의 역할 구분

```cds
view MenusRoleAppliedList as 
    select distinct
        key menu.menu_code,           -- ← Menu 테이블에서 직접 가져옴 (데이터)
        menu.menu_app_id,             -- ← Menu 테이블에서 직접 가져옴 (데이터)
        menu.menu_app_path,           -- ← Menu 테이블에서 직접 가져옴 (데이터)
        menu.menu_route_path,         -- ← Menu 테이블에서 직접 가져옴 (데이터)
        menu.menu_name,
        menu.menu_repo_path,
        menu.menu_icon,
        menu.menu_type_code,
        menu.menu_level_number,
        core_user.user_id             -- ← User 테이블에서 가져옴 (@restrict 참조용)
    from core_menu as menu            -- ← 기준 테이블 (데이터 원천)
    inner join core_role_menu as role_menu        -- ← 필터 역할만
        on menu.menu_code = role_menu.menu_code
    inner join core_role as core_role             -- ← 필터 역할만
        on core_role.role_code = role_menu.role_code
    inner join core_role_rolegroup as role_role_group  -- ← 필터 역할만
        on role_role_group.role_code = core_role.role_code
    inner join core_rolegroup as role_group            -- ← 필터 역할만
        on role_group.role_group_code = role_role_group.role_group_code
    inner join core_user_rolegroup as user_role_group  -- ← 필터 역할만
        on user_role_group.role_group_code = role_role_group.role_group_code
    inner join core_user as core_user                  -- ← 필터 + @restrict 대상
        on core_user.user_id = user_role_group.user_id
    where menu.menu_use_flag = true
      and role_menu.use_flag = true
```

**JOIN의 두 가지 역할**:

| 테이블 | 역할 | 비유 |
|--------|------|------|
| `core_menu` (menu) | **데이터 원천** — 실제 반환할 메뉴 정보 | "책의 내용" |
| `core_role_menu` ~ `core_user` | **필터** — 어떤 메뉴를 누구에게 보여줄지 결정 | "이 책은 누구에게 보여줄까?" |

`Role_Menu` 테이블의 데이터(어떤 역할이 어떤 메뉴를 보는지)는 SELECT 결과에 **직접 나타나지 않는다.** JOIN의 ON 조건으로만 사용되고, 최종 SELECT는 Menu 테이블의 컬럼들만 뽑아낸다.

### 4.3 `@restrict`: SQL WHERE 절 주입 (사후 필터링이 아니다!)

```cds
@(restrict: [
    { grant: 'READ', where: 'user_id = $user' }
])
view MenusRoleAppliedList as ...
```

#### ❌ 오해: 사후 필터링이 아니다

```
-- 이런 식으로 동작하지 않는다:
① SQL 실행: SELECT ... FROM ... JOIN ... (필터 없음, 100행)
② CAP이 결과를 받아서: "user_id 컬럼 값이 $user랑 같으면 통과" (6행 추림)
```

#### ✅ 실제 동작: SQL 인젝션

```
① HTTP 요청 도착: GET /MenusRoleAppliedList
② CAP: "@restrict 있네. $user = 'admin1'"
③ CAP: 기존 SQL 뒤에 "AND core_user.user_id = 'admin1'"을 추가 (WHERE 절 주입)
④ SQL 실행: 필터링된 6행만 DB에서 반환
```

**실제 실행되는 SQL**:
```sql
SELECT DISTINCT
    menu.menu_code,
    menu.menu_name,
    ...
    core_user.user_id
FROM core_menu AS menu
INNER JOIN core_role_menu AS role_menu ON ...
INNER JOIN core_user AS core_user ON ...
WHERE 1=1
  AND menu.menu_use_flag = true
  AND role_menu.use_flag = true
  AND core_user.user_id = ?    -- ← @restrict가 추가한 WHERE 절!
--                          ↑
--                    파라미터 바인딩: 'admin1'
```

필터링은 **SQL 실행 전에** WHERE 절로 완료된다. DB가 필터링된 결과만 반환하므로, 네트워크 전송량과 메모리 사용량 모두 최소화된다.

### 4.4 왜 `core_user.user_id`를 SELECT에 넣어야 하는가?

일반 SQL에서는 WHERE 절이 테이블의 모든 컬럼을 볼 수 있다:

```sql
-- 일반 SQL: SELECT에 없어도 WHERE에서 user_id 사용 가능
SELECT name, age FROM users WHERE user_id = 5;  -- ✅ OK
```

하지만 CDS View는 **"SELECT 결과로 만들어지는 가상의 엔티티"** 다. 이 엔티티가 가진 컬럼은 **오직 SELECT 목록에 있는 것들뿐**이다.

`@restrict`는 이 CDS 엔티티의 컬럼만 볼 수 있다:

```cds
view MenusRoleAppliedList as select distinct
    menu.menu_code,       -- → 이 view는 menu_code 컬럼을 가짐
    menu.menu_name,       -- → menu_name 컬럼을 가짐
    core_user.user_id     -- → user_id 컬럼을 가짐 ← ★ 이게 없으면 @restrict 실패!
```

만약 `core_user.user_id`를 SELECT에서 빼면, 이 view 엔티티에는 `user_id`라는 컬럼이 존재하지 않게 된다:

```cds
@(restrict: [
    { grant: 'READ', where: 'user_id = $user' }
    -- ❌ 에러! "user_id? 이 view에 그런 컬럼 없는데?"
])
view MenusRoleAppliedList as select distinct
    menu.menu_code,
    menu.menu_name
    -- core_user.user_id  ← 빠졌다 → @restrict가 user_id를 못 찾음
```

| | 일반 SQL | CDS View |
|---|---|---|
| WHERE가 볼 수 있는 범위 | FROM에 있는 모든 테이블의 모든 컬럼 | **SELECT 목록에 있는 컬럼만** |
| user_id를 WHERE에서 쓰려면 | JOIN만 되어 있으면 OK | **반드시 SELECT에도 있어야 함** |

> `core_user.user_id`를 SELECT에 넣은 것은 필터링 자체를 위해서가 아니라, **"이 view에 user_id 컬럼이 있다"고 CDS에게 알려주기 위해서**다. 그게 없으면 `@restrict`가 실패한다.

### 4.5 MenuLoader: OData 호출

```javascript
// library_ott/src/util/MenuLoader.js
_fetchList: function() {
    const oODataModel = new ODataV4Model({
        serviceUrl: "/srv-api/odata/v4/ott-core/MenuManagement/"
    });
    const oBinding = oODataModel.bindList("/MenusRoleAppliedList");
    return oBinding.requestContexts();
    // CAP이 @restrict로 $user 필터를 자동 적용 → 사용자별 메뉴만 반환!
}
```

| 로그인 사용자 | 자동 추가되는 WHERE | 결과 메뉴 |
|-------------|-------------------|----------|
| `U001` | `AND user_id = 'U001'` | OTT_MAIN, DETAIL, MYPAGE, BOARD (4개) |
| `admin1` | `AND user_id = 'admin1'` | 위 4개 + SETTLEMENT, TREND (6개) |
| `hsan@zen...` | `AND user_id = 'hsan@...'` | CSV에 정의된 대로 |
