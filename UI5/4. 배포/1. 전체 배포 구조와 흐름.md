## 2. 배포 진입점: approuter_ott의 `mta.yaml`

모든 배포는 `approuter_ott`에서 시작됩니다. `mta.yaml`이 MTA(Multi-Target Application) 빌드를 총괄합니다.

```yaml
# cap-app/approuter_ott/mta.yaml
ID: com-approuterott
version: 1.0.0

modules:
  - name: com-approuterott          # ① approuter 자체 (Node.js)
    type: approuter.nodejs
    requires:
      - name: com-app-repo-runtime   # HTML5 Application Repository
      - name: com-app-destination    # BTP Destination 서비스
      - name: com-app-xsuaa          # XSUAA 인증
      - name: com-app-connectivity   # Connectivity

  - name: com-sysmgtott-content      # ② sysmgt_ott 콘텐츠
    type: com.sap.application.content
    requires:
      - name: com-sysmgtott-repo-host
        parameters:
          content-target: true
    build-parameters:
      requires:
        - name: comsysmgtott

  - name: comsysmgtott               # ③ sysmgt_ott 빌드 모듈
    type: html5
    path: ../sysmgt_ott              # ← 상대경로로 참조
    build-parameters:
      build-result: dist
      builder: custom
      commands:
        - npm install
        - npm run build:cf           # ui5 build preload ...
```

> 참고: board, settlement, trendAnalysis, portal_ott, library_ott도 동일한 패턴으로 mta.yaml에 정의되어 있습니다. 각각 `type: html5`, `path: ../{모듈명}` 구조입니다.

## 3. 배포 명령어 체인

```json
// approuter_ott/package.json
"scripts": {
    "build:dev": "node -e \"require('fs').copyFileSync('xs-app-dev.json','xs-app.json')\" && mbt build",
    "build:cf": "mbt build",
    "deploy:cf": "cf deploy mta_archives/com-approuterott_1.0.0.mtar",
    "deploy:mta": "npm run build:dev && npm run deploy:cf"
}
```

### `deploy:mta` 실행 흐름

```
npm run deploy:mta
  │
  ├─ ① npm run build:dev
  │     ├─ xs-app-dev.json → xs-app.json (덮어쓰기!)
  │     └─ mbt build (MTA 아카이브 생성)
  │           ├─ approuter_ott 자체 빌드
  │           ├─ portal_ott → ui5 build → dist/
  │           ├─ library_ott → ui5 build → dist/
  │           ├─ sysmgt_ott → ui5 build → dist/
  │           ├─ board → ui5 build → dist/
  │           ├─ settlement → ui5 build → dist/
  │           └─ trendAnalysis → ui5 build → dist/
  │     → mta_archives/com-approuterott_1.0.0.mtar 생성
  │
  └─ ② npm run deploy:cf
        └─ cf deploy mta_archives/...mtar → Cloud Foundry 배포
```

## 4. xs-app-dev.json → xs-app.json 덮어쓰기

```
개발자가 수정하는 파일:  xs-app-dev.json
                            │
                  build:dev 시점에 복사
                            ▼
실제 배포에 포함되는 파일:  xs-app.json
```

**루트의 `xs-app.json`은 의미가 없습니다.** `build:dev`에서 `xs-app-dev.json`으로 완전히 덮어쓰기 때문입니다. BTP 배포 시 실제로 사용되는 라우팅 규칙은 `xs-app-dev.json`의 내용입니다.

## 5. BTP 서비스 구조

```
com-approuterott (approuter)
  ├── com-app-repo-runtime (html5-apps-repo)    ← 정적 파일 서빙
  ├── com-app-destination (destination)          ← 백엔드 주소록
  ├── com-app-xsuaa (xsuaa)                      ← 사용자 인증
  └── com-app-connectivity (connectivity)        ← 외부 연결

BTP Destination (BTP Cockpit에서 설정):
  ├── onCapService  → core_ott 백엔드 (인증, 메뉴, 사용자)
  └── onOTTService  → ott 백엔드 (콘텐츠, 멤버십, 정산)
```

### Destination 인증 방식

두 destination 모두 `OAuth2JWTBearer` 방식:
- 사용자의 XSUAA JWT 토큰을 bearer assertion으로 사용
- XSUAA token service에서 새 토큰 발급
- 백엔드(CAP)로 JWT 포함하여 전달

## 6. HTML5 Application Repository

BTP에서는 정적 파일(HTML, JS, CSS)을 `html5-apps-repo-rt` 서비스가 서빙합니다. 로컬의 `localDir`과 대응됩니다:

| 환경 | 정적 파일 제공 방식 |
|------|---------------------|
| 로컬 | `localDir`: 디스크에서 직접 읽기 |
| BTP | `service: "html5-apps-repo-rt"`: Repository에서 읽기 |

각 HTML5 모듈은 빌드 시 `dist/` 폴더에 번들링되어 Repository에 업로드됩니다.
