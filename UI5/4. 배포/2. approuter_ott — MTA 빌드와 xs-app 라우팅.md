# approuter_ott — MTA 빌드와 xs-app 라우팅

> **작성일**: 2026-07-13
> **목적**: approuter_ott의 MTA 빌드 프로세스와 BTP에서의 xs-app 라우팅 동작을 이해한다.

---

## 1. MTA 빌드 트리거

```json
// approuter_ott/package.json
"build:dev": "node -e \"require('fs').copyFileSync('xs-app-dev.json','xs-app.json')\" && mbt build"
```

**두 단계**:
1. `xs-app-dev.json`을 `xs-app.json`으로 **복사 덮어쓰기**
2. `mbt build`로 MTA 아카이브 생성

---

## 2. xs-app-dev.json 라우트 분석 (BTP 운영)

### 2.1 API 라우트 (3-way 분기)

```json
// ① ott-core 경로 → core_ott 백엔드
{
    "source": "^.*/srv-api/(odata/v[24]/ott-core/.*)$",
    "target": "$1",
    "destination": "onCapService",
    "authenticationType": "none"
}

// ② "ott"가 포함된 경로 → ott 백엔드
{
    "source": "^.*/srv-api/(.*ott.*)$",
    "target": "$1",
    "destination": "onOTTService",
    "authenticationType": "none"
}

// ③ 그 외 모든 API → ott 백엔드 (XSUAA 인증)
{
    "source": "^.*/srv-api/(.*)$",
    "target": "$1",
    "destination": "onOTTService",
    "authenticationType": "xsuaa"
}
```

### 2.2 정적 파일 라우트 (html5-apps-repo-rt)

```json
// portal_ott
{ "source": "^.*/portal_ott/(.*)$",
  "target": "/comportalott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa" }

// sysmgt_ott (detailPage, myPage, mainPage)
{ "source": "^.*/sysmgt_ott/(.*)$",
  "target": "/comsysmgtott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa" }

// library_ott
{ "source": "/common_ott.lib/(.*)",
  "target": "/common_ottlib-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa" }

// board, settlement, trendAnalysis도 동일 패턴
```

### 2.3 Catch-all

```json
{ "source": "(.*)",
  "target": "/comportalott-1.0.0/$1",
  "service": "html5-apps-repo-rt",
  "authenticationType": "xsuaa" }
```

## 4. authenticationType 차이와 영향

| authenticationType | 동작 |
|:---:|---|
| `"xsuaa"` | approuter가 인증 강제. 미인증 시 XSUAA 로그인 페이지로 리다이렉트. JWT 토큰 보유 보장 → OAuth2JWTBearer 교환 가능 |
| `"none"` | 인증 검사 없음. **단, OAuth2JWTBearer destination에 대해 JWT 토큰 교환을 수행하지 않을 수 있음** |

### 주의: Route ② (ott 경로)의 `authenticationType: "none"`

```
myPage OData: /srv-api/odata/v2/ott/Mypage → Route ②
  → authenticationType: "none"
  → OAuth2JWTBearer 교환 미수행 가능성
  → ott 백엔드에 인증 토큰 없이 전달 → 401
```

Route ③으로 빠지는 `/srv-api/odata/v2/detail`은 `authenticationType: "xsuaa"`라 정상 동작합니다. Route ②도 `xsuaa`로 변경하거나, `forwardAuthToken: true`를 추가하는 것이 더 안전합니다.

## 5. 로컬 vs BTP 비교

| | 로컬 (`dev/xs-app.json`) | BTP (`xs-app-dev.json`) |
|---|---|---|
| 정적 파일 | `localDir`: 디스크 직접 읽기 | `service: "html5-apps-repo-rt"`: Repository |
| API 인증 | `authenticationType: "none"` → CAP mocked auth | `authenticationType: "xsuaa"` → XSUAA |
| Destination | `dev/default-env.json` (localhost:8084 등) | BTP Cockpit Destination 서비스 (실제 CF URL) |
