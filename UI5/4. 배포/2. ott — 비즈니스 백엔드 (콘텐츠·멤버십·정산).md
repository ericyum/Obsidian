# 2. ott — 비즈니스 백엔드 (콘텐츠·멤버십·정산)

> **작성일**: 2026-07-13
> **역할**: ZEN-OTT의 핵심 비즈니스 데이터를 다루는 백엔드. 콘텐츠 상세, 멤버십·구독, 정산, 게시판 등의 API를 제공합니다. core_ott가 먼저 배포된 후 배포합니다.

---

## 📂 모듈 개요

```
cap-node/ott/
├── .cdsrc.json          ← CAP 프로젝트 설정 (HANA + 개발환경 분기)
├── mta.yaml             ← MTA 배포 정의
├── package.json         ← npm 스크립트
├── index.cds            ← CDS 진입점
├── db/data/             ← BTP HANA 배포용 CSV (콘텐츠·사용자·리뷰 등)
├── srv/
│   ├── cds/             ← 서비스 정의 (DetailService, MembershipService 등)
│   ├── src/config/      ← 핸들러 로직 (TypeScript)
│   └── server.ts        ← x-forwarded-for 패치
└── xs-security.json     ← XSUAA 보안 설정 (독자적 xsappname)
```

## 🔑 핵심 배포 파일들

### ① `.cdsrc.json` — DB·인증 환경 분기

```json
{
  "requires": {
    "db": { "kind": "hana" },              // 프로덕션: HANA
    "[development]": {
      "db": { "kind": "sqlite", "credentials": { "url": ":memory:" } },
      "auth": {
        "kind": "mocked",
        "users": {
          "admin1": { "roles": ["Admin", "authenticated-user"] },
          "U001": { "roles": ["authenticated-user"] },
          ...
        }
      }
    }
  }
}
```

core_ott와 거의 동일한 구조입니다. 로컬은 SQLite + Mock 인증, BTP는 HANA + XSUAA JWT입니다.

`[hybrid]` 프로파일도 정의되어 있어, 로컬에서 BTP HANA에 연결하는 하이브리드 테스트도 가능합니다.

---

### ② `mta.yaml` — MTA 배포 정의

```yaml
_schema-version: 3.3.0
ID: com-cap-ott
version: 1.0.0
description: "OTT streaming platform CAP service."
parameters:
  enable-parallel-deployments: true

build-parameters:
  before-all:
    - builder: custom
      commands:
        - npm install
        - npx cds build --production
        - tsc -p tsconfig.cdsbuild.json

modules:
  # 모듈 1: Node.js 서버
  - name: com-cap-ott
    type: nodejs
    path: gen/srv
    parameters:
      instances: 1
      buildpack: nodejs_buildpack
    build-parameters:
      builder: npm-ci
    provides:
      - name: srv-api
        properties:
          srv-url: ${default-url}
    requires:
      - name: com-hdi-container               # ← ott 전용 HDI 컨테이너
      - name: com-cap-ott-destination
      - name: com-cap-ott-xsuaa

  # 모듈 2: HANA DB Deployer
  - name: com-db-deployer
    type: hdb
    path: gen/db
    parameters:
      buildpack: nodejs_buildpack
    requires:
      - name: com-hdi-container

resources:
  - name: com-hdi-container
    type: com.sap.xs.hdi-container
    parameters:
      service: hana
      service-plan: hdi-shared
      # ★ 핵심: core_ott와 별도의 HDI 컨테이너 사용
      # 같은 컨테이너 공유 시 deployer 간 테이블 드롭 위험
      service-name: com-cap-ott-hdi-container-${space}

  - name: com-cap-ott-destination
    type: org.cloudfoundry.existing-service
    parameters:
      service-name: com-app-destination-${space}

  - name: com-cap-ott-xsuaa
    type: org.cloudfoundry.existing-service
    parameters:
      service-name: com-app-xsuaa-${space}
```

**core_ott와의 차이점**:

| 항목 | core_ott | ott |
|------|----------|-----|
| HDI 컨테이너 | `com-ott-core-hdi-container` | `com-cap-ott-hdi-container` ← **별도!** |
| Destination | `existing-service` (공유) | `existing-service` (공유) |
| XSUAA | `existing-service` (공유) | `existing-service` (공유) |
| 포트 (로컬) | 8084 | 8083 |

> **HDI 컨테이너를 분리한 이유**: 두 백엔드가 같은 HDI를 공유하면 DB Deployer가 서로의 테이블을 드롭할 위험이 있기 때문입니다. 데이터는 분리되어 있지만, `user_id`로 논리적 연결됩니다.

### ④ `db/data/` — 비즈니스 CSV 데이터

```
db/data/
└── com.cap.ott-Users.csv   ← 구독 사용자 데이터 (U001~U011)
```

**중요**: ott의 Users는 **인증이 아닌 비즈니스 데이터**입니다.

- `user_id` 컬럼: core_ott의 User 테이블을 참조하는 FK
- 로컬: `U001`, `U002` 등 mock 계정
- BTP: XSUAA 이메일 (`hsan@zen.com` 등) — `@restrict`가 있는 서비스는 이 값이 `$user`와 일치해야 함

```csv
user_id,user_name,subscription_plan,subscription_status,...
U001,Kim Minjun,PREMIUM,active,...
U002,Lee Seoyeon,BASIC,active,...
```

**BTP 배포 시 주의**: `@restrict`가 있는 엔티티(예: MembershipService의 UserSet)는 `$user`(XSUAA 이메일)와 일치하는 레코드가 DB에 존재해야 데이터가 반환됩니다. `DetailService`의 Users는 `@restrict`가 없어서 모든 사용자에게 데이터가 보입니다.

### ⑤ `xs-security.json` — 독자적 보안 설정

```json
{
  "xsappname": "com-cap-ott",
  "tenant-mode": "dedicated",
  "scopes": [
    {
      "name": "$XSAPPNAME.Admin",
      "description": "OTT 관리자 권한"
    }
  ],
  "role-templates": [
    {
      "name": "Admin",
      "description": "OTT 관리자 (정산·구독자 추세 분석)",
      "scope-references": ["$XSAPPNAME.Admin"]
    }
  ]
}
```

> ⚠️ ott의 `xs-security.json`은 **현재 mta.yaml에서 참조되지 않습니다.** XSUAA는 approuter_ott가 만든 `com-app-xsuaa`를 `existing-service`로 재사용하므로, 실제 인증에 사용되는 xsappname은 approuter_ott의 `xs-security.json`에 정의된 `com-app-xsapp`입니다. 이 파일은 추후 ott가 독자적 XSUAA 인스턴스를 가질 때를 대비한 예비 설정입니다.

### ⑥ `srv/server.ts` — x-forwarded-for 패치

```typescript
import cds from '@sap/cds';
import express from 'express';

cds.on('bootstrap', (app: express.Application) => {
    app.use((req, res, next) => {
        if (req.headers['x-forwarded-for'] === 'undefined') {
            delete req.headers['x-forwarded-for'];
        }
        next();
    });
});

export default (options: any): any => {
    return cds.server(options);
}
```

core_ott의 server.ts에서 i18n 관련 코드를 제외한 것과 동일한 x-forwarded-for 패치가 적용되어 있습니다.

## 🔄 ott가 제공하는 서비스들

| 서비스 | 용도 | approuter 라우트 |
|--------|------|-----------------|
| DetailService | 콘텐츠 상세·리뷰 CRUD | `odata/v2/detail/` |
| MembershipService | 멤버십·구독 관리 | `odata/v2/ott/Mypage/` |
| FreeBoard | 자유 게시판 | `odata/v2/ott/FreeBoard/` |
| SettlementService | 정산 데이터 | `odata/v2/ott/Settlement/` |

이 요청들은 approuter의 `onOTTService` destination을 통해 ott로 전달됩니다.

## 🔗 core_ott ↔ ott 연결 구조

```
core_ott (HDI #1: com-ott-core-hdi-container)
├── User        ← user_id (PK)
├── Menu
├── Role
└── Message

        ↑ user_id로 논리적 연결 (FK)

ott (HDI #2: com-cap-ott-hdi-container)
├── Users       ← user_id (FK → core_ott.User)
├── Contents
├── Reviews
└── Memberships
```

두 백엔드는 **물리적으로 분리된 HDI 컨테이너**를 사용하지만, `user_id`를 통해 논리적으로 연결됩니다.

## ⚠️ 배포 시 주의사항

| # | 주의사항 |
|---|---------|
| 1 | HDI 컨테이너가 core_ott와 **분리**되어 있는지 확인 (`com-cap-ott-hdi-container`) |
| 2 | BTP `$user` = XSUAA 이메일. `@restrict` 적용된 서비스는 Users 테이블에 해당 이메일 레코드 필요 |
| 3 | 포트: 로컬은 **8083** (core_ott의 8084와 겹치지 않도록) |
| 4 | `xs-security.json`은 현재 사용되지 않음 (approuter의 XSUAA 재사용). 착오 금지 |
