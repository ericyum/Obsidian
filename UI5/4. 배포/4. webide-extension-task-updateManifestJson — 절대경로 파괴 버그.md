# webide-extension-task-updateManifestJson — 절대경로 파괴 버그

> **작성일**: 2026-07-13
> **목적**: `webide-extension-task-updateManifestJson`가 무엇이고 왜 제거해야 하는지 이해한다.

---

## 1. 이 태스크가 무엇인가

`@sap/ui5-builder-webide-extension` 패키지의 빌드 태스크입니다. SAP Web IDE / BAS 시절에 만들어졌으며, `manifest.json`의 `dataSources[].uri` 값을 "배포 환경에 맞게" 변환합니다.

```yaml
# sysmgt_ott/ui5-deploy.yaml에 있었던 설정 (현재 제거됨)
customTasks:
  - name: webide-extension-task-updateManifestJson
    afterTask: replaceVersion
    configuration:
      appFolder: webapp
      destDir: dist
```

---

## 2. 이 태스크가 한 짓

**manifest.json의 URI에서 맨 앞의 `/`를 제거했습니다.**

```
빌드 전 (원본 manifest.json):
  "uri": "/srv-api/odata/v2/ott/Mypage"
         ↑
         절대경로 (슬래시로 시작)

빌드 후 (dist/manifest.json):
  "uri": "srv-api/odata/v2/ott/Mypage"
         ↑
         상대경로로 변환됨!
```

---

## 3. 왜 문제가 되는가

### 절대경로 (`/`로 시작)

```
페이지 위치: https://호스트/sysmgt_ott/myPage/webapp/index.html
OData URI:  /srv-api/odata/v2/ott/Mypage
실제 요청:  https://호스트/srv-api/odata/v2/ott/Mypage  ✅
```

### 상대경로 (`/` 없음)

```
페이지 위치: https://호스트/sysmgt_ott/myPage/webapp/index.html
OData URI:  srv-api/odata/v2/ott/Mypage
실제 요청:  https://호스트/sysmgt_ott/myPage/webapp/srv-api/odata/v2/ott/Mypage  ❌
```

**상대경로가 되면 현재 HTML 페이지 위치를 기준으로 URL이 조합됩니다.** 전혀 엉뚱한 URL이 되어 `xs-app-dev.json`의 어떤 라우트에도 매칭되지 않고 404가 발생합니다.

---

## 4. 왜 원래 이런 게 있었는가

SAP Web IDE 시절, 앱이 FLP(SAP Fiori Launchpad) 안에서 실행될 때는 URL 구조가 달랐습니다. 이 태스크는 그 환경에 맞춰 경로를 보정하기 위해 만들어졌습니다.

**approuter로 직접 서빙하는 현대적인 독립 배포 방식에서는 전혀 필요 없고 오히려 해롭습니다.**

---

## 5. 해결책

`ui5-deploy.yaml`에서 해당 태스크를 **제거**합니다:

```yaml
# 제거할 부분
customTasks:
  - name: webide-extension-task-updateManifestJson  # ← 이 블록 전체 삭제
    afterTask: replaceVersion
    configuration:
      appFolder: webapp
      destDir: dist
```

---

## 6. 영향받는 모듈

이 태스크가 있었던 모듈:
- `sysmgt_ott/ui5-deploy.yaml` → 제거 완료

없었던 모듈 (영향 없음):
- `board/ui5-deploy.yaml`
- `portal_ott/ui5-deploy.yaml`
- `library_ott/ui5-deploy.yaml`

---

## 7. 교훈

**`ui5-deploy.yaml`의 `customTasks`를 검토할 때 `webide-extension-task-updateManifestJson`가 있다면 무조건 제거해야 합니다.** 모든 OData 호출의 URI가 깨져서 앱 전체가 동작하지 않습니다.
