# 백엔드 — CAP 배포와 HANA CSV 데이터

> **작성일**: 2026-07-13
> **목적**: core_ott와 ott 백엔드의 BTP 배포 구조와 CSV 데이터 로딩 메커니즘을 이해한다.

---

## 1. CAP 프로덕션 설정 (`.cdsrc.json`)

```json
// ott/.cdsrc.json
{
  "requires": {
    "db": { "kind": "hana" },             // ← 프로덕션: HANA DB
    "[development]": {
      "db": { "kind": "sqlite", ... },    // ← 로컬: SQLite
      "auth": { "kind": "mocked", ... }   // ← 로컬: Mocked Auth
    }
  }
}
```

| 환경 | DB | 인증 |
|------|:--:|------|
| 로컬 (`cds watch`) | SQLite (in-memory) | Mocked (U001, admin1) |
| BTP (프로덕션) | HANA | XSUAA JWT |

---

## 2. CSV 데이터 배포 — `db/data/` vs `test/data/`

```
core_ott/
├── db/data/     ← BTP HANA 배포용 CSV
│   ├── Menu.csv
│   ├── Role.csv
│   ├── RoleGroup.csv
│   ├── Role_Menu.csv
│   ├── Role_RoleGroup.csv
│   └── User_RoleGroup.csv
│
└── test/data/   ← 로컬 SQLite 테스트용 CSV
    ├── User.csv
    └── User_RoleGroup.csv
```

| 폴더 | 대상 | 로딩 시점 |
|------|------|-----------|
| `db/data/` | BTP HANA | `cds deploy` 또는 MTA 배포 시 |
| `test/data/` | 로컬 SQLite | `cds watch` 실행 시 |

**주의**: `db/data/`에 User.csv를 직접 넣으면 **개인정보(이메일)가 Git에 노출**됩니다. 대신:
- BTP Cockpit에서 HANA DB에 직접 INSERT
- 또는 SCIM 프로비저닝으로 XSUAA → HANA 자동 동기화 (정석)

## 3. ott의 Users 테이블 — 인증이 아니라 구독 데이터

```csv
# ott/db/data/com.cap.ott-Users.csv
user_id,user_name,subscription_plan,subscription_status,...
U001,Kim Minjun,PREMIUM,active,...
U002,Lee Seoyeon,BASIC,active,...
...
U011,Seo Jiwan,,none,,...
```

**중요**: ott의 Users는 인증을 담당하지 않습니다. 인증은 core_ott + XSUAA가 처리합니다. ott의 Users는 **"이 user_id를 가진 고객의 구독 플랜은 무엇인가"** 라는 비즈니스 데이터입니다.

`user_id`는 core_ott의 User 테이블과 ott의 Users 테이블을 연결하는 **외래키**입니다:
- 로컬: `U001`, `admin1` 등 mock 계정
- BTP: XSUAA 이메일 (`hsan@zen.com` 등)

---

## 4. `@restrict`와 BTP 사용자

```cds
// ott/srv/cds/Membership-service.cds
@(restrict: [
    { grant: 'READ', where: 'user_id = $user' }
])
entity UserSet as projection on Users;
```

BTP에서 `$user` = XSUAA 이메일이므로, ott Users 테이블에 **동일한 이메일로 레코드가 존재해야** 데이터가 반환됩니다.

```
로컬:  $user = 'U001'      → Users 테이블에 U001 있음 → ✅ 데이터 나옴
BTP:   $user = 'hsan@zen.com' → Users 테이블에 U001~U011만 있음 → ❌ 빈 결과
```

**해결**: `hsan@zen.com`을 `user_id`로 하는 레코드를 ott Users 테이블에 추가해야 합니다.

> 반면 `DetailService.cds`의 `Users` 엔티티는 `@restrict`가 없어서 모든 사용자에게 데이터가 반환됩니다. 그래서 상세 페이지는 BTP에서도 정상 동작합니다.

## 5. 핵심 요약

| # | 내용 |
|---|------|
| 1 | 로컬은 SQLite + mocked auth, BTP는 HANA + XSUAA JWT |
| 2 | `db/data/` CSV는 BTP HANA 배포용, `test/data/`는 로컬 SQLite 테스트용 |
| 3 | User.csv는 PII(이메일) 포함 → 직접 커밋 금지. DB 직접 INSERT 또는 SCIM 사용 |
| 4 | ott의 Users는 인증 테이블이 아닌 **구독 비즈니스 데이터** 테이블 |
| 5 | `@restrict`가 있는 엔티티는 BTP `$user`(XSUAA 이메일)와 일치하는 레코드가 반드시 필요 |
| 6 | core_ott와 ott는 `user_id`로 연결 — 양쪽에 동일한 사용자 레코드가 있어야 함 |
