# 5. DetailMain.view.xml 완전 이해

**날짜**: 2026-07-03
**범위**: `detailPage/webapp/view/DetailMain.view.xml` — 컨텐츠 상세 페이지의 전체 UI 구조
**이전 단계**: [[4. BaseController.js_완전_이해]], [[3. index.html_완전_이해]], [[2. Component.js_완전_이해]], [[1. manifest.json_완전_이해]]

## 개요: 이 파일의 정체

`DetailMain.view.xml`은 **detailPage의 유일한 실질적 UI 화면**이다. 사용자가 `#/Detail/C001` 같은 URL로 접근하면 이 XML이 렌더링된다. `manifest.json` → `Component.js` → 라우터 → 이 파일 로드 순서로 도달한다.

**코드량**: 약 215줄
**연결된 컨트롤러**: `detailPage.controller.DetailMain` (`controllerName="detailPage.controller.DetailMain"`)
**연결된 모델**: `viewModel` (JSONModel), `detail` (OData V2), `i18n` (ResourceModel), `User` (Core)

## 1. XML 네임스페이스 — "사용할 부품 상자들"

```xml
<mvc:View
    controllerName="detailPage.controller.DetailMain"
    xmlns:mvc="sap.ui.core.mvc"
    xmlns="sap.m"
    xmlns:f="sap.f"
    xmlns:uxap="sap.uxap"
    xmlns:t="sap.ui.table"
    xmlns:form="sap.ui.layout.form"
    xmlns:layout="sap.ui.layout"
    xmlns:core="sap.ui.core"
    height="100%">
```

|    접두어    | 네임스페이스               | 용도                         | 이 뷰에서 쓰는 곳                                          |
| :-------: | -------------------- | -------------------------- | --------------------------------------------------- |
|  `mvc:`   | `sap.ui.core.mvc`    | 뷰 자체의 정의                   | `<mvc:View>` 전체를 감쌈                                 |
|  *(기본)*   | `sap.m`              | 기본 UI 컨트롤 (버튼, 텍스트, 리스트 등) | `Button`, `Text`, `List`, `MessageStrip` 등 가장 많이 사용 |
|   `f:`    | `sap.f`              | Fiori 고급 컨트롤               | 이 뷰에서는 **사용 안 함** (선언만, 과거 `DynamicPage` 잔재)        |
|  `uxap:`  | `sap.uxap`           | ObjectPage 패턴              | `ObjectPageLayout`, `ObjectPageDynamicHeaderTitle`  |
|   `t:`    | `sap.ui.table`       | 데이터 테이블                    | 출연진 테이블 (`t:Table`, `t:Column`, `t:template`)       |
|  `form:`  | `sap.ui.layout.form` | 폼 레이아웃                     | 리뷰 작성 폼 (`SimpleForm`)                              |
| `layout:` | `sap.ui.layout`      | 배치 도구                      | `GridData` (폼 내 필드의 열 너비 설정)                        |
|  `core:`  | `sap.ui.core`        | 코어 요소                      | 이 뷰에서는 사용 안 함 (선언만)                                 |

> **암기**: XML에서 `<Button>`이라고만 쓰면 `sap.m.Button`을 의미한다. 기본 네임스페이스(`xmlns="sap.m"`) 덕분이다. `sap.ui.table.Table`을 쓰려면 `<t:Table>`이라고 접두어를 붙여야 한다.

## 2. 전체 레이아웃 구조 — "4개 구역"

```
mvc:View (height="100%")
└── uxap:ObjectPageLayout          ← 루트 컨테이너
    ├── uxap:headerTitle           ← ① 상단 헤더
    │   └── ObjectPageDynamicHeaderTitle
    └── uxap:sections              ← ②~④ 섹션 배열
        ├── ObjectPageSection: "기본 정보"
        │   └── Card/CSSGrid (content_id, type, partner, 장르, 태그)
        ├── ObjectPageSection: "출연진"
        │   └── sap.ui.table.Table (이름·역할·캐릭터명)
        └── ObjectPageSection: "리뷰"
            ├── MessageStrip (이미 작성 시)
            ├── List (리뷰 목록)
            └── SimpleForm (작성/수정 폼)
```

### 왜 `ObjectPageLayout`이 루트인가?

과거에는 `f:DynamicPage` + `uxap:ObjectPageLayout`을 중첩해서 썼다. 그런데 두 컨트롤러가 **각자 자신의 스크롤 컨테이너**를 만들기 때문에, spacer div가 무한히 확장되는 버그가 있었다. `ObjectPageLayout` 하나만으로 `snappedContent`(스크롤 축소), `expandedContent`(펼친 상태), `actions`(우측 버튼)을 모두 지원하므로 **래퍼 없이 루트로 사용**한다.

## 3. ① 헤더 — `ObjectPageDynamicHeaderTitle`

```xml
<uxap:ObjectPageDynamicHeaderTitle>
    <uxap:heading>          <!-- 항상 보이는 제목 -->
        <Title text="{viewModel>/Detail/title}" />
    </uxap:heading>

    <uxap:expandedContent>  <!-- 스크롤 맨 위에서만 보이는 확장 영역 -->
        <ObjectStatus text="..." state="Information" icon="sap-icon://video" />
        <RatingIndicator value="{viewModel>/Detail/avg_rating}" maxValue="5" enabled="false" />
        <Text text="{viewModel>/Detail/avg_rating}" />
        <Text text="{path: '...', formatter: '.formatReviewCount'}" />
    </uxap:expandedContent>

    <uxap:snappedContent>   <!-- 스크롤 내리면 축소되어 남는 영역 -->
        <ObjectStatus text="{viewModel>/Detail/content_type}" state="Information" icon="sap-icon://video" />
    </uxap:snappedContent>

    <uxap:actions>          <!-- 우측 상단 고정 액션 버튼 -->
        <Button icon="sap-icon://nav-back" press="onNavBack" />
    </uxap:actions>
</uxap:ObjectPageDynamicHeaderTitle>
```

### expandedContent vs snappedContent — 시각적 이해

```
[스크롤 최상단 — expandedContent 보임]
 ┌──────────────────────────────────────────────┐
 │  🎬 MOVIE  ⭐⭐⭐⭐ 3.75  (리뷰 4개)   [← 뒤로]  │  ← expandedContent
 │  어벤져스: 엔드게임                             │  ← heading (항상)
 ├──────────────────────────────────────────────┤
 │  [기본 정보]                                   │
 │  ...                                          │
 │                                               │
 │  ↓ 스크롤 내리면...                            │
 │                                               │
 ├──────────────────────────────────────────────┤
 │  🎬 MOVIE  어벤져스: 엔드게임      [← 뒤로]    │  ← snappedContent로 전환
 ├──────────────────────────────────────────────┤
 │  [기본 정보]                                   │
```

> **핵심**: `expandedContent`는 스크롤이 맨 위일 때만 보이고, 내리면 `snappedContent`가 그 자리를 대체한다. `heading`(타이틀)은 항상 보인다. Fiori의 표준 UX 패턴이다.

## 4. 데이터 바인딩 — 3가지 경로

이 뷰에서 데이터를 가져오는 경로는 3가지다.

### 4.1 `viewModel` — 화면 상태 + API 응답 데이터

```xml
text="{viewModel>/Detail/title}"
text="{viewModel>/Detail/content_id}"
value="{viewModel>/NewReview/rating}"
visible="{= !!${viewModel>/HasReviewed} }"
```

| 바인딩 | 의미 | 데이터 출처 |
|--------|------|-----------|
| `/Detail/title` | 현재 보고 있는 컨텐츠 데이터 전체 | `_loadDetail()` → OData 응답을 `viewModel.setProperty("/Detail", oData)` |
| `/NewReview/rating` | 작성 중인 리뷰의 별점 | `_onRouteMatched()`에서 `{rating:0, review_text:""}`로 초기화 |
| `/HasReviewed` | 현재 사용자가 이미 리뷰를 썼는지 | `_checkMyReview()`에서 `true`/`false` 설정 |
| `/Detail/reviews/results` | 리뷰 목록 배열 | OData 응답의 expanded `reviews` |
| `/MyReview` | 내가 쓴 리뷰 객체 | `_checkMyReview()`에서 찾아서 저장 |

**Model 타입**: `sap.ui.model.json.JSONModel` — Component.init() 시점에 빈 모델로 생성되고, 런타임에 `setProperty()`로 채워진다.

### 4.2 `i18n` — 다국어 텍스트

```xml
text="{i18n>button.submitReview}"
text="{i18n>section.basicInfo}"
placeholder="{i18n>placeholder.review}"
```

**Model 타입**: `sap.ui.model.resource.ResourceModel` — `i18n.properties` 파일에서 키-값 쌍을 읽어온다. 모든 레이블, 버튼 텍스트, 메시지, 플레이스홀더가 여기서 나온다.

### 4.3 `User` — 로그인 사용자 정보

```xml
visible="{= ${viewModel>user_user_id} === ${User>/user_id} }"
```

**Model 타입**: `sap.ui.model.json.JSONModel` — `UserSessionLoader`가 Core에 등록. 뷰가 직접 Core 모델을 참조할 수 있다. `/user_id` 프로퍼티로 현재 로그인한 사용자 ID를 가져온다.

> **주의**: `User` 모델은 JSONModel이지만 뷰의 `setModel`이 아니라 `sap.ui.getCore().setModel()`로 전역 등록되었기 때문에, 뷰에서 `{User>/user_id}`로 바로 접근 가능하다.

## 5. Expression Binding — `{= ... }` 문법

일반 바인딩 `{}`과 달리 `{= ... }`는 **JavaScript 표현식을 평가**한다.

```xml
<!-- 조건부 표시: HasReviewed가 true일 때만 -->
visible="{= !!${viewModel>/HasReviewed} }"

<!-- 산술: 데이터 개수와 15 중 작은 값 -->
visibleRowCount="{= Math.min(${viewModel>/Detail/casts/results}.length, 15) }"

<!-- 문자열 연산: '#' 접두어를 동적으로 붙임 -->
text="{= '#' + ${viewModel>tag/name}}"

<!-- 삼항 연산자: editing_review_id가 있으면 수정, 없으면 신규 -->
title="{= !!${viewModel>/NewReview/editing_review_id} ? ${i18n>label.editReview} : ${i18n>label.writeReview} }"

<!-- 비교: 현재 리뷰의 user_id가 로그인 사용자와 같으면 내 리뷰 -->
visible="{= ${viewModel>user_user_id} === ${User>/user_id} }"
```

### 규칙

| 요소 | 의미 |
|------|------|
| `{= ... }` | Expression binding 선언 |
| `${viewModel>/path}` | viewModel의 해당 경로 값 (복잡한 경로일 때 `$` 접두어 필수) |
| `{viewModel>simplePath}` | 단순 경로 (중첩 경로 없을 때만) |

> **언제 `$`가 필요한가?** : `${viewModel>/Detail/casts/results}.length` 처럼 바인딩 경로에 `.length` 같은 후처리가 붙을 때 `$`로 경계를 명확히 해야 한다. 단순히 `{viewModel>/Detail/title}` 같은 건 `$` 없이도 된다.

## 6. ② 기본 정보 섹션 — Card/CSSGrid 패턴

### 디자인 의도

SAP Fiori의 `f:Card` 컴포넌트를 쓰려 했지만, UI5 1.136 CDN에서 default aggregation을 지원하지 않아 **순수 CSS로 Card 패턴을 구현**했다.

```xml
<VBox class="detailInfoCard" width="100%">         ← 카드 박스 (흰 배경 + 둥근 모서리 + 그림자)
    <VBox class="detailCardGrid" width="100%">     ← CSS Grid (2열 반응형)
        <VBox class="detailCardGroup">              ← 그룹 1: 컨텐츠 식별
            <Title text="{i18n>card.identification}" level="H6" />
            <VBox class="detailCardGroupContent">
                <HBox class="detailCardRow">        ← 한 행: Label + Value
                    <Label text="{i18n>label.contentId}" class="sapMLabelShowColon" />
                    <Text text="{viewModel>/Detail/content_id}" />
                </HBox>
                ...
            </VBox>
        </VBox>
        <VBox class="detailCardGroup">              ← 그룹 2: 제공 정보
            ...
        </VBox>
        <VBox class="detailCardGroup">              ← 그룹 3: 분류
            ...
        </VBox>
    </VBox>
</VBox>
```

### Label:Value 패턴

```xml
<HBox alignItems="Center" class="detailCardRow">
    <Label text="{i18n>label.contentId}" class="sapMLabelShowColon detailCardLabel" />
    <Text text="{viewModel>/Detail/content_id}" class="detailCardValue" />
</HBox>
```

`Label`과 `Text`를 `HBox`로 가로 배치한 단순한 패턴. `sapMLabelShowColon` CSS 클래스가 라벨 뒤에 자동으로 `:` 콜론을 붙인다 (`컨텐츠 코드:`).

### CSS Grid의 역할

```css
.detailCardGrid {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(max(220px, 50%), 1fr));
    gap: 1rem;
}
```

- `auto-fill`: 공간이 허용하는 만큼 열을 자동 생성
- `minmax(max(220px, 50%), 1fr)`: 각 열의 최소 너비는 220px 또는 화면의 50% 중 큰 값, 최대는 남은 공간 균등 분배
- **결과**: 넓은 화면에서는 2열, 좁은 화면(모바일)에서는 1열로 자동 전환

> `sap.ui.layout.cssgrid` 라이브러리가 UI5 1.136 CDN에 없어서 순수 CSS Grid로 대체. BTP Destination Detail 페이지의 패턴을 그대로 가져왔다.

### 날짜 포매팅

```xml
<Text text="{ path: 'viewModel>/Detail/createdAt',
    type: 'sap.ui.model.odata.type.DateTime',
    formatOptions: { pattern: 'yyyy-MM-dd' } }" />
```

OData의 DateTime 문자열을 `yyyy-MM-dd` 형식으로 변환한다. `type` 속성에 SAP UI5의 내장 타입을 지정하면 `formatOptions`으로 출력 포맷을 제어할 수 있다.

## 7. ③ 출연진 섹션 — `sap.ui.table.Table`

```xml
<t:Table
    id="castTable"
    rows="{viewModel>/Detail/casts/results}"
    visibleRowCount="{= Math.min(${viewModel>/Detail/casts/results}.length, 15) }"
    enableSelectAll="false"
    selectionMode="None"
    enableColumnFreeze="true"
    width="auto">
```

### 왜 `sap.m.Table`이 아니라 `sap.ui.table.Table`인가?

| 특성 | `sap.m.Table` | `sap.ui.table.Table` |
|------|:------------:|:-------------------:|
| 정렬 | ❌ 수동 구현 | ✅ `sortProperty` 기본 지원 |
| 열 고정 | ❌ | ✅ `enableColumnFreeze` |
| 열 너비 조절 | ❌ | ✅ 사용자 리사이즈 가능 |
| 용도 | 단순 목록 | 데이터 그리드 |

출연진은 정렬·열 고정이 필요한 **데이터 그리드**에 가까워서 `sap.ui.table.Table`을 선택했다.

### 동적 행수

```xml
visibleRowCount="{= Math.min(${viewModel>/Detail/casts/results}.length, 15) }"
```

출연진이 15명 이하면 전체를 보여주고, 15명 초과면 15행 + 스크롤바. `Math.min(데이터개수, 15)`로 테이블 하단에 빈 행이 생기는 걸 방지한다.

### 컬럼 정의

```xml
<t:columns>
    <t:Column hAlign="Left" width="12rem" sortProperty="name">
        <Label text="{i18n>label.castName}" />           ← 헤더
        <t:template>
            <Text wrapping="false" text="{viewModel>name}" />  ← 각 행의 셀
        </t:template>
    </t:Column>
    ...
</t:columns>
```

`sap.ui.table.Table`의 컬럼은 `Label`(헤더) + `t:template`(셀) 구조로 정의한다. `sortProperty="name"`을 지정하면 사용자가 헤더 클릭 시 해당 필드로 정렬된다.

### `t:extension` — 테이블 상단 툴바

`sap.ui.table.Table`은 `extension` aggregation으로 테이블 상단에 툴바를 넣을 수 있다 (`sap.m.Table`의 `headerToolbar`에 해당):

```xml
<t:Table ...>
    <t:extension>
        <OverflowToolbar>
            <ToolbarSpacer />                    ← 오른쪽으로 밀기
            <Button
                icon="sap-icon://excel-attachment"
                text="{i18n>button.exportCast}"
                press="onExportCast" />
        </OverflowToolbar>
    </t:extension>
    <t:columns> ... </t:columns>
</t:Table>
```

```
┌──────────────────────────────────────────────┐
│                              [📊 출연진 내보내기] │  ← t:extension
├──────────┬──────────┬──────────────────────┤
│  이름    │  역할    │  캐릭터명              │  ← t:columns
├──────────┼──────────┼──────────────────────┤
│ 로다주   │ 주연     │ 아이언맨              │  ← rows (데이터)
└──────────┴──────────┴──────────────────────┘
```

`ToolbarSpacer`는 빈 공간을 전부 차지해서 뒤에 오는 버튼을 **오른쪽 끝**으로 밀어낸다.

## 8. ④ 리뷰 섹션

### 8.1 구조

```
Section: "리뷰"
├── SubSection 1: MessageStrip (visible=HasReviewed)        ← 조건부 최상단
├── SubSection 2: List (리뷰 목록, 항상)
└── SubSection 3: SimpleForm (작성/수정 폼, 조건부)
```

### 8.2 MessageStrip — "이미 리뷰를 작성하셨습니다"

```xml
<uxap:ObjectPageSubSection
    visible="{= !!${viewModel>/HasReviewed} }">
    <MessageStrip
        text="{i18n>message.alreadyReviewed}"
        type="Information"
        showIcon="true" />
</uxap:ObjectPageSubSection>
```

- `visible="{= !!${viewModel>/HasReviewed} }"` → `_checkMyReview()`가 찾은 내 리뷰가 있을 때만 표시
- `type="Information"` → 파란색 정보 배너 (⚠️경고가 아닌 ℹ️안내)
- 리뷰 섹션 **최상단**에 배치 (과거에는 "내 리뷰" 하위 섹션 안에 있었음)

### 8.3 리뷰 목록 — `List` + `CustomListItem`

```xml
<List
    items="{viewModel>/Detail/reviews/results}"
    noDataText="{i18n>message.noReview}">
    <CustomListItem>
        <VBox>
            <HBox>
                <RatingIndicator value="{viewModel>rating}" maxValue="5" enabled="false" iconSize="0.875rem" />
                <Text text="{viewModel>user/user_name}" />
            </HBox>
            <Text text="{viewModel>review_text}" />
            <HBox>
                <Text text="{ path: 'viewModel>createdAt', type: 'sap.ui.model.odata.type.DateTime', formatOptions: { pattern: 'yyyy-MM-dd' } }" />
                <ToolbarSpacer />
                <Button icon="sap-icon://edit" press="onEditReview"
                    visible="{= ${viewModel>user_user_id} === ${User>/user_id} }" />
                <Button icon="sap-icon://delete" press="onDeleteReview"
                    visible="{= ${viewModel>user_user_id} === ${User>/user_id} }" />
            </HBox>
        </VBox>
    </CustomListItem>
</List>
```

#### 본인 확인 로직

```xml
visible="{= ${viewModel>user_user_id} === ${User>/user_id} }"
```

리뷰의 `user_user_id`(작성자 FK)와 `User` 모델의 `/user_id`(로그인 사용자)를 비교한다. 일치하면 ✏️수정·🗑️삭제 버튼이 보인다. **자신이 쓴 리뷰만 수정/삭제 가능**.

#### `ToolbarSpacer`의 역할

날짜와 버튼 사이를 최대한 벌려서, **날짜는 왼쪽, 버튼은 오른쪽 끝**에 정렬한다. `FlexBox justifyContent="SpaceBetween"` 없이도 간단하게 구현하는 SAP UI5 패턴.

#### RatingIndicator

```xml
<RatingIndicator value="{viewModel>rating}" maxValue="5" enabled="false" iconSize="0.875rem" />
```

- `enabled="false"` → 읽기 전용 (클릭 불가)
- `iconSize="0.875rem"` → 작은 별 (헤더의 1rem보다 작게)
- `maxValue="5"` → 최대 5점

### 8.4 리뷰 작성/수정 폼

```xml
<uxap:ObjectPageSubSection
    visible="{= !${viewModel>/HasReviewed} || !!${viewModel>/NewReview/editing_review_id} }"
    title="{= !!${viewModel>/NewReview/editing_review_id} ? ${i18n>label.editReview} : ${i18n>label.writeReview} }">
    <form:SimpleForm layout="ResponsiveGridLayout" editable="true">
        <form:content>
            <VBox>
                <Label text="{i18n>label.rating}" />
                <RatingIndicator value="{viewModel>/NewReview/rating}" maxValue="5" iconSize="1.5rem" />
            </VBox>
            <VBox>
                <TextArea value="{viewModel>/NewReview/review_text}" placeholder="{i18n>placeholder.review}" maxLength="1000" rows="3" width="100%" />
            </VBox>
            <FlexBox justifyContent="End">
                <Button text="{= !!${viewModel>/NewReview/editing_review_id} ? ${i18n>button.saveEdit} : ${i18n>button.submitReview} }"
                    type="Emphasized" press="onSubmitReview" />
                <Button text="{i18n>button.cancel}" press=".onCancelEdit"
                    visible="{= !!${viewModel>/NewReview/editing_review_id} }" />
            </FlexBox>
        </form:content>
    </form:SimpleForm>
</uxap:ObjectPageSubSection>
```

#### 표시 조건

```xml
visible="{= !${viewModel>/HasReviewed} || !!${viewModel>/NewReview/editing_review_id} }"
```

- `!HasReviewed` → 아직 리뷰를 안 썼으면 **신규 작성 폼** 표시
- `!!editing_review_id` → 수정 모드 진입 시 **수정 폼** 표시 (기존 리뷰가 있어도)

#### 동적 타이틀 + 버튼 텍스트

```xml
title="{= !!${viewModel>/NewReview/editing_review_id} ? ${i18n>label.editReview} : ${i18n>label.writeReview} }"
text="{= !!${viewModel>/NewReview/editing_review_id} ? ${i18n>button.saveEdit} : ${i18n>button.submitReview} }"
```

`editing_review_id`의 존재 여부로 **신규 등록 vs 수정** 모드를 구분한다:
- 신규: 타이틀="리뷰 작성", 버튼="등록", 취소 버튼 숨김
- 수정: 타이틀="리뷰 수정", 버튼="수정 완료", 취소 버튼 표시

#### RatingIndicator (작성용)

```xml
<RatingIndicator value="{viewModel>/NewReview/rating}" maxValue="5" iconSize="1.5rem" />
```

목록의 RatingIndicator와 달리 `enabled` 속성이 없다 → 기본값 `true` → **클릭 가능**. 사용자가 별을 클릭하면 `viewModel>/NewReview/rating` 값이 즉시 업데이트된다 (양방향 바인딩).

#### `SimpleForm` — 폼 전용 레이아웃 엔진

```xml
<form:SimpleForm layout="ResponsiveGridLayout" editable="true">
    <form:content>
        ...
    </form:content>
</form:SimpleForm>
```

그냥 `VBox`로 쌓는 것과 `SimpleForm`의 차이:

| VBox | SimpleForm |
|------|-----------|
| 그냥 세로로 쌓음 | `ResponsiveGridLayout`으로 반응형 배치 |
| 라벨 정렬 없음 | 라벨-필드 자동 정렬 |
| 열 제어 불가 | `GridData`로 L/M/S 각각 span 지정 |
| editable 상태 없음 | `editable="true"`로 폼 모드 전환 |

`form:content`는 `SimpleForm`의 여러 aggregation 중 **폼 필드를 넣는 슬롯**이다. ObjectPageLayout에서 `uxap:headerTitle`, `uxap:sections`로 구분한 것과 같은 원리.

#### GridData — 폼 필드의 열 점유

```xml
<layoutData>
    <layout:GridData span="L12 M12 S12" />
</layoutData>
```

`SimpleForm` + `ResponsiveGridLayout` 조합에서 각 필드가 차지할 열 수를 지정한다. `L12 M12 S12` = Large·Medium·Small 모든 화면에서 12컬럼(전체) 차지 → 별점과 텍스트 영역이 각각 한 줄 전체를 사용한다.

#### 폼 내 FlexBox — 순수 레이아웃용 (items 없음)

```xml
<FlexBox justifyContent="End">
    <Button
        text="{= !!editing_review_id ? '수정 완료' : '리뷰 등록'}"
        type="Emphasized" press="onSubmitReview" />
    <Button
        text="취소"
        visible="{= !!editing_review_id }"
        press=".onCancelEdit" />
</FlexBox>
```

여기서 `FlexBox`는 `items=`가 없으므로 반복 렌더링이 아니라 **순수 레이아웃 컨테이너**다. `justifyContent="End"`로 자식 버튼들을 오른쪽 끝으로 정렬한다.

| 모드 | 보이는 버튼 |
|------|-----------|
| **신규 작성** | `[리뷰 등록]` 만 보임 (취소 버튼 `visible=false`) |
| **수정** | `[수정 완료] [취소]` 둘 다 보임 |

#### `press=".onCancelEdit"` 의 `.` 의미

```xml
<Button text="{i18n>button.cancel}" press=".onCancelEdit" />
```

컨트롤러의 메서드를 참조할 때 `.` 접두어는 **"이 뷰의 컨트롤러"**를 의미한다. `press="onSubmitReview"`(`.` 없음)와 기능적으로 동일하지만, `.onCancelEdit`는 현재 컨트롤러의 메서드임을 **명시적**으로 나타낸다.

## 9. `press` 이벤트 — 뷰와 컨트롤러의 연결고리

뷰에서 정의된 모든 `press` 핸들러는 자동으로 컨트롤러의 동명 메서드를 호출한다.

| press 속성 | 호출되는 컨트롤러 메서드 | 기능 |
|-----------|------------------------|------|
| `press="onNavBack"` | `onNavBack()` | 뒤로가기 |
| `press="onExportInfo"` | `onExportInfo()` | 기본정보 엑셀 다운로드 |
| `press="onExportCast"` | `onExportCast()` | 출연진 엑셀 다운로드 |
| `press="onEditReview"` | `onEditReview(oEvent)` | 내 리뷰 수정 모드 진입 |
| `press="onDeleteReview"` | `onDeleteReview(oEvent)` | 리뷰 삭제 |
| `press="onSubmitReview"` | `onSubmitReview()` | 리뷰 등록/수정 |
| `press=".onCancelEdit"` | `onCancelEdit()` | 수정 취소 |

## 10. 장르·태그 표현 — `FlexBox` + `ObjectStatus`

### FlexBox의 두 가지 용법

FlexBox는 `items=`의 유무에 따라 완전히 다른 용도로 쓰인다:

| 용법 | 코드 | 동작 |
|------|------|------|
| **Aggregation Binding (반복)** | `<FlexBox items="{배열}">` | 배열 길이만큼 자식 복제 — for문처럼 각 요소를 출력 |
| **순수 레이아웃** | `<FlexBox justifyContent="End">` | 자식을 정렬하는 컨테이너 역할만 |

장르·태그 부분은 **반복 용법**이고, 리뷰 폼의 버튼 영역은 **레이아웃 용법**이다.

### Aggregation Binding 원리 (`items=` 있을 때)

```xml
<FlexBox items="{viewModel>/Detail/genres/results}">   ← ① 배열을 items에 바인딩
    <ObjectStatus                                       ← ② 이게 "템플릿"
        text="{viewModel>genre/name}"                   ← ③ 각 요소의 데이터를 읽음
        ... />
</FlexBox>
```

```
① 배열 데이터:
/Detail/genres/results = [
    { genre: { name: "액션" } },
    { genre: { name: "스릴러" } }
]

② FlexBox가 배열 길이만큼 템플릿을 복제:
[🎬 액션] [🎬 스릴러]

③ 템플릿 안에서는 컨텍스트가 각 요소로 바뀜:
{viewModel>genre/name}  →  results[0].genre.name → "액션"
```

`List`, `Table` 등 `items` aggregation을 가진 컨트롤들은 전부 동일한 원리로 동작한다.

- `state="None"` → **검정색 텍스트** (과거 `state="Information"`은 파란색이라 링크로 오인됨)
- `class="detailChip"` → 칩 스타일 CSS

```xml
<!-- 태그는 '#' 접두어를 동적으로 붙임 -->
<FlexBox items="{viewModel>/Detail/tags/results}">
    <ObjectStatus
        text="{= '#' + ${viewModel>tag/name}}"
        class="detailChip sapUiTinyMarginEnd" />
</FlexBox>
```

태그는 장르와 달리 `state` 없음 → 기본 텍스트 색상. `{= '#' + ...}`로 앞에 `#`을 붙여 `#인기 #미국`처럼 표시.

## 11. 전체 뷰 → 컨트롤러 생명주기

```
① URL 변경 (#/Detail/C001)
    ↓
② Router가 DetailMain.view.xml 로드
    ↓
③ DetailMain.controller.js 인스턴스 생성
    ↓
④ onInit() 실행 → router 이벤트 구독
    ↓
⑤ _onRouteMatched() 실행 → _loadDetail(C001)
    ↓
⑥ readOdata()로 OData 호출 ($expand로 장르·태그·출연진·리뷰 포함)
    ↓
⑦ 응답 도착 → viewModel.setProperty("/Detail", oData)
    ↓
⑧ 데이터 바인딩에 의해 UI 자동 갱신
    ├── 헤더: 타이틀, content_type, avg_rating, rating_count
    ├── 기본 정보: content_id, content_type, partner, createdAt, 장르, 태그
    ├── 출연진: Table에 cast 데이터
    └── 리뷰: List에 review 데이터 + MessageStrip 조건부 표시
    ↓
⑨ _checkMyReview() → HasReviewed 설정 → 작성 폼 visible 결정
```

| # | 내용 |
|---|------|
| 1 | **ObjectPageLayout이 루트 + 지휘자** — DynamicPage와 중첩 금지. 스크롤 통합·스냅·앵커바·레이아웃을 총괄 |
| 2 | **Aggregation 태그 = "어디에"** — `<uxap:headerTitle>`는 슬롯 이름, `<ObjectPageDynamicHeaderTitle>`는 실제 컨트롤 |
| 3 | **expandedContent vs snappedContent** — 스크롤 위치에 따라 전환. `heading`만 항상 보임 |
| 4 | **3가지 모델 경로**: `viewModel`(JSONModel, `/`로 객체 탐색), `i18n`(ResourceModel, `.`은 키 이름의 일부), `User`(Core 모델) |
| 5 | **Expression Binding `{= ... }`** — JavaScript 표현식. `!!`로 boolean 캐스팅, 삼항 연산자로 동적 전환 |
| 6 | **단축형 vs 확장형 바인딩**: `{경로}`는 값 그대로, `{path:, formatter:, type:}`는 가공해서 표시 |
| 7 | **FlexBox 두 가지 용법**: `items=` 있으면 배열 반복(템플릿), 없으면 순수 레이아웃 컨테이너 |
| 8 | **SimpleForm은 폼 엔진** — VBox와 달리 라벨 정렬·반응형·GridData 지원. `form:content`는 aggregation 슬롯 |
| 9 | **t:extension** — `sap.ui.table.Table`의 상단 툴바. `ToolbarSpacer`로 버튼을 오른쪽 끝으로 |
| 10 | **viewModel 데이터 출처 구분**: `/Detail`은 백엔드, `/NewReview`·`/HasReviewed`·`/MyReview`는 프론트엔드 자체 계산 |
| 11 | **`sap.uxap`은 ObjectPage 특화 라이브러리** — `sap.f`(목록/레이아웃), `sap.m`(원자재)와 역할이 다름 |
| 12 | **Card/CSSGrid**: `f:Card`와 `sap.ui.layout.cssgrid`가 1.136 CDN에 없어 CSS로 구현 |
| 13 | **리뷰 본인 확인**: `{viewModel>user_user_id} === {User>/user_id}` — FK 비교로 수정·삭제 버튼 제어 |
| 14 | **신규 vs 수정 모드**: `editing_review_id`의 존재 여부로 폼 타이틀·버튼 텍스트·취소 버튼을 동적 전환 |
| 15 | **`press` 이벤트**: 뷰의 `press="메서드명"`이 컨트롤러의 동명 메서드를 자동 호출 |
| 16 | **날짜 포매팅**: `type: 'sap.ui.model.odata.type.DateTime'` + `formatOptions: { pattern: 'yyyy-MM-dd' }` |
