# 사용자 권한 기반 동적 메뉴 시스템 완전 분석

> **작성일**: 2026-07-10
> **목적**: portal_ott에서 manifest.json의 정적 targets가 아닌, DB 기반으로 사용자 권한에 따라 메뉴가 동적으로 생성되는 전체 메커니즘을 이해한다.

---

## 1. 문제 인식

`manifest.json`에는 `sap.ui5.routing.targets`에 정적인 target들이 정의되어 있다. 그러나 portal_ott의 메뉴는 **사용자마다 다르게** 보인다:

- `U001` (OTT_USER_GROUP) → 5개 메뉴 (OTT_MAIN, DETAIL, MYPAGE, BOARD 등)
- `admin1` (OTT_USER_GROUP + OTT_ADMIN_GROUP) → 7개 메뉴 (위 + SETTLEMENT, TREND)

이 차이는 manifest.json만으로 설명되지 않는다. **어딘가에 사용자 권한에 따라 메뉴를 동적으로 구성하는 코드**가 있다.

---

## 2. 정답: 모든 건 CDS View `@restrict` 한 줄에 있다

```cds
@(restrict: [
    { grant: 'READ', where: 'user_id = $user' }
])
view MenusRoleAppliedList as select distinct ...
```

**이 한 줄이 전부다.** JavaScript 코드 어디에도 `if (role === 'Admin')` 같은 분기문은 없다.

---

## 3. 전체 메커니즘: 4단계

### 3.1 DB: 관계형 권한-메뉴 구조

```
User ──< User_RoleGroup >── RoleGroup ──< Role_RoleGroup >── Role ──< Role_Menu >── Menu
```

| 테이블 | 예시 데이터 | 의미 |
|--------|-----------|------|
| `User` | U001, admin1 | 사용자 식별 |
| `User_RoleGroup` | U001→OTT_USER_GROUP, admin1→OTT_ADMIN_GROUP | 사용자-그룹 매핑 |
| `Role_RoleGroup` | OTT_USER→OTT_USER_GROUP, Admin→OTT_ADMIN_GROUP | 역할-그룹 매핑 |
| `Role_Menu` | OTT_USER→OTT_MAIN, Admin→OTT_SETTLEMENT | 역할-메뉴 매핑 |
| `Menu` | OTT_MAIN, OTT_SETTLEMENT 등 | 메뉴 항목 (menu_app_id, menu_route_path 등) |

### 3.2 CDS View: `$user`로 자동 필터링

```cds
view MenusRoleAppliedList as 
    select distinct
        menu.menu_code, menu.menu_name, menu.menu_app_id,
        menu.menu_app_path, menu.menu_route_path, ...
    from
        core_user
        → core_user_rolegroup
        → core_role_rolegroup
        → core_role_menu
        → core_menu
    where menu.menu_use_flag = true
```

`@restrict`에 의해 CAP이 쿼리 실행 시 자동으로 `WHERE user_id = $user`를 추가한다.

| 로그인 사용자 | 자동 추가되는 SQL | 결과 메뉴 |
|-------------|------------------|----------|
| `U001` | `WHERE user_id = 'U001'` | OTT_MAIN, DETAIL, MYPAGE, BOARD |
| `admin1` | `WHERE user_id = 'admin1'` | 위 + SETTLEMENT, TREND |
| `hsan@zen...` | `WHERE user_id = 'hsan@...'` | User_RoleGroup CSV에 정의된 대로 |

### 3.3 MenuLoader: OData 호출

```javascript
// MenuLoader.js
_fetchList: function() {
    const oODataModel = new ODataV4Model({
        serviceUrl: "/srv-api/odata/v4/ott-core/MenuManagement/"
    });
    const oBinding = oODataModel.bindList("/MenusRoleAppliedList");
    return oBinding.requestContexts();
    // CAP이 @restrict로 $user 필터를 자동 적용 → 사용자별 메뉴만 반환!
}
```

### 3.4 Component.js: 동적 라우트/타겟 생성

```javascript
// portal_ott/Component.js
_setRouteAndTarget: function(aMenuList) {
    aMenuList.forEach(oTile => {
        const { menu_app_id, menu_code, menu_app_path, menu_route_path } = oTile;

        // ① 모듈 경로 등록
        sap.ui.loader.config({ paths: { [menu_app_id]: menu_app_path } });

        // ② 동적 라우트 생성
        oRouter.addRoute({
            pattern: `${menu_app_id}/${menu_route_path || ''}`,
            name: menu_code,
            target: { name: menu_code, prefix: menu_app_id }
        });

        // ③ 동적 타겟 생성
        oRouter.getTargets().addTarget(menu_code, {
            type: "Component",
            usage: menu_code,
            ...
        });

        // ④ manifest componentUsages에 동적 추가
        oConfig.componentUsages[menu_code] = {
            name: menu_app_id,
            lazy: true
        };
    });

    // 모든 라우트 정의 후 라우터 초기화
    oRouter.initialize();
}
```

## 4. 정적 target vs 동적 target 비교

| | manifest.json targets | DB 기반 동적 메뉴 |
|---|---|---|
| 정의 시점 | 빌드타임 (소스코드) | **런타임** (DB 조회 결과) |
| 권한 연동 | ❌ 모든 사용자 동일 | ✅ `$user`로 사용자별 필터링 |
| 변경 방법 | 코드 수정 → 재배포 | DB 데이터만 수정 |
| 사용처 | Home, Search 등 공통 화면 | OTT_MAIN, MYPAGE 등 역할 기반 메뉴 |

## 5. 전체 흐름 다이어그램

```
① XSUAA 로그인 (또는 로컬 Basic Auth)
   → $user = "U001"
        │
② MenuLoader → /srv-api/.../MenuManagement/MenusRoleAppliedList
   → core_ott (8084)
   → CAP: @restrict → 자동으로 WHERE user_id = 'U001'
        │
③ HANA DB (또는 SQLite) 조인:
   User ── U001 ── OTT_USER_GROUP ── OTT_USER role ── Role_Menu ── [4개 메뉴]
        │
④ 결과 반환: [OTT_MAIN, OTT_DETAIL, OTT_MYPAGE, OTT_BOARD]
        │
⑤ Component._setRouteAndTarget()
   → 4개 메뉴 각각 addRoute() + addTarget()
   → componentUsages 동적 등록
   → Router.initialize()
        │
⑥ Layout.view.xml
   → 메뉴 버튼 4개 렌더링
   → SETTLEMENT, TREND는 admin1에게만 보임!
```

---

## 6. admin1 로그인 시 차이

```
User ── admin1 ── OTT_USER_GROUP ── OTT_USER role ── [기본 5개 메뉴]
              └── OTT_ADMIN_GROUP ── Admin role ── [+ SETTLEMENT, TREND]

결과: 7개 메뉴 → 버튼 7개 렌더링!
```

---

## 7. BTP 배포 시 주의사항

로컬에서는 `test/data/`의 CSV가 SQLite에 로드되지만, BTP HANA에는 `db/data/`의 CSV만 로드된다.

```
로컬 (SQLite)                     BTP (HANA)
─────────────                     ─────────
test/data/User.csv ✅             db/data/User.csv → PII 이메일 → .gitignore
test/data/User_RoleGroup.csv ✅   db/data/User_RoleGroup.csv → 별도 공유 필요
```

BTP에 배포 시 `db/data/`에 실제 XSUAA 사용자 이메일을 `user_id`로 하는 CSV 파일을 넣고 `deploy:mta` 해야 메뉴가 정상 출력된다.

> 정석은 SCIM 프로비저닝(XSUAA → core_user 자동 동기화). 현재는 수동 시드로 우회 중.

| # | 내용 |
|---|------|
| 1 | `@(restrict: { where: 'user_id = $user' })` 한 줄이 사용자별 메뉴 필터링의 전부 |
