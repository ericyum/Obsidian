

- 대상 POC를 PM 관점에서 회고하며 Lessons Learned를 공유한 세션
- 핵심 주제: POC 진행 과정의 문제점 분석, PM이 갖춰야 할 역량, AI 기반 의사결정의 필요성, 강화학습 모델 한계, POC 종료 조건
- POC는 Proof of Concept로서 "AI 기반 구매 의사결정이 가능하다"는 컨셉 증명에 초점 — 세부 로직 미비는 POC 단계에서 감안 가능하나, 본 프로젝트에서는 현실 반영이 필수

### 액션 아이템

- [ ] 향후 AI 프로젝트 PM은 도메인 지식 + AI 기술 메커니즘 사전 학습 필수
- [ ] 본 프로젝트 전환 시 UAT(User Acceptance Test) 기간을 충분히 확보
- [ ] 가정 기반 설계 시 고객과 합의된 절차(공지-확인-승인) 수립
- [ ] 강화학습 모델 고도화 과제 검토 (가격 커브 예측 → 최적 구매 시점/물량 도출)
- [ ] 정량적 클로징 조건(예: 모델 예측력 기준) 사전 정의

### POC 진행 환경의 특수성

- **첫 프로젝트**: 대상 POC는 조직 내 첫 AI 프로젝트로 레퍼런스 부재
- **불명확한 요구사항**: 고객이 원하는 바가 구체화되지 않은 상태에서 출발
- **외부 인력 소싱**: 내부 인력만으로 수행 불가 → 외부 엔지니어 협업
- **원격 소통**: 외부 엔지니어가 사무실 상주가 아니어서 커뮤니케이션 난이도 상승

---

### WBS 관리와 탐색 스프린트

- **문제**: 불확실한 상황에서 WBS 정의 자체가 어려움 → 정리 미흡 → 관리 어려움
- **대안**:
    - 초기 1~2주는 **탐색 스프린트**로 운영하여 "무엇을 해야 하는 일인가"를 정의
    - 프로젝트 시작 전부터 사전 탐색 기간 확보
    - 단계별로 확정될 때마다 다음 단계의 Task를 정의하며 점진적 구체화 (Rolling Wave Planning)

---

### 산출물 투명성과 테스트 타이밍

- **문제**: 첫 프로젝트라 산출물 이해에 시간 소요 → 테스트 지연 → 문제 대응 지연 → 전체 일정 밀림
- **대안**:
    - **강제적 테스트 기간 확보**: PM이 최소 테스트 시간을 사전에 설정 (실제 2주 계획 → 1주만 남은 사례 교훈)
    - **고객 참여 UAT 필수**: 내부 통합 테스트만으로는 고객 요구사항 누락 발생 → POC 종료 후 추가 요구사항 쇄도
    - 매주 정기적인 산출물 리뷰 루틴 수립 (회의만으로 부족, **산출물 형식 자체를 사전 요구**)
    - 엔지니어와의 회의는 **정기적·강제적**으로 설정

### 가정 기반 설계의 리스크

- **사례**: "최대 주문량 = 탱크 용량 - 재고량"이라는 가정으로 로직 설계 → 실제 6,000톤 주문은 현업에서 발생하지 않음 → 로직과 현실 불일치
- **POC 대응**: 최종 리포트에서 프롬프트 조작으로 6,000톤 노출 방지 (최대 3,000톤으로 제한)
- **POC에서 가능했던 이유**: POC의 목적은 **개념 증명(Proof of Concept)** — "AI가 구매 의사결정에 활용될 수 있다"는 컨셉만 입증하면 됨
- **본 프로젝트에서의 시사점**:
    - 가정 기반 설계 시 반드시 **고객과 합의·공지·확인**하는 절차 수립
    - 중간중간 시뮬레이션 결과를 고객과 공유하여 현실 부합 여부 검증
    - AI 프로젝트 특성상 가정 기반 설계는 불가피하므로, **서로 합의된 가정**임을 명확히 하고 단계적으로 검증

### PM의 필수 사전 역량

- **도메인 지식**: 고객 업무 프로세스, 의사결정 구조, 용어 등 사전 이해 필수
- **AI 기술 메커니즘**: 사용 모델의 Input/Output 구조, 한계점, 적용 가능 범위 파악
- **이유**: 고객도 AI에 대한 이해도가 낮아 "무엇을 해야 할지" 모르는 상태 → PM이 올바른 질문을 던지고 답변을 설계에 반영하려면 두 영역 모두 이해해야 함
- **인터뷰 스킬**: 오픈 퀘스천만으로는 부정확한 답변 가능 → PM이 구조화된 질문으로 가이드

### 형상관리와 이슈 트래킹

- **문제**: 외부 엔지니어와의 협업에서 버전 충돌, 이슈 트래킹 누락 발생
    - 엔지니어가 일정 지연 이슈를 등록했으나 PM이 1주일간 미응답 → 지연 가속화
- **대안**:
    - 이슈 트래킹 도구(예: GitHub Issues) 사용 시 **응답 규칙과 역할 사전 정의**
    - PM은 정기적으로 이슈를 모니터링하고 빠르게 담당자 지정/응답
    - 소통 채널과 규칙을 프로젝트 초기에 확립

### AI 기반 의사결정의 필요성 (POC 배경)

- **비정형 데이터 활용**: 뉴스, 기상정보, 정부정책 등 비정형 정보가 가격에 영향 → 데이터 양이 방대하여 인간이 직접 분석 불가 → AI로 의미 있는 신호 필터링 및 가격 예측 반영
- **비선형적 가격 변동**: 전통적 선형 모델로는 포착 불가한 복잡한 패턴 → 딥러닝 기반 모델(XGBoost, TFT)로 비선형 패턴 학습
- **경험·직관 탈피**: 구매 담당자의 경험·직관 의존 의사결정 → 강화학습으로 최적 구매 시점·물량 도출 시도

---

### 강화학습 모델의 한계와 고도화 방향

- **결과**: "지금 사라 / 기다려라 / 분할 구매해라" 수준의 의사결정만 가능 — 최적 시점·물량 구체적 도출 실패
- **원인 분석**:
    - 가격 예측 모델이 **특정 시점 기준 예측**만 제공 (가격 커브/흐름 예측 미제공)
    - 강화학습 모델이 "언제가 가장 싼가"를 판단할 수 있는 입력을 받지 못함
- **향후 과제**:
    - 기존 모델을 기반으로 가격 흐름(커브) 예측 기능 보완
    - 강화학습 고도화 → 최적 구매 시점·물량 도출
    - 내부 역량으로 자체 업그레이드 추진

---

### POC 종료 및 본 프로젝트 전환 조건

- **POC 목적 재확인**: Proof of Concept — "AI가 대두유 구매 의사결정에 활용될 수 있다"는 개념을 증명하는 것
- **본 프로젝트 클로징 조건**: 정량적 기준 필요 (예: 모델 예측력이 특정 AUC 이상 도달 시 종료)
- **UAT 일정**: POC 2개월(4~5월) 기준, 6월 정도의 피드백 루핑 기간을 추가 확보했어야 함
- **본 프로젝트 필수 요소**:
    - 가정 기반 설계 → 고객 합의 절차 공식화
    - 시스템 시뮬레이션 기반 검증
    - UI 수준의 UAT 환경 고민 (Python Print 출력만으로는 부족)

---

### POC 결론

- POC는 내부 로직의 세부 미비점을 감안하더라도 **컨셉 증명에 성공**
- AI가 구매 의사결정 프로세스에 통합될 수 있다는 가능성을 확인
- 다만 본 프로젝트 전환 시 현실 반영도 높은 설계, 고객 참여 검증, 정량적 클로징 조건이 선행되어야 함
