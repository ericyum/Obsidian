# Meeting Whisper 프로젝트 개요

최종 갱신: 2026-07-20

기준 소스: `meeting_whisper` `0182231` (`main`)

## 한 줄 설명

Meeting Whisper는 브라우저 녹음이나 기존 음성 파일을 받아 SAP AI Core로 전사·요약하고, 권한이 적용된 회의록을 편집·공유하는 SAP BTP 내부 서비스다.

## 현재 운영 결론

- 사용자 UI, CAP srv와 STT Worker는 Cloud Foundry에서 운영한다.
- 사용자는 Approuter/XSUAA를 통해 접속한다.
- 전사는 CF Python Worker가 SAP AI Core `gemini-2.5-flash`를 호출한다.
- 요약은 CAP이 SAP AI Core Orchestration template를 호출한다.
- HANA Cloud에는 metadata, transcript, summary, review, dispatch와 AI usage를 저장한다.
- 업로드 오디오는 Object Store에 임시 저장하고 처리 후 정리한다.
- Kyma CPU Whisper와 로컬 FastAPI는 현재 운영 경로가 아니다.

## 아키텍처

```text
Browser
  -> CF Approuter / XSUAA
  -> CAP srv
     -> HANA Cloud
     -> Object Store
     -> TranscriptionDispatches
  -> CF STT Worker
     -> SAP AI Core STT, gemini-2.5-flash
  -> CAP internal saveTranscript
     -> transcript validation
     -> background summary/review
     -> SAP AI Core Orchestration
```

### Cloud Foundry modules

| module | 역할 |
| --- | --- |
| `meeting-whisper-approuter` | 사용자 진입점, XSUAA 인증, 정적 UI 전달 |
| `meeting-whisper-srv` | 공개 API, 권한, HANA/Object Store, queue, 요약·검토 |
| `meeting-whisper-stt-worker` | 오디오 다운로드·분할, AI Core STT, transcript 검증 |
| `meeting-whisper-db-deployer` | HANA HDI schema 배포 task |

### BTP services

| service | 용도 |
| --- | --- |
| `zen-meeting-whisper` | HANA HDI container |
| `meeting-whisper-auth` | XSUAA 사용자·Worker 인증 |
| `objectstore` / `objectstoreRead` | 오디오 write와 Worker download |
| `default_aicore` | STT와 LLM Orchestration |

### 주요 소스 위치

```text

```

## 사용자 기능

### 회의 생성과 찾기

- 브라우저 녹음 또는 M4A, MP4, WebM, MP3, WAV 업로드
- 제목, 카테고리, 참여자와 공유자 입력
- 기본·사용자 정의 카테고리와 카테고리 기반 제목 생성
- 제목·참여자·요약·전사 내용 검색
- 완료·처리 중·실패 상태와 카테고리 필터

기본 카테고리:

- 주간회의
- 고객미팅
- 프로젝트회의
- 내부회의
- 인터뷰
- 내부 세미나 참석

### 회의록

- transcript와 summary 직접 편집
- 발화별 수동 화자 지정
- 카테고리·제목 변경
- 요약·검토 후보 재생성
- Markdown/전사본 다운로드와 요약 복사
- Outlook 기본 메일 앱으로 요약 전달
- 브라우저 직접 녹음 종료 시 로컬 원본 백업 선택

전사 모델, 화자 수와 자동 화자분리 옵션은 운영 UI에 노출하지 않는다. 전체 화자 일괄 관리 대신 발화별 화자 지정만 유지한다.

### 접근 권한

| 사용자 | 열람 | 제목·카테고리·내용 편집 | 참여자/공유자 관리 | 삭제 |
| --- | ---: | ---: | ---: | ---: |
| 생성자 | 가능 | 가능 | 가능 | 가능 |
| 참여자 | 가능 | 가능 | 불가 | 불가 |
| 공유자 | 가능 | 불가 | 불가 | 불가 |

## 처리 흐름

### 전사

```text
브라우저 SpeechRecognition
  -> 실시간 임시 전사
  -> 화면 미리보기와 브라우저 로컬 draft metadata에만 저장
  -> 서버·최종 전사·요약에는 전달하지 않음

회의 생성
  -> audio upload
  -> Object Store
  -> CAP dispatch queue
  -> Worker claim/download
  -> 길이 판단과 WAV chunk
  -> AI Core STT
  -> 병합·검증
  -> CAP saveTranscript 재검증
```

- 사용자가 실시간 임시 전사 내용을 수정해도 `createJob` 요청에는 포함되지 않는다.
- 최종 전사와 요약의 입력은 녹음 종료 후 업로드된 원본 오디오와 그 오디오에서 생성한 AI Core STT 전사본이다.
- CAP의 `durationSec`를 `expectedDurationSec`로 Worker에 전달한다.
- 측정 또는 예상 길이가 720초를 넘으면 파일 크기와 관계없이 분할한다.
- 잘못된 STT 응답은 같은 chunk를 최대 2회 요청한다.
- 계속 실패한 12분 chunk는 약 6분 단위로 다시 나눈다.
- JSON 구조, 발화 text, timestamp 순서·범위와 raw JSON 중첩을 검증한다.
- 검증 실패 결과를 하나의 raw text segment로 저장하지 않는다.

### 요약과 검토

- transcript 저장 후 Worker 요청은 종료되고 CAP background task가 요약을 생성한다.
- 요약은 `meeting-whisper-summary-ko-v1 / orchestration / 0.0.1` template를 사용한다.
- 요약 모델은 등록 template가 결정하며 `CAP_LLM_MODEL`로 덮어쓰지 않는다.
- 저장된 summary의 검토 후보 생성은 direct orchestration이며 현재 `CAP_LLM_MODEL=anthropic--claude-4.7-opus`를 사용한다.
- 긴 transcript는 12,000자 단위 부분 요약 후 16,000자 단위로 반복 결합한다.
- 카테고리별 guidance를 prompt에 추가한다.
- 검토 후보는 transcript 전체가 아니라 저장된 summary를 대상으로 만든다.
- 사용자에게는 최종 transcript 1개와 summary 1개만 노출한다.

## 상태와 queue

대표 상태:

```text
created -> audio_uploaded -> queued -> transcribing -> summarizing -> done
                                                               \-> failed/error
```

- Worker instance 1개가 최대 2개 job을 동시에 처리한다.
- Worker가 바쁘면 `429`를 반환하고 CAP은 meeting을 queued로 유지한다.
- UI는 queue position, dispatch attempt, Worker stage/message와 오류를 표시한다.
- stale queued/transcribing 작업은 cleanup 정책에 따라 실패 또는 retry 대상이 된다.

## 데이터와 원본 정책

### HANA Cloud

- Meeting metadata와 권한 token
- TranscriptionDispatches와 Worker heartbeat
- Transcript JSON
- Summary JSON·Markdown
- ReviewItems
- AiCoreUsageLogs

### Object Store

- 현재 `OBJECT_STORE_ENABLED=true`다.
- Worker는 signed download URL로 오디오를 가져간다.
- transcript가 저장되고 summary background task가 종료되면 오디오를 삭제한다.
- STT 검증 실패로 transcript가 저장되지 않으면 재전사를 위해 원본을 유지한다.
- 장기 원본 보관소가 아니므로 사용자는 원본 파일을 별도로 보관해야 한다.

## 세션

- Approuter `SESSION_TIMEOUT=180`분
- UI 만료 경고: 약 10분 전
- 녹음 중 keep-alive: 약 30분 간격
- 파일·녹음·복구 업로드 전 세션 확인
- `401` 또는 연장 실패 시 재로그인 안내

## 주요 운영값

| 항목 | 값 |
| --- | --- |
| `TRANSCRIPTION_ENGINE` | `aicore` |
| `AICORE_STT_MODEL` | `gemini-2.5-flash` |
| `AICORE_STT_CHUNK_SECONDS` | `720` |
| `AICORE_STT_MAX_ATTEMPTS` | `2` |
| `WORKER_CONCURRENCY` | `2` |
| Worker memory / disk | `2G` / `2G` |
| `TRANSCRIPTION_DISPATCH_LIMIT` | `2` |
| `CAP_LLM_SUMMARY_CHUNK_CHARS` | `12000` |
| `CAP_LLM_SUMMARY_REDUCE_CHARS` | `16000` |
| upload max | `157286400` bytes, 약 150MiB |

실제 배포값은 `mta.yaml`을 최종 기준으로 확인한다.

## 현재 제약

- 자동 화자분리는 운영 기본값에서 꺼져 있다.
- 60~110분 실제 파일의 반복 처리시간·AI Core 비용·chunk 경계 품질은 계속 관측해야 한다.
- 150MiB를 넘는 ALAC M4A 등은 사용자가 별도 압축해야 한다.
- 과거 raw JSON 형태로 저장된 transcript는 자동 교정하지 않는다.
- Outlook 내보내기는 `mailto:`와 기본 메일 앱의 URL 길이 제한을 받는다.
- 현재 배포는 CF `dev` space이며 staging/prod는 분리되지 않았다.
