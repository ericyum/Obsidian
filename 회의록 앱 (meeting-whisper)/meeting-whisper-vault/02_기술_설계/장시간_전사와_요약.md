# 장시간 전사와 요약

최종 갱신: 2026-07-20

기준 소스: `meeting_whisper` `0182231`

## 목적

장시간 오디오에서 다음 실패를 막고 사용자에게 transcript 1개와 summary 1개를 제공한다.

- duration metadata가 없어 긴 파일을 짧은 파일로 오판
- AI Core의 잘못된 JSON·timestamp 저장
- 한 번의 큰 LLM 요청으로 요약 누락
- Worker와 CAP의 중복 callback 또는 저장

## 전사 흐름

```text
Meetings.durationSec
  -> dispatch.expectedDurationSec
  -> Worker request
  -> 실제/예상 길이 판단
  -> WAV chunk
  -> AI Core STT
  -> 병합·검증
  -> CAP saveTranscript 재검증
```

### 긴 파일 판단

브라우저 WebM 등은 미디어 duration metadata가 없거나 부정확할 수 있다.

- CAP은 Meeting의 표시 길이를 `expectedDurationSec`로 전달한다.
- Worker는 실제 측정 길이와 예상 길이를 함께 본다.
- 둘 중 하나가 720초를 넘으면 파일 크기와 관계없이 분할한다.
- 기존 queue payload에 값이 없으면 dispatch 시 Meeting을 다시 조회한다.

### chunk와 재시도

```text
12분 chunk
  -> AI Core STT
  -> 구조·timestamp 검증
  -> 실패 시 같은 chunk 재시도, 총 2회
  -> 계속 실패하면 약 6분 단위 소분할
  -> timestamp offset을 더해 최종 transcript 병합
```

운영 기본값:

| 환경변수 | 값 |
| --- | ---: |
| `AICORE_STT_CHUNK_SECONDS` | `720` |
| `AICORE_STT_MAX_ATTEMPTS` | `2` |
| `AICORE_STT_INLINE_MAX_AUDIO_BYTES` | `25165824` |
| `AICORE_STT_MAX_AUDIO_BYTES` | `157286400` |

## transcript 검증

Worker와 CAP 양쪽에서 저장 전에 검증한다.

- 완전한 JSON object
- `segments` 배열
- 발화 text 존재
- start/end가 유한한 0 이상 숫자
- `end >= start`
- timestamp 순서
- 실제 또는 예상 오디오 길이 대비 timestamp 범위
- segment text 안에 transcript JSON 전체가 들어간 형태 차단

JSON parse 실패 결과를 하나의 raw text segment로 저장하지 않는다.

검증 위치:

1. Worker `aicore_stt.py` 응답 직후
2. Worker `main.py` callback 직전
3. CAP `WorkerInternalService.saveTranscript`
4. 사용자 transcript 저장과 요약 실행 전

끝까지 실패하면 meeting을 실패 처리하고 원본 오디오는 재전사를 위해 유지한다.

## 장시간 요약

```text
전체 transcript
  -> 12,000자 ordered chunk
  -> chunk별 partial meeting note
  -> 16,000자 단위 partial group
  -> 필요 시 반복 reduce
  -> 최종 meeting note 1개
```

| 환경변수 | 값 |
| --- | ---: |
| `CAP_LLM_SUMMARY_CHUNK_CHARS` | `12000` |
| `CAP_LLM_SUMMARY_REDUCE_CHARS` | `16000` |
| `CAP_LLM_SUMMARY_ATTEMPTS` | `2` |

모든 partial은 동일한 meeting note schema를 사용한다. 부분 결과는 사용자에게 노출하거나 별도 저장하지 않는다.

## 저장과 cleanup

- Worker는 transcript를 저장한 뒤 바로 응답한다.
- CAP은 background task로 summary를 생성한다.
- summary가 성공하면 최종 note와 Markdown을 저장한다.
- summary만 실패하면 transcript는 유지하고 사용자가 재요약할 수 있다.
- summary task가 종료되면 임시 오디오는 정리한다.
- STT 검증 실패로 transcript가 저장되지 않으면 원본 오디오는 유지한다.

## 중복 방어

- Worker run ID와 claim으로 현재 실행을 식별한다.
- CAP은 오래되거나 다른 run의 callback을 거부한다.
- transcript/summary 저장 전에 현재 meeting 상태를 확인한다.
- dispatcher retry가 같은 결과를 중복 저장하지 않도록 한다.

## 사용자 영향

- 내부 chunk 수와 상관없이 transcript 1개, summary 1개를 본다.
- 장시간 작업은 queue position과 Worker stage로 진행을 확인한다.
- STT 검증 실패 시 잘못된 transcript 대신 재전사 오류를 본다.
- 과거 raw JSON transcript는 자동 교정하지 않는다.

## 검증해야 할 항목

- 표시 길이만으로 긴 파일을 강제 분할하는가
- 잘못된 JSON과 역순 timestamp를 거절하는가
- 2회 실패 후 작은 chunk로 복구하는가
- 병합된 timestamp가 전체 timeline을 유지하는가
- 긴 transcript의 모든 chunk가 최종 요약에 반영되는가
- STT 실패 시 원본이 남고 summary가 시작되지 않는가
- 정상 완료 후 임시 오디오가 정리되는가

## 남은 운영 관측

- 60~110분 실제 오디오의 반복 처리시간
- 1시간 파일 2개 동시 처리 시 memory/tmp disk
- AI Core rate limit과 실패 chunk 호출량
- chunk 경계 문장의 누락·중복
- 150MiB 초과 ALAC 파일의 사용자 압축 UX
