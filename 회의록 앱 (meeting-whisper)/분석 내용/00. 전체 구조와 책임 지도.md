# Meeting Whisper 전체 구조와 책임 지도

> 분석 기준: 현재 워크스페이스의 실제 소스  
> 문서 목적: “어디에서 시작해 어디까지 읽어야 하는가”를 한 장에서 안내하는 프로젝트 지도

## 한 문장 요약

Meeting Whisper는 브라우저에서 녹음하거나 업로드한 음성을 **CAP가 소유권과 작업 상태를 관리하면서 Object Store에 보관하고**, 별도 Python Worker가 **SAP AI Core STT**로 전사한 뒤, CAP가 **요약·검토·검색 가능한 회의록**으로 완성하는 BTP 애플리케이션이다.

## 런타임 구성

```text
Browser
  └─ CF Approuter + XSUAA
       ├─ 정적 Vanilla JS UI
       └─ /api/* → CAP MeetingService
                    ├─ HANA: 회의·상태·결과·사용량
                    ├─ Object Store: 임시 원본 음원
                    ├─ DB Dispatch Queue
                    └─ Python Worker /transcribe
                           ├─ CAP internal claim/heartbeat
                           ├─ Object Store signed URL 다운로드
                           ├─ SAP AI Core Gemini STT
                           └─ CAP internal saveTranscript/failTranscription
                                  └─ CAP background AI 요약·검토
```

## 디렉터리 책임

| 디렉터리 | 책임 | 핵심 진입점 |
|---|---|---|
| `app/meeting-ui` | 화면, 녹음, 업로드, polling, 결과 표시 | `index.html`, `js/app.js` |
| `approuter` | 사용자 로그인, 정적 파일, CAP 프록시 | `xs-app.json` |
| `cap/db` | 영속 데이터 모델 | `schema.cds` |
| `cap/srv` | 사용자 API, 내부 Worker API, 비즈니스 로직 | `meeting-service.*` |
| `workers/transcription` | 음원 다운로드와 STT | `app/main.py` |
| `deploy` | CF/Kyma 운영 자료 | `OPERATIONS.md` |
| `scripts` | build, smoke, migration, 복구, benchmark | 각 실행 스크립트 |
| `tests` | CAP/Worker/legacy 회귀 검증 | JS와 pytest 테스트 |
| `legacy/fastapi` | 이전 단일 FastAPI 구현 | `app/server.py` |

## 현재 운영 경로와 보존 경로

- 현재 주 경로: Cloud Foundry Approuter + CAP + HANA/Object Store + CF Python Worker + AI Core.
- `deploy/kyma`: 이전 Kyma Worker를 위한 fallback/reference.
- `legacy/fastapi`: CAP 전환 이전 로컬 단일 앱. 현재 운영 API로 혼동하지 않는다.

## 구조상 가장 중요한 설계 결정

- 긴 음원은 HANA BLOB 대신 Object Store에 임시 저장한다.
- 사용자 요청과 긴 STT 작업을 분리하기 위해 DB 기반 dispatch queue를 둔다.
- Worker가 전사를 저장하면 HTTP 요청을 길게 유지하지 않고 CAP background 요약으로 이어간다.
- 공개 MeetingService와 WorkerInternalService의 권한과 신뢰 경계를 분리한다.
- `Meetings.status/progress/stage`가 UI, CAP, Worker 사이의 공통 진행 상태 계약이다.
