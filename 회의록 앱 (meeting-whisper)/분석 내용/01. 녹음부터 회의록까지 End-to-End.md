# 녹음부터 회의록까지 End-to-End

## 1. 화면 초기화

`index.html`이 `js/app.js`를 module로 로드한다. app.js는 세션 모니터, 화면 이벤트, 목록 조회를 초기화한다. API 호출은 `api.js`, 순수 렌더링은 `ui.js`로 분리된다.

## 2. 녹음과 로컬 복구

`recorder.js`가 MediaRecorder를 생성하고 일정 간격의 blob chunk를 내보낸다. `drafts.js`는 chunk를 IndexedDB에 저장해 새로고침·탭 종료 같은 사고 뒤에도 복구할 수 있게 한다. `session.js`는 녹음 중 세션을 유지하고 업로드 직전 유효성을 다시 확인한다.

## 3. 회의와 음원 생성

프론트엔드는 MeetingService에 회의 메타데이터를 만들고 음원을 올린다. 일반 JSON/base64 경로 외에 `server.js`가 장착한 바이너리 업로드 경로가 큰 payload의 메모리 확대를 줄인다. CAP는 소유권, 최대 크기, MIME type을 확인하고 Object Store object와 AudioObjects 행을 함께 관리한다.

## 4. Dispatch queue

`enqueueTranscription`은 TranscriptionDispatches 행을 만든다. dispatcher는 due 상태의 작업을 lock하고 Worker `/transcribe`를 호출한다. 실패는 attempts와 nextAttemptAt에 기록되고 지수형 backoff 뒤 재시도된다.

## 5. Worker 전사

Worker는 shared bearer token을 먼저 검사하고 CAP internal `claimTranscription`에 성공해야 background 작업을 받는다. signed URL 또는 base64 fallback으로 음원을 준비하고, 필요하면 WAV chunk로 분할하여 AI Core STT를 호출한다. 처리 중 heartbeat와 progress를 CAP에 보낸다.

## 6. 결과 저장과 요약

전사 JSON은 timestamp 순서, 구간 범위, 비정상 raw JSON text 등을 검증한 뒤 `saveTranscript`로 저장된다. CAP는 전사 완료를 먼저 확정하고, 요약은 background task로 실행한다. 긴 전사는 chunk summary → reduce summary로 합친다.

## 7. UI polling

UI는 `getMeetingStatus`를 polling해 queue position, dispatch 상태, worker heartbeat, progress, stage를 표시한다. terminal 상태인 `done` 또는 `failed`에서 polling을 끝내고 전사·요약·검토 데이터를 다시 읽는다.

## 실패 경로

| 실패 지점 | 기록 위치 | 복구 |
|---|---|---|
| 업로드/크기/MIME | HTTP 오류와 Meeting error | 사용자가 다시 선택 |
| Worker 호출 | Dispatch lastError/attempts | backoff 재시도 |
| Worker timeout | Meeting/dispatch failed | 운영 retry |
| STT JSON 이상 | Worker 오류, CAP 검증 거부 | 작은 chunk 재분할 또는 실패 |
| 요약 실패 | Meeting error/stage | `runSummary` 재실행 |
| stale 작업 | heartbeat/lockedAt | cleanup이 reset 또는 fail |
