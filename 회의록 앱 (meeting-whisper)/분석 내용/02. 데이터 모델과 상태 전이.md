# 데이터 모델과 상태 전이

## 핵심 관계

```text
Meetings 1 ── 0..1 AudioObjects
Meetings 1 ── 0..N TranscriptionDispatches
Meetings 1 ── 0..1 Transcripts
Meetings 1 ── 0..1 Summaries
Meetings 1 ── 0..N ReviewItems
Meetings 1 ── 0..N AiCoreUsageLogs
```

## 엔터티별 의미

| 엔터티 | 영속화하는 것 | 수명 |
|---|---|---|
| Meetings | 사용자 소유권, 제목, 분류, 전체 진행 상태 | 회의 삭제까지 |
| AudioObjects | Object Store key, MIME, size, 만료 | 전사/요약 뒤 정리 |
| TranscriptionDispatches | queue, lock, attempt, payload, 오류 | 운영 추적/정리 정책 |
| Transcripts | 정규화된 segment JSON | 회의 삭제까지 |
| Summaries | 구조화된 note JSON과 Markdown | 회의 삭제까지 |
| ReviewItems | 의심 구간·교정 후보 | 재검토/편집 시 갱신 |
| AiCoreUsageLogs | 모델, token, request id, latency, 원문 메타 | 운영·비용 추적 |

## 상태 축을 분리해서 읽기

- `status`: queued, transcribing, summarizing, done, failed 같은 사용자 관점 전체 상태.
- `stage`: 현재 세부 처리 단계.
- `progress`: 0~100 표시용 값.
- Dispatch `status`: queued/dispatching/dispatched/failed로 Worker 전달 자체의 상태.
- heartbeat: 실제 Worker가 살아서 해당 작업을 처리 중인지 판단하는 시간 증거.

상태 문자열을 한 곳에서 바꾸면 UI 조건, cleanup 기준, retry 스크립트, 테스트가 동시에 영향을 받는다.

## 일관성 규칙

1. Meeting은 owner 또는 관리자 권한 없이 읽거나 수정할 수 없다.
2. 음원 object와 AudioObjects 행은 함께 생성·삭제되어야 한다.
3. 전사 저장 전에 worker claim과 transcript 구조 검증이 완료되어야 한다.
4. 요약은 전사 이후에만 만들며 전사가 바뀌면 이전 요약·검토 결과의 유효성을 재평가한다.
5. stale lock은 무한 대기를 피하기 위해 cleanup이 회수한다.
