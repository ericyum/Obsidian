# 인증·권한·신뢰 경계

## 경계 1: Browser → Approuter

XSUAA 로그인 세션을 Approuter가 관리한다. 운영 `xs-app.json`은 공개 정적 리소스와 인증이 필요한 API 경로를 구분하고, CAP destination으로 사용자 token을 전달한다.

## 경계 2: Approuter → MeetingService

`MeetingService @(requires: 'MeetingUser')`가 역할을 요구한다. 서비스 역할만으로 충분하지 않으며, handler가 owner 조건을 query와 action에 적용한다. `MeetingAdmin`은 운영상 전체 접근이 필요한 경우에만 예외가 된다.

## 경계 3: CAP → Worker

CAP는 `TRANSCRIPTION_WORKER_TOKEN`을 bearer token으로 Worker에 전달한다. 운영에서 token 누락을 허용하면 공개 Worker route가 작업을 수락할 위험이 있으므로 Worker는 필수 설정을 검증한다.

## 경계 4: Worker → CAP internal

`WorkerInternalService @(requires: 'Worker')`는 사람용 MeetingUser와 별도다. Worker는 XSUAA client credentials 또는 제한된 fallback token으로 claim, heartbeat, 결과 저장, 실패 보고만 수행한다.

## 경계 5: 서비스 바인딩

- HANA: CAP만 직접 접근.
- Object Store write/read: 목적별 binding 이름을 분리.
- AI Core: CAP 요약과 Worker STT가 binding credential을 사용.
- secret은 소스나 Obsidian 노트에 실제 값으로 복사하지 않는다.

## 점검 체크리스트

- Approuter에서 `forwardAuthToken`이 의도대로 설정되었는가?
- 공개 서비스와 internal 서비스의 role이 분리되어 있는가?
- 모든 entity read뿐 아니라 action에도 owner 검사가 있는가?
- Worker token이 로그와 오류 본문에 노출되지 않는가?
- signed URL TTL과 AudioObjects 만료 시간이 운영 시간보다 충분하고 과도하지 않은가?
