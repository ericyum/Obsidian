## 모듈 2: `meeting-whisper-stt-worker` (Python Worker)

```yaml
type: python
path: workers/transcription
memory: 2G
disk-quota: 2048M
command: python -m uvicorn app.main:app --host 0.0.0.0 --port $PORT
health-check-type: http
health-check-http-endpoint: /health

provides:
  worker-api → worker-url: ${default-url}
  # → CAP srv의 TRANSCRIPTION_WORKER_URL로 자동 연결!

requires:
  - meeting-whisper-auth    # XSUAA (Worker scope)
  - meeting-whisper-aicore  # AI Core
```

**MTA Dependency Wiring**:

```yaml
# CAP srv → Worker 연결 (자동!)
meeting-whisper-srv:
  requires:
    - name: worker-api
      properties:
        TRANSCRIPTION_WORKER_URL: ~{worker-url}
        # ~{worker-url}은 Worker의 provides에서 자동 주입됨

meeting-whisper-stt-worker:
  provides:
    - name: worker-api
      properties:
        worker-url: ${default-url}
```

## 모듈 4: `meeting-whisper-approuter` (Approuter)

```yaml
type: approuter.nodejs
path: approuter
memory: 256M
disk-quota: 512M
properties:
  SESSION_TIMEOUT: 180   # 세션 180분

requires:
  - meeting-whisper-auth
  - srv-api:
      group: destinations
      properties:
        name: srv-api
        url: ~{srv-url}       # CAP srv URL 자동 주입
        forwardAuthToken: true
```

**Approuter → CAP srv 연결**:
- `~{srv-url}` → `${default-url}` → 배포된 CAP 서버 URL 자동 주입
- `forwardAuthToken: true` → 사용자 JWT를 CAP으로 전달
- MTA가 `destinations` 그룹으로 등록 → BTP Destination 서비스에 자동 설정

## BTP 서비스 (Resources)

| 리소스명 | 타입 | 서비스/플랜 | 용도 |
|---------|------|-----------|------|
| `meeting-whisper-db` | `com.sap.xs.hdi-container` | hana / hdi-shared | HANA DB |
| `meeting-whisper-auth` | `managed-service` | xsuaa / application | 사용자·Worker 인증 |
| `objectstore` | `existing-service` | `tf-objectstore` | 오디오 write |
| `objectstoreRead` | `existing-service` | `objectstoreRead` | 오디오 read |
| `meeting-whisper-aicore` | `existing-service` | aicore / extended (`default_aicore`) | STT + LLM |

### Important: `existing-service` vs `managed-service`

| 타입 | 설명 | 예시 |
|------|------|------|
| `managed-service` | MTA가 **직접 생성** | `meeting-whisper-auth` (XSUAA) |
| `existing-service` | **기존 서비스 재사용** (다른 MTA나 수동 생성) | `objectstore`, `default_aicore` |

Object Store와 AI Core는 별도 프로세스로 이미 생성된 서비스를 이름으로 참조.

## 인증 흐름

```
① 사용자 브라우저 → Approuter URL
② Approuter → XSUAA 로그인 페이지 (미인증 시)
③ 로그인 성공 → JWT 발급
④ Approuter → CAP srv (/api/*) → JWT forward (forwardAuthToken: true)
⑤ CAP srv → Worker POST /transcribe
   → Authorization: Bearer TRANSCRIPTION_WORKER_TOKEN
⑥ Worker → CAP srv /internal/*
   → XSUAA binding + Worker scope JWT
```

## Key Differences from ZEN-OTT

| 항목 | ZEN-OTT | Meeting Whisper |
|------|---------|-----------------|
| **프론트엔드** | UI5 + approuter 정적 파일 | Vanilla JS + CAP static serve |
| **html5-apps-repo** | 사용 (6개 repo-host) | 미사용 (CAP이 직접 서빙) |
| **MTA 모듈 수** | 10+ (분리된 MTA들) | 1개 MTA, 4개 모듈 |
| **백엔드 연결** | BTP Destination 수동 설정 | MTA provides/requires 자동 연결 |
| **Worker** | 없음 (CAP이 직접 처리) | Python FastAPI 별도 앱 |
| **배포 복잡도** | 높음 (여러 MTA 순서 배포) | 낮음 (단일 MTA 한 번에) |
